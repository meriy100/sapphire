# 07. 発展篇: 型クラスと HKT

前章までで、`do` 記法と `<-` が `Maybe` でも `Result` でも `Ruby`
でも **同じ形のまま** 動くことを見てきました。不思議だと思いません
でしたか。`Maybe` は失敗の短絡、`Result` はエラー値つきの短絡、
`Ruby` は Ruby スレッド上の逐次実行。三者は意味も振る舞いも違う
はずなのに、ソースコードの形はそっくりそのまま流用できる。これを
支えているのが **型クラス** と **高階カインド (higher-kinded
types, HKT)** の組み合わせです。

本章はその仕組みを解き明かす発展篇です。**ここまでの章を読まなく
ても Sapphire は書けます**。書けるけれど「なぜ書けるのか」が気に
なった方に向けた、一度読めば十分な密度の補足です。背景は主に文書
07（型クラスと高階カインド）と文書 09（prelude）、それから文書 11
（作用モナド）です。

## カインド: 型の型

これまで「型」と呼んできたものの中に、実は **二種類** が混ざって
います。

```sapphire
x : Int
xs : List Int
```

`Int` と `List Int` は **どちらも値の型** です。`x` も `xs` も
具体的な値をひとつ持ちます。一方、

```sapphire
List
Maybe
```

は **それ自体では値の型になれません**。`xs : List` とは書けない。
`List` は「`Int` を受け取ると `List Int` という値の型ができる」
形、つまり **型を受け取って型を返す関数的なもの** です。型コンス
トラクタ、と呼びます。

値の世界で関数と値を型で区別したように、型の世界でも関数的なもの
とそうでないものを区別する仕組みが要ります。これが **カインド
(kind)** です。値の型に **「型の型」** を与える、と言ってもいい。

カインドは二つの記法だけで書けます（文書 07 §Kind system）:

- `*` — 値を分類する普通の型のカインド。
- `κ₁ -> κ₂` — 左のカインドを受け取って右のカインドを返す
  型レベル関数のカインド。右結合です。

具体例を並べると直観が掴めます。

| 型 / 型コンストラクタ | カインド | 意味 |
|------|------|------|
| `Int`, `Bool`, `String` | `*` | 値を持てる普通の型 |
| `List Int`, `Maybe String` | `*` | 適用した結果も普通の型 |
| `List`, `Maybe` | `* -> *` | 型を 1 つ受け取ると `*` になる |
| `Result` | `* -> * -> *` | 型を 2 つ受け取ると `*` になる |
| `Result String` | `* -> *` | 1 つ部分適用した状態 |
| `(->)` (関数矢印) | `* -> * -> *` | 引数型と結果型で `*` になる |

**値の型として現れられるのは `*` のものだけ** です。`x : List` が
型エラーになるのは、`List :: * -> *` であって `*` ではないから。
カインドの合わない型は、そもそも値の型として書けません。

カインドは **推論される** ので、ソースに `::` で明記することは
ありません（文書 07 §Kind system）。`data Maybe a = ...` と書けば
`a :: *` と `Maybe :: * -> *` が自動的に決まる、それだけです。
本章で以降 `T :: κ` という表記を使うときは、Sapphire のソース
構文ではなく **「`T` のカインドは `κ`」というメタ記法** だと
思ってください。

## 型コンストラクタを量化する: `Functor` の読み方

ここで発展篇の核に入ります。`Maybe` と `List` は **中身の型さえ
決まれば値の型になる** 似た形をしている。同じ形の仲間には「中身
を `a` から `b` へ変換する操作」が自然に定義できるはずです。
`Maybe Int -> Maybe String` とか `List Int -> List String` とか。

Sapphire はこの「同じ形の仲間に共通の操作」を、普通の型クラスで
表現します（文書 07 §Standard classes）:

```sapphire
class Functor f where
  fmap : (a -> b) -> f a -> f b
```

**決定的なのは `f` の正体** です。ここでの `f` は `Int` や
`String` のような「カインドが `*` の型変数」ではなく、**カインド
`* -> *` の型変数**、言い換えれば型コンストラクタを表す変数
です。`f a` と書けていることから「`f` は型を 1 つ受け取ると
`*` になる」と **カインド推論が逆算** します。

これが HKT — **高階カインドの変数に対して型クラスを書ける** と
いう力です。「普通の型は `*`、`* -> *` の位置には `Maybe` や
`List` のような型コンストラクタが入る」と読んでください。

### `Functor` のインスタンス

`Maybe` の場合:

```sapphire
instance Functor Maybe where
  fmap f Nothing  = Nothing
  fmap f (Just x) = Just (f x)
```

`Nothing` は `f` を捨て、`Just x` は `f x` で包み直す。素直です。

`List` の場合（文書 09 §Functor / Applicative / Monad instances）:

```sapphire
instance Functor List where
  fmap f Nil         = Nil
  fmap f (Cons x xs) = Cons (f x) (fmap f xs)
```

各要素に `f` を適用した新しいリストを返します。

### 部分適用でカインドを合わせる: `Result`

`Result` には型引数が 2 つあります。`data Result e a = Err e | Ok
a` なので `Result :: * -> * -> *`。これを `Functor` のインスタン
スにしようとすると、`Functor` が要求するのは **`* -> *`** の何か。
カインドが合いません。

ここで「1 つ部分適用する」手を使います:

```sapphire
instance Functor (Result e) where
  fmap f (Err e) = Err e
  fmap f (Ok  x) = Ok (f x)
```

`Result e` の部分で **エラー型 `e` を固定** し、残った成功側の型
だけを `Functor` の `a` として使います。カインドの流れは

```
Result :: * -> * -> *        -- もとの型コンストラクタ
Result e :: * -> *           -- e を 1 つ食わせた形
```

となって、`Functor` の `f :: * -> *` の位置にぴたり収まります。
`fmap` で変換されるのは **成功側だけ、失敗側は素通し** という
Sapphire 的に嬉しい挙動も、この書き方から自然に導かれます。

型引数の順番が `Result e a` と **エラーが先、成功が後** で固定
されているのは、こうして部分適用したときに `Functor` / `Monad`
のカインドに合うようにするためです（文書 09 §Error-handling
ADTs）。

## `Applicative` と `Monad` の階層

`Functor` を下敷きに `Applicative`、さらにその上に `Monad` と、
三段の階層が積まれています（文書 07 §Standard classes）:

```sapphire
class Functor f => Applicative f where
  pure  : a -> f a
  (<*>) : f (a -> b) -> f a -> f b

class Applicative m => Monad m where
  (>>=)  : m a -> (a -> m b) -> m b
  return : a -> m a
  return = pure
```

`=>` の左はスーパークラス制約。**`Monad` にするには先に
`Applicative`、`Applicative` にするには先に `Functor` のイン
スタンスが要る**、という順番を型レベルで強制しています。

三者の役割分担は次のとおりです。

| クラス | 主要メソッド | 何をもたらすか |
|------|------|------|
| `Functor f` | `fmap : (a -> b) -> f a -> f b` | 中身を「関数で」変換する |
| `Applicative f` | `pure : a -> f a` / `(<*>) : f (a -> b) -> f a -> f b` | 純粋値を持ち上げる / 「包まれた関数」で包まれた値に適用する |
| `Monad m` | `(>>=) : m a -> (a -> m b) -> m b` | 直前の結果に依存した **次の作用** を連鎖する |

`do` 記法が依存しているのは `Monad` です。`a <- ma` の右辺（`ma :
m a`）が分かってからでないと、左辺の `a` を使った続き（`\a -> ...
: a -> m b`）が組み立てられない。この **「前のステップの結果が
次のステップの組み立てに効く」** 依存関係を表す道具が `>>=` で、
`do` ブロックは `>>=` の糖衣（文書 07 §`do` notation）です。

### prelude インスタンスを一つずつ辿る

文書 09 §Functor / Applicative / Monad instances には、これまでの
章で使ってきた型に対するインスタンスが並んでいます。

**`Maybe`**:

```sapphire
instance Monad Maybe where
  Nothing >>= _ = Nothing
  Just x  >>= f = f x
```

失敗 (`Nothing`) なら続きを呼ばずにそのまま `Nothing`。成功なら
中身を `f` に渡す。ch5 で手作業の `case` ネストで書いていた
「失敗の短絡」そのものです。

**`Result e`**:

```sapphire
instance Monad (Result e) where
  Err e >>= _ = Err e
  Ok  x >>= f = f x
```

形は `Maybe` と同じ。失敗なら `Err` を保持したまま素通し、成功
なら中身を `f` に流す。エラー値を **持ったまま** 短絡できるので、
`Maybe` に比べて「なぜ失敗したか」の情報量が増えます。

**`List`**:

```sapphire
instance Monad List where
  xs >>= f = concatMap f xs
```

これは「失敗の短絡」とはだいぶ違う顔をしています。`List` モナド
は **非決定的選択** のモナドで、「`xs` の各要素を `f` に流した
結果のリスト達を **平坦に連結** する」。Haskell 由来の慣用で、
`do` の中で複数候補を列挙できる書き方になります。

**`Ruby`**:

```sapphire
instance Monad Ruby where
  (>>=) = primBind
```

中身は `primBind`（= 実行時プリミティブ。文書 11 §Primitives）
に委譲されます。`primBind ra f` は「`ra` を Ruby スレッドで走ら
せて、結果を marshaller で Sapphire 側に戻して、それを `f` に
適用して得た次のアクションを走らせる」という遅延計算を構築する。
`>>=` や `do` の形は他のモナドと同じでも、意味は **Ruby 評価の
逐次化** です。

この 4 つを並べると、同じ `>>=` という窓口から、**型ごとに異なる
連鎖の意味** が提供されていることがよく分かります。`do` が同じ
ソースで 4 つの型に対して動くのは、ディスパッチ先の `Monad`
インスタンスが 4 通りあって、型検査器が文脈に合わせてどれを呼ぶ
か決めているからです。

### 作用モナド (effect monad) と snake_case

`Ruby` は特殊な立ち位置の `Monad` インスタンスです。文書 11
§Role term ではこの役回りを **作用モナド (effect monad)** と
呼びます。ch6 の散文でそう呼んでいたのはこのためで、型位置では
あくまで `Ruby a` と書きます（呼称は役割、型は `Ruby`）。

実装プリミティブの名前には Ruby 慣用の snake_case が透けて見え
ます。仕様上の名前は `primReturn` / `primBind` ですが、裏で
Sapphire が生成する Ruby ランタイム側には `prim_return` /
`prim_bind` として降りる、という対応関係になっています（実装
側の詳細。チュートリアルでは意識不要）。型クラス利用者として
触るのは `pure` と `>>=` と `return` だけで、`primReturn` /
`primBind` を直接呼ぶことはありません。

## モナド則

`Monad` インスタンスには、実装者が守ることになっている **三つの
等式** があります（文書 07 §Standard classes / Laws、文書 11
§Class instances）:

```
return a >>= f          ≡ f a                       -- (left identity)
m >>= return            ≡ m                         -- (right identity)
(m >>= f) >>= g         ≡ m >>= \x -> f x >>= g     -- (associativity)
```

読み方:

- **左単位元**: 「値を `return` で包んでから `>>=` で `f` に流す」
  ことと「直接 `f` に `a` を渡す」ことは同じ。
- **右単位元**: 「`m` の結果を `return` で包み直すだけ」なら `m`
  のまま。
- **結合則**: `>>=` を連ねる順序（`(m >>= f)` を先にまとめるか、
  `f x >>= g` を先にまとめるか）で結果が変わらない。`do` 記法が
  `>>=` の連鎖に展開できるのはこの則があるからです。

`Maybe` で軽く確かめるなら、左単位元は

```sapphire
return 3 >>= \x -> Just (x + 1)
  == Just 3 >>= \x -> Just (x + 1)         -- return = Just
  == (\x -> Just (x + 1)) 3                -- Just の >>= 定義
  == Just 4
```

で、ラムダに `3` を直接渡した `Just 4` と一致します。`Result`
でも `List` でも `Ruby` でも、同じ 3 則が成り立つよう実装されて
います。「`do` で書いた式を等価変形して読みやすくできる」と
いう日常的な快適さは、これらの則が保証してくれているもの。

## 現状スコープの境界

Sapphire が **最初の実装で取り込まないもの** が、ここに関連して
いくつもあります（文書 07 §Open questions）:

- **多引数型クラス (MPTC)**: `class C a b where ...` の形。
  現状は単引数のみ。
- **柔軟なインスタンス形**: `instance Eq (List Int)` のような
  具体型を混ぜたインスタンスや、型変数を繰り返し使うインスタンス。
  Haskell 98 形 (`C T` で `T` は ground か「異なる型変数を並べた
  型コンストラクタ適用」) に制限。
- **型族・関連型 (type families, associated types)**: クラス本体
  で型を計算する機能。
- **`deriving`**: `data T = ... deriving (Eq, Show)` の形でイン
  スタンスを自動生成。現状は手書きのみ。
- **オーファン・重複インスタンス**: どちらも禁止。1 クラス × 1
  型コンストラクタに 1 インスタンスだけ、と強く縛ります。
- **ソース上のカインド注記**: `(a :: * -> *)` のような書式は
  現状サポートなし。カインドは常に推論。

これらはすべて「最初のコンパイラを軽く保つため」の一時的な制限で、
将来拡張として検討される予定です。現段階で手書きのボイラープレー
ト（特に `Eq` / `Ord` / `Show` の `instance`）は多少増えますが、
その分、型クラス解決は決定的で、モジュールのインポート順に依存
しない結果になります。

## Haskell との比較 (参考)

用語と構文は **ほぼ Haskell の流儀を踏襲** しています。`Functor`
/ `Applicative` / `Monad` の階層、`fmap` / `pure` / `>>=` の名前、
`do` 記法の脱糖規則。違いは表面的なもの中心で、たとえば `::`（型
注記）が Sapphire では `:` だったり、`Either e a` が `Result e a`
になっていたり、タプルの代わりにレコードを使うといった点。概念
レベルでは Haskell 98 から Monad 階層の現代化版 (`Functor =>
Applicative => Monad`) までを素直に取り込んだ、と思ってもらって
構いません。

## もっと深く

型クラスと HKT は言語機能としてはここで尽きていますが、プログラ
ミング手法としての奥行きは相当なものです。`Functor` / `Applicative`
/ `Monad` の法則と、それが日常のコードを等価変形で読みやすくする
仕組みは、Haskell 向けの教材で同じ概念が厚く扱われています。
`Reader` / `State` / `Writer` のような古典的な「作用モナド」群、
モナド変換子 (monad transformers)、`traverse` / `sequence` と
いった Applicative 由来の道具立て — いずれも Sapphire の現状
スコープの外ですが、同じ道具立てでプログラムが組めることを知って
おくと、将来 Sapphire がそれらを取り込んでいく過程が読みやすく
なります。

Sapphire 本編に戻る読者は、ここで手に入れた「`m` の正体は `* ->
*` の型コンストラクタで、`do` は `>>=` の糖衣」という理解を持って、
ch5 / ch6 をもう一度読んでみるのもおすすめです。前はブラック
ボックスだった箇所が、型クラス解決と HKT の組み合わせで成立し
ていた、と見え方が変わるはずです。
