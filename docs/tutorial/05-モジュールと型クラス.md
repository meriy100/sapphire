# 05. モジュールと型クラス

ここまでは「一つのファイルにすべて書いてある世界」の話だった。
この章では、コードを **モジュール** に分け、抽象的な振る舞いを
**型クラス** で扱う方法を扱う。最後には汎用 `Monad` と `do`
記法に到達する。

このチュートリアルでいちばん重い章である。Ruby の世界に対応
させづらい概念がいくつか出てくるので、ゆっくり進める。

主に文書 07（型クラスと higher-kinded types）、08（モジュール）、
09（prelude）を背景にしている。

## モジュール

### ファイル ＝ モジュール

Sapphire は **1 ファイル 1 モジュール** で、ファイルパスが
モジュール名に対応する（文書 08 §One module per file）。

| ファイルパス | モジュール名 |
|--------------|--------------|
| `src/Main.sp` | `Main` |
| `src/Data/List.sp` | `Data.List` |
| `src/Data/List/Extra.sp` | `Data.List.Extra` |

ファイルの一行目で

```
module Data.List
  ( map
  , filter
  , foldr
  )
  where
```

のように宣言する。`( ... )` はそのモジュールが **公開する
名前のリスト** で、書かれていない名前はモジュールの外からは
見えない。

Ruby に比較すると、Ruby の `module Data; module List; ...; end; end`
のネストと、`private` 宣言を組み合わせたもの、と思えば近い。
ただし Sapphire では「公開する」を **エクスポートリスト** で
宣言的に列挙する方式になっている。

### 型・コンストラクタ・クラスの公開

エクスポートリストの書き方には少しコツがある（文書 08
§Visibility）。

```
module Geometry
  ( Point        -- 型 Point だけ公開（コンストラクタは隠す）
  , Shape(..)    -- 型 Shape とそのコンストラクタすべて公開
  , area         -- 関数 area を公開
  , class Eq     -- 型クラス Eq だけ公開（メソッドは隠す）
  , class Ord(..) -- 型クラス Ord とそのメソッドすべて公開
  )
  where
```

- 型を **名前だけ** 公開すると、外からはその型を変数の型注釈
  などで使えるが、`Just 1` のように **コンストラクタを直接呼ぶ
  ことはできない**。データを抽象化したいときに使う。
- 型クラスを **名前だけ** 公開すると、`Eq a =>` のような
  制約位置では使えるが、外でその型クラスのインスタンスを
  書くことはできない（メソッド名が見えないから）。

### `import`

別のモジュールを使うには `import`。

```
import Data.List              -- 全部を非修飾で取り込む
import Data.List (map, filter) -- map と filter だけ
import qualified Data.List    -- 必ず Data.List.map のように修飾して使う
import Data.List as L         -- 略称で参照（L.map）
import Data.List hiding (map) -- map を除いたすべて
```

Ruby の `require` + `include` の両方を兼ねる感じ。`qualified`
や `as` は名前の衝突を避けるための仕組みで、Ruby の世界では
あまり登場しないが、関数型では珍しくない。

### 公開関数にはシグネチャを書く

文書 08 §Top-level signatures が決めているルール:

- **公開する** トップレベル束縛には型シグネチャ必須。
- 私的 (private) なものは省略可（型推論されるから）。

これは「モジュール境界の型を、内部実装の変更に振り回されない
ようにロックする」ための規律で、Ruby の `# typed: strict` を
公開メソッドに限って付けるようなノリ。

## 型クラス: ある型に「振る舞い」を与える

ここから先は Ruby に翻訳しづらい概念が増える。最初に
**型クラスは何のためにあるのか** をはっきりさせておく。

### 動機: 同じ操作を複数の型で

Ruby では「等しさを比べる」「文字列にする」「ソートする」
といった操作は、すべての値が（多くの場合 `Object` 経由で）
持っている。たとえば `[1, 2, 3].sort` も `["c", "a", "b"].sort`
も動く。

Sapphire は静的型付きで、しかも `<` のような比較演算子は
本来 `Int -> Int -> Bool` という固定の型を持っている
（文書 05）。これでは文字列をソートできない。

そこで **型クラス** という仕組みが要る。ざっくり言うと:

- **型クラス** は、ある型がどんな操作をサポートしているかを
  宣言した「インターフェース」。
- **インスタンス** は、特定の型がそのインターフェースをどう
  実装するかを与える「実装」。

Ruby の `module Comparable` を `include` する話に近い。違いは

- Ruby は実行時に `respond_to?` でも調べられるが、Sapphire は
  **静的に解決される**。
- Ruby は「クラスに対してミックスインする」が、Sapphire は
  **クラスを使う側の型** に対して、後から外部でも実装を与え
  られる（一定の制約はある）。

### `Eq` の例

文書 07 §Standard classes 冒頭で定義されている `Eq` クラス:

```
class Eq a where
  (==) : a -> a -> Bool
  (/=) : a -> a -> Bool
  x /= y = not (x == y)   -- デフォルト実装
```

「型 `a` が `Eq` のインスタンスである」とは、`(==)` と `(/=)`
が定義されていることである。`(/=)` には **デフォルト実装** が
あるので、各インスタンスは普通 `(==)` だけ書けばよい。

prelude には `instance Eq Int` などが用意されていて
（文書 09 §Eq instances）、自分が新しい型を作ったときは

```
data Color = Red | Green | Blue

instance Eq Color where
  Red   == Red   = True
  Green == Green = True
  Blue  == Blue  = True
  _     == _     = False
```

のように **インスタンス宣言** を書く。

### 制約付き型シグネチャ

型クラスを使う関数の型は次のような形になる。

```
isMember : Eq a => a -> List a -> Bool
isMember x []        = False
isMember x (y :: ys) = if x == y then True else isMember x ys
```

`Eq a =>` の部分が **型クラス制約**。「`a` は `Eq` のインス
タンスであるような型なら何でもよい」と読む。

Ruby に翻訳すると、Duck typing の世界で `x` が `==` に応答する
ことを期待しているのに似ているが、Sapphire はそれを **型シス
テムが静的に保証してくれる**。

### 主な標準型クラス

文書 07 §Standard classes と文書 09 §Class instances が定めて
いるもの:

| クラス | できること | 主要メソッド |
|--------|-----------|--------------|
| `Eq a` | 等しさの比較 | `(==)`, `(/=)` |
| `Ord a` | 順序比較 | `compare`, `(<)`, `(>)` ほか |
| `Show a` | 文字列化 | `show` |
| `Functor f` | 「容器の中身」に関数を適用 | `fmap` |
| `Applicative f` | 並列っぽい合成 | `pure`, `(<*>)` |
| `Monad m` | 連鎖的合成 | `(>>=)`, `return` |

`Eq` `Ord` `Show` あたりは Ruby の `Comparable` `to_s` と
だいたい同じ感覚。問題は下三つで、これがチュートリアル後半の
山場である。

## Higher-kinded type と「容れ物」のクラス

`Functor` 以下のクラス定義に出てくる `f` は、**型ではなく
型コンストラクタ** を表す変数である。`Maybe` や `List` の
ような「中に何かを入れる入れ物」をひとくくりにする仕組み
（文書 07 §Kind system）。

少し丁寧に。Sapphire の型には **kind**（型の型）がある:

- `Int`、`Bool`、`String`、`Maybe Int` の kind は `*`
  （ふつうの値を分類する型）。
- `Maybe`、`List` の kind は `* -> *`（型を一つ受け取って
  普通の型を返す型コンストラクタ）。
- `Result` の kind は `* -> * -> *`。

`class Functor f where ...` の `f` は kind `* -> *` を持つ。
だから `Functor Maybe` や `Functor List` は意味があって、
`Functor Int` は意味がない（kind が合わない）。

Ruby だとこの区別はそもそも存在しない。型がないから「中身」
という概念は型システムの外。Sapphire ではこれを正面から
扱うために、higher-kinded な型変数を許している。

### `Functor`

```
class Functor f where
  fmap : (a -> b) -> f a -> f b
```

`fmap` は「`f` という容れ物の中身に `a -> b` を適用して、
中身が `b` の容れ物にする」操作。`Maybe` のインスタンスは:

```
instance Functor Maybe where
  fmap f Nothing  = Nothing
  fmap f (Just x) = Just (f x)
```

これはまさに前章の `mapMaybe` と同じ。

`List` のインスタンスは

```
instance Functor List where
  fmap f Nil         = Nil
  fmap f (Cons x xs) = Cons (f x) (fmap f xs)
```

で、これも前章の `map` と同じ。つまり `fmap` は `map` を
**`Maybe` でも `List` でも他の容れ物でも** 同じ名前で呼べる
ようにする一般化である。

Ruby の `Enumerable#map` を「列挙可能な何にでも `map` を
生やすミックスイン」と思うと近い。Sapphire の `Functor` は
それの「型レベル版」。

### `Applicative`

`Applicative` まで来ると説明がやや込み入ってくる。要点だけ。

```
class Functor f => Applicative f where
  pure  : a -> f a
  (<*>) : f (a -> b) -> f a -> f b
```

- `pure x` は「`x` を `f` の中に最小限の形で包む」。
  `Maybe` なら `Just x`、`List` なら `[x]`。
- `(<*>)` は「容れ物の中の関数」を「容れ物の中の値」に適用
  する。これがあると、二引数以上の関数を容れ物の中で持ち
  回せる。

実用上は `Monad` の方が出番が多いので、`Applicative` は
「`Monad` の前段階」として理解しておけば最初は十分である。

### `Monad`

```
class Applicative m => Monad m where
  (>>=)  : m a -> (a -> m b) -> m b
  return : a -> m a
  return = pure
```

`>>=` は **bind** と読む。型を見ると

```
m a -> (a -> m b) -> m b
```

「`m` の容れ物に入った `a`」と「`a` を受け取って `m` の容れ物に
入った `b` を返す関数」を渡すと、「`m` の容れ物に入った `b`」を
返す。

…字面ではピンと来ないので、`Maybe` で具体化する
（文書 09 §Functor / Applicative / Monad instances）。

```
instance Monad Maybe where
  Nothing  >>= _ = Nothing
  Just x   >>= f = f x
```

つまり

- 左辺が `Nothing` なら、右の関数を呼ばずに `Nothing` を伝播。
- 左辺が `Just x` なら、`x` を取り出して `f` に渡す。

これだけ。これを使って前章の `addMaybe` を書き直すと:

```
addMaybe : Maybe Int -> Maybe Int -> Maybe Int
addMaybe ma mb =
  ma >>= \a ->
  mb >>= \b ->
  pure (a + b)
```

`ma` から `a` を取り出し、`mb` から `b` を取り出し、
両方成功したら `pure (a + b)`。途中で `Nothing` があれば
そこで止まって全体が `Nothing` になる。

Ruby で同じことをすると、こうなる:

```ruby
def add_maybe(ma, mb)
  return nil if ma.nil?
  return nil if mb.nil?
  ma + mb
end
```

`ma` と `mb` の型を `Maybe Int` ではなく `Integer | nil` と
すれば、Ruby の `nil` チェックの連鎖と同じ気持ちになる。
`>>=` はこの `return nil if x.nil?` の **連鎖を一般化したも
の**、と思ってよい。

### `do` 記法

`>>=` を直に書くのは慣れないと辛い。文書 07 §`do` notation が
定める **`do` 記法** で楽になる:

```
addMaybe : Maybe Int -> Maybe Int -> Maybe Int
addMaybe ma mb = do
  a <- ma
  b <- mb
  pure (a + b)
```

これは上の `>>=` 連鎖と **完全に同じ意味の糖衣**（文書 07
が定めている脱糖規則そのまま）。

- `a <- ma` は「`ma` から値を取り出して `a` に束縛」。
- 最後の行は `m` で包まれた値を直接書く（`pure (a + b)` は
  `Maybe Int` 型）。

Ruby との対応で言うなら、Ruby の

```ruby
def add_maybe(ma, mb)
  a = ma or return nil
  b = mb or return nil
  a + b
end
```

の `or return nil` の自動化だと考えると親しみやすい。`<-` が
「失敗ならそこで打ち切って全体を失敗にし、成功なら値を取り
出して続ける」を一行で表している。

### `Result` でも同じ仕組み

`Result e a` の `Monad` インスタンス（文書 09）:

```
instance Monad (Result e) where
  Err e >>= _ = Err e
  Ok  x >>= f = f x
```

これで、`Result String Int` を返す `parseInt` を二回呼んで
合計するみたいなことが、

```
parseAndAdd : String -> String -> Result String Int
parseAndAdd s1 s2 = do
  a <- parseInt s1
  b <- parseInt s2
  pure (a + b)
```

と書ける。途中で `Err "not a number"` が出たら、そこで全体が
`Err "not a number"` になる。

Ruby で

```ruby
def parse_and_add(s1, s2)
  a = parse_int(s1)
  return a if a.is_a?(Err)
  b = parse_int(s2)
  return b if b.is_a?(Err)
  Ok.new(a.value + b.value)
end
```

と毎回書いていた早期リターンの気持ちを、`Monad Result` が
言語レベルで吸収している。

### 「Monad とは何か」を一言で

教科書的には「ある形のしるし `m` と `pure` `>>=` の三つ組」と
書かれるが、Ruby 利用者向けには次のように覚えると実用上は
回る:

> **`Monad m` とは「`m a` という型で表される計算を、`do` 記法で
> 連鎖させられる」ためのインターフェースである。**

`m` が `Maybe` なら「失敗するかもしれない計算」、`Result e` なら
「失敗理由つき計算」、後の章で出てくる `Ruby` なら「Ruby を
評価する計算」。これらすべてが `do` で同じ書き方になる、と
いうのが `Monad` という抽象の御利益である。

## ちょっと書いてみる

`Show` クラスを使って、自前の `Color` を文字列にしてみる。

```
data Color = Red | Green | Blue

instance Show Color where
  show Red   = "red"
  show Green = "green"
  show Blue  = "blue"

-- 任意の Show 可能な値をリストごとに連結
allShown : Show a => List a -> String
allShown []        = ""
allShown [x]       = show x
allShown (x :: xs) = show x ++ ", " ++ allShown xs
```

`allShown [Red, Green, Blue]` は `"red, green, blue"` を返す。
`allShown [1, 2, 3]` は `"1, 2, 3"` を返す（`Show Int` インス
タンスは prelude にある）。同じ関数が `Color` のリストに対しても
`Int` のリストに対しても動く。

## 仕様への気付き

- 型クラスの説明は「インターフェースをモジュール側で後付け
  できる」という Duck typing 寄りの言い方で何とか入門できる。
  ただ、higher-kinded type の説明は **kind** という新しい
  概念をどうしても呼び出さないといけない。文書 07 §Kind
  system は手短だが、入門段階では `Functor f` の `f` を見た
  瞬間に詰まる読者が想像できる。
- `Monad` を Ruby 利用者に通すための翻訳としては「早期リターン
  の連鎖を抽象化したもの」が最有力で、本チュートリアルでも
  これを採った。ただこの説明は `Maybe` と `Result` には効くが
  `List` モナド（非決定計算として読む）や次章の `Ruby` モナド
  には素直には効かない。後者については `Monad` の比喩を一段
  抽象化し直す必要があり、本当はチュートリアルとしては章を
  もう一つ割きたい量である。
- 文書 07 自体の重さについて: 入門資料で扱おうとすると、
  - Kind の話、
  - 型クラスの宣言と instance、
  - 制約付きスキームの読み方、
  - `do` の脱糖、
  - `Functor` → `Applicative` → `Monad` の階層、
  という五本柱が同時に立ち上がる。今回のチュートリアルでは
  「`Applicative` は名前だけ通り過ぎる」「`Functor` は `map`
  の一般化として軽く触れる」と削った。それでも本章は他章の
  1.5〜2 倍の重さになっている。
- もし「Elm-Haskell の中間」を狙う方向に倒すとすれば、
  - `Applicative` を表面から削って `Functor` と `Monad` だけ
    残す、
  - higher-kinded を出さずに済むよう、`Maybe` 専用 `mapMaybe`
    のような **specialised 版** を prelude の主要 API として
    並べる、
  あたりが具体的な軽量化候補になる。`do` 記法を残すだけなら
  `Monad` 単独でもよい（`pure` は別途プリミティブとして
  prelude に置く）。
- 単項クラス・no-orphans・no-overlap という現状の制約は、
  入門の重さには影響していない（むしろ「考えなくていい」と
  いう意味で軽くなっている）ので、ここはこのままが妥当に思う。

## 次の章では

ここまで貯めた `do` と `Monad` の道具立てを使って、Sapphire の
最大の特徴である **`Ruby` モナド** に踏み込む。`:=` で Ruby
コードを埋め込み、`run` で結果を取り出す、という Sapphire 独自
