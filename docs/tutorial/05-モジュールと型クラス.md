# 05. モジュールと型クラス

ここまでは「一つのファイルにすべて書いてある世界」の話だった。
本章では、同じ操作を複数の型で使うための **型クラス** と、
コードを分ける **モジュール** を導入する。最後に、`Maybe` や
`Result` のような「失敗するかもしれない計算」を `do` 記法で
つなぐ道具立てに到達する。

Sapphire の表現力の中核に当たる章で、少し長い。**まず具体的に
困る場面を置いて、抽象化は必要になった分だけ入れる** という順で
進める。

主に文書 07（型クラス）、08（モジュール）、09（prelude）を
背景にしている。

## 型クラス: 同じ操作を複数の型で

### 具体的な困りごと

リストを昇順に並べる関数を `Int` 版と `String` 版で別々に
書きたくはない。Ruby なら `[1, 2, 3].sort` も
`["c", "a", "b"].sort` も同じ名前で動くが、そこには「`<=>` を
持つ」という暗黙の約束があった。Sapphire は静的型付きなので、
この約束を **型** で表明する仕組みがいる。それが **型クラス**
である。

### `Eq` クラス

文書 07 §Standard classes の冒頭に登場する `Eq`:

```sapphire
class Eq a where
  (==) : a -> a -> Bool
  (/=) : a -> a -> Bool
  x /= y = not (x == y)   -- デフォルト実装
```

「型 `a` が `Eq` のインスタンスである」とは、`(==)` と `(/=)` が
定義されていること。`(/=)` には **デフォルト実装** があるので、
普通は `(==)` だけ書けばよい。

自作の `Color` に `Eq` を与える:

```sapphire
data Color = Red | Green | Blue

instance Eq Color where
  Red   == Red   = True
  Green == Green = True
  Blue  == Blue  = True
  _     == _     = False
```

これで `Red == Green` や `[Red] == [Red]` のような式が書ける
（リストの `Eq` は prelude が `Eq a => Eq (List a)` の形で用意
している。文書 09 §Eq instances）。

Ruby の `Comparable` を `include` するのに近いが、決定的な違いは
**型システムがインスタンスの有無を静的に確かめる** こと。
`respond_to?` で実行時に調べる不確かさはない。

### 制約付き型シグネチャ

複数の型で使える関数の型には **制約** を書く。

```sapphire
isMember : Eq a => a -> List a -> Bool
isMember x []        = False
isMember x (y :: ys) = if x == y then True else isMember x ys
```

`Eq a =>` の部分が **型クラス制約**。「`a` は `Eq` のインス
タンスである何かなら何でも」と読む。`isMember Red [Red, Blue]`
も `isMember 1 [1, 2, 3]` も同じ関数で動く。

### よく使う標準クラス

prelude が用意しているクラスのうち、入門で覚えておきたい三つ
（文書 09 §Class instances）:

| クラス | できること | 主要メソッド |
|--------|-----------|--------------|
| `Eq a` | 等しさの比較 | `(==)`, `(/=)` |
| `Ord a` | 順序比較 | `compare`, `(<)`, `(>)`, `(<=)`, `(>=)` |
| `Show a` | 文字列化 | `show` |

`Ord` は `class Eq a => Ord a where ...` と宣言されていて、`Eq`
が **スーパークラス** になる（文書 07 §Class declarations）。
つまり「`Ord` にするには先に `Eq` が要る」。

使うときは

```sapphire
smaller : Ord a => a -> a -> a
smaller a b = if a < b then a else b
```

のように `Ord a =>` を置く。`smaller 3 1` は `1`、`smaller "a"
"b"` は `"a"` を返す。

### インスタンスを書ける範囲

`instance C T` の形で書けるのは、だいたい以下のいずれか
（文書 07 §Instance declarations）:

- **`C` を定義したモジュール** で `T` のインスタンスを書く。
- **`T` を定義したモジュール** で `C` のインスタンスを書く。

どちらでもない場所に書く「孤児インスタンス (orphan)」は **禁止**。
「同じ `(C, T)` に相反するインスタンスが二つ生まれる」のを防ぐ
規律である。`deriving (Eq, Show)` のような自動導出は現仕様では
まだ入っていない（07 OQ 8）。当面は手書きする。

## モジュール: ファイル単位の名前空間

型クラスの話を区切って、コードを分ける仕組みに移る。

### ファイル ＝ モジュール

Sapphire は **1 ファイル 1 モジュール** で、ファイルパスが
モジュール名に対応する（文書 08 §One module per file）。

| ファイルパス | モジュール名 |
|--------------|--------------|
| `src/Main.sp` | `Main` |
| `src/Data/List.sp` | `Data.List` |
| `src/Data/Color.sp` | `Data.Color` |

ファイルの先頭で

```sapphire
module Data.Color
  ( Color(..)
  , isPrimary
  )
  where
```

と宣言する。`( ... )` は **エクスポートリスト**: 公開する名前を
列挙する。書かれていない名前は外からは見えない。

単一ファイルのスクリプトは `module` ヘッダを省略でき、その場合は
暗黙に `module Main where ...` と同じ扱いになる（文書 08 §One
module per file）。ただし他モジュールから `import` されるファイル
にはヘッダが必須。

### エクスポートの書き分け

型・コンストラクタ・クラスで書き方が少しずつ違う（文書 08
§Visibility）:

```sapphire
module Data.Color
  ( Color         -- 型 Color だけ公開（コンストラクタは隠す）
  , Shape(..)     -- 型 Shape とコンストラクタ全部
  , isPrimary     -- 関数
  , class Named   -- 型クラス Named だけ公開（メソッドは隠す）
  , class Ord(..) -- 型クラス Ord とメソッド全部
  )
  where
```

- 型を **名前だけ** 公開すると、外ではシグネチャに `Color` を
  書けるがコンストラクタは呼べない。データの内部表現を抽象化
  したいときに使う。
- 型クラスを **名前だけ** 公開すると、`Named a =>` の制約位置で
  使えるが、外でインスタンスは書けない。

prelude は `Eq` `Ord` などを **すべて `class Eq(..)` 形で** 公開
しているので、利用者は自作型に対して自由に `instance Eq MyType
where ...` を書ける（文書 09 §Module export list）。

**公開する型の中身に私的な型が漏れる** のは禁止されている
（文書 08 §Visibility）。`isPrimary : PrivateColor -> Bool` を
公開しながら `PrivateColor` を非公開のまま、という状態は定義時に
コンパイラが拒否する。

### `import` の形

別モジュールを使うには `import`。

```sapphire
import Data.Color                    -- 公開名を非修飾で全部
import Data.Color (Color, isPrimary) -- 指名
import qualified Data.Color          -- 修飾必須 (Data.Color.isPrimary)
import Data.Color as C               -- 略称 (C.isPrimary)
import Data.Color hiding (isPrimary) -- 除外
```

`qualified` と `as` は名前衝突を避けるための道具。prelude
（`Maybe`、`Ord`、`map` など）は **暗黙に** 全モジュールに
インポートされるので `import Prelude` は普通書かない（文書 09
§The prelude as a module）。

### 公開名にはシグネチャ必須

文書 08 §Top-level signatures が定めるルール:

- **公開する** トップレベル束縛には型シグネチャ必須。
- 私的な束縛は省略可（推論される）。

モジュール境界の型を「内部実装の変更で勝手に変わる」のを防ぐ
規律。

## 失敗するかもしれない計算を連ねる — `do` 記法

型クラスの道具立ての上に、Sapphire で最も実戦的な記法が乗る。
目的地は **`Maybe` や `Result` を扱うコードを読みやすく書く**
こと。

### `Maybe` の連鎖を手作業で

二つの `Maybe Int` を足す関数を、既存の道具で書くと:

```sapphire
addMaybe : Maybe Int -> Maybe Int -> Maybe Int
addMaybe ma mb =
  case ma of
    Nothing -> Nothing
    Just a  -> case mb of
      Nothing -> Nothing
      Just b  -> Just (a + b)
```

動くが、`Maybe` を増やすほど `case` のネストが深くなる。Ruby の

```ruby
def add_maybe(ma, mb)
  return nil if ma.nil?
  return nil if mb.nil?
  ma + mb
end
```

という「早期リターンの連鎖」相当の、軽い記法が欲しい。

### `do` 記法

Sapphire には **`do` 記法** がある（文書 07 §`do` notation）:

```sapphire
addMaybe : Maybe Int -> Maybe Int -> Maybe Int
addMaybe ma mb = do
  a <- ma
  b <- mb
  pure (a + b)
```

- `a <- ma` は「`ma` の中身を取り出して `a` に束縛」。`ma` が
  `Nothing` ならそこで終わり、全体が `Nothing` になる。
- 最後の `pure (a + b)` は「計算結果を `Maybe` で包む」
  （ここでは `Just (a + b)`）。

つまり **`<-` が「成功なら値を取り出して続き、失敗なら全体を
失敗にする」を一行で表す**。Ruby の `or return nil` 相当を言語が
肩代わりしている。

### 同じ記法が `Result` にも効く

`Result e a` でも `do` は同じ形で書ける。`readInt : String ->
Maybe Int`（文書 09）の失敗理由を文字列で付け直した補助関数を
挟むと:

```sapphire
parseE : String -> Result String Int
parseE s = case readInt s of
  Just n  -> Ok n
  Nothing -> Err ("not a number: " ++ s)

parseAndAdd : String -> String -> Result String Int
parseAndAdd s1 s2 = do
  a <- parseE s1
  b <- parseE s2
  pure (a + b)
```

途中で `Err` が出たら、そこで全体が同じ `Err` になる。Ruby で
毎回書いていた「`is_a?(Err)` なら早期リターン」が、`do` に
潰れている。

**同じ `do` が `Maybe` でも `Result` でも動く** のは型クラスの
恩恵である。`<-` も `pure` も、特定の型ではなく `Monad` という
型クラスに対して定義されているので、どちらでも使える。

## `Monad` という抽象

最後に、いまの話を一段抽象化しておく。

```sapphire
class Monad m where
  (>>=)  : m a -> (a -> m b) -> m b
  return : a -> m a
```

`>>=` は bind と読む。「`m` の中の `a`」と「`a` を受け取って `m`
の中の `b` を返す関数」を渡すと、「`m` の中の `b`」を返す。
`do` 記法は **この `>>=` への糖衣** である（文書 07 §`do`
notation）。`addMaybe` を脱糖すると

```sapphire
addMaybe ma mb =
  ma >>= \a ->
  mb >>= \b ->
  pure (a + b)
```

となる。`<-` が `>>=` に、束縛の右辺がラムダ本体に、順に展開
される。

`Maybe` と `Result e` には prelude が `Monad` インスタンスを
用意している:

```sapphire
instance Monad Maybe where
  Nothing >>= _ = Nothing
  Just x  >>= f = f x

instance Monad (Result e) where
  Err e >>= _ = Err e
  Ok  x >>= f = f x
```

どちらも「失敗ならそこで止めて伝播、成功なら値を `f` に流す」。
`do` の直感そのまま。

### 発展: `Monad m` の `m` は何か

（正確には `class Applicative m => Monad m where ...` である。
`Monad` には二つのスーパークラスが控えていて、`Functor` →
`Applicative` → `Monad` の階層で、それぞれ `fmap` / `pure` / `>>=`
を足していく。入門段階では **`Monad` だけ掴んでおけば `do` は
使える** ので、ここでは上流の階層は省略した。）

もう一点だけ。`class Monad m where ...` の `m` は、`Int` や
`String` のような「ふつうの型」ではなく、`Maybe` や `List` の
ような **型を一つ受け取る型コンストラクタ** を表している。
`Maybe Int` は型、`Maybe` は型コンストラクタ、という区別。
厳密には **kind** という「型の型」が要る話だが（文書 07 §Kind
system）、当面は「`Maybe` のまま `Monad` に書ける」と受け止めて
先に進めばよい。

## 次の章では

ここで手に入れた `do` と `Monad` の道具立てを、Sapphire 最大の
特徴である **Ruby インタロップ** に適用する。埋め込み Ruby
コードを表す `Ruby a` 型（「作用モナド」とも呼ぶ — 文書 11）は、
`Maybe` や `Result` と同じ `Monad` インスタンスを持つ。`:=` で
Ruby コードを埋め込み、`run` で pure な Sapphire 側に結果を
戻す、という流儀を次章で見ていく。
