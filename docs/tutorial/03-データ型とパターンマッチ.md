# 03. データ型とパターンマッチ

この章では Sapphire の **代数的データ型 (ADT)** と、それを
分解する **パターンマッチ** を扱う。これが入って初めて、
Sapphire は「データを設計する言語」になる。

主に文書 03（データ型）、06（パターンマッチ）、09（prelude）を
背景にしている。

## ADT とは

Ruby に翻訳しづらい概念の最右翼かもしれないので、まずは
実例から。

```
data Bool = False | True
```

これは「`Bool` という型は、`False` か `True` のどちらかである」
という宣言である。`False` も `True` も、引数を取らないコンスト
ラクタとして、`Bool` 型の値を生み出す道具になる
（文書 03 §Constructor schemes）。

Ruby でこの構造に近いものを無理に書くなら

```ruby
class Bool; end
class False < Bool; end
class True  < Bool; end
```

のようなクラス階層と、`is_a?` でぶら下がるパターンマッチング
ライブラリ（`case ... in` の Ruby 3.0+ 機能や `dry-types` 系）の
組み合わせになる。Sapphire ではこれが言語の基本機能であり、
**「コンストラクタが何個あるか」** を型システムが知っていて、
網羅的に処理できているかどうかを検査してくれる。

### 引数を取るコンストラクタ

コンストラクタは値を抱えてもよい。

```
data Maybe a = Nothing | Just a
```

`Maybe a` は「`a` 型の値が **あるかもしれない**」を表す。
`a` は型変数で、利用者が `Maybe Int` や `Maybe String` のように
具体化する。

- `Nothing` はそれ単体で `Maybe a` 型の値（任意の `a` について）。
- `Just 42` は `Maybe Int` 型の値。
- `Just "hi"` は `Maybe String` 型の値。

Ruby に翻訳すると

```ruby
# Just(42) のつもり
{ tag: :Just, values: [42] }
# Nothing のつもり
{ tag: :Nothing, values: [] }
```

実際、コンパイル後の Ruby 表現はだいたいこの形になる
（文書 10 §ADTs）。`nil` を `Nothing` の代わりに使うことは
あえて避けている。`Just nil` と `Nothing` を区別したいから
である。

### 失敗を表す `Result`

```
data Result e a = Err e | Ok a
```

`Maybe` は「失敗の理由」を持たないが、`Result e a` は失敗側に
理由 (`e`) を持てる。Ruby で `[:ok, value]` / `[:error, msg]` の
ペアを返す慣用にだいたい対応する。

- 成功なら `Ok 42`、失敗なら `Err "parse failed"`。
- 型パラメータが二つあるが、エラー型 `e` を先に書く（文書 09）。

Sapphire では Haskell の `Either` に当たるものをこの名前で扱う。
Ruby の "ok/err" ボキャブラリに合わせるためである。

### 再帰的なデータ型

コンストラクタは自分自身の型を抱いてもよい。

```
data Tree a = Leaf | Node a (Tree a) (Tree a)
```

これは「`Tree a` 型の値は、`Leaf` であるか、`a` の値と二つの
`Tree a` を持つ `Node` のどちらかである」と読む。リストや
木構造はみなこの形で書ける。

`List` も例外ではなく、文書 09 では

```
data List a = Nil | Cons a (List a)
```

として定義されている（リストの専用糖衣 `[]` `[1, 2, 3]` `::`
は次章で扱う）。

## 値を取り出す: `case ... of`

ADT を作っただけでは何もできない。中身を取り出すには
**パターンマッチ** が必要である（文書 06）。

```
describeMaybe : Maybe Int -> String
describeMaybe m = case m of
  Nothing -> "なし"
  Just n  -> "値は " ++ show n
```

`case 式 of` の右側に、改行とインデントで腕を並べる。それぞれの
腕は **`パターン -> 式`** の形。

腕は次のような種類のパターンを取れる（文書 06 §Abstract syntax）:

| パターン | 意味 |
|----------|------|
| `_` | ワイルドカード。何にでもマッチし、束縛しない |
| `n` | 変数パターン。何にでもマッチし、`n` という名前で束縛する |
| `42` | リテラルパターン。その値とちょうど等しいときマッチ |
| `Just n` | コンストラクタパターン。`Just` の中身を `n` で受ける |
| `Cons x xs` | 同上、引数を二つ受ける |
| `{ x = a, y = b }` | レコードパターン（次章） |
| `(p : type)` | 型注釈つきパターン |

Ruby の `case ... in` パターン構文を触ったことがあれば、感覚は
近い。`Just n` は Ruby で言えば `[:Just, n]` を分解するような
ものだ、と思っておけばよい。

### 網羅性は静的に保証される

```
-- これは静的エラーになる
bad : Maybe Int -> Int
bad m = case m of
  Just n -> n
  -- Nothing がない！
```

文書 06 §Exhaustiveness によれば、網羅していない `case` は
**コンパイル時の警告ではなくエラー** である。Ruby のように
「実行してみたら `NoMatchingPatternError`」はあり得ない。

これは Sapphire が Ruby を吐く言語であることに由来する。
パターンマッチが漏れていると、生成された Ruby コードに走時
例外が紛れ込み、静的型検査の意味が損なわれるからである。

### 文字列・整数のパターン

```
describeNumber : Int -> String
describeNumber n = case n of
  0 -> "ゼロ"
  1 -> "イチ"
  _ -> "それ以外"
```

`Int` や `String` は値が無限にあるので、列挙では網羅できない。
最後にワイルドカード `_` で **catch-all** を置くこと、と決まって
いる（文書 06 §Exhaustiveness の三番目の箇条）。

## 関数定義の中での暗黙のパターンマッチ

文書 07 が一般化した clause 形式 (`f x y = body`) は、引数位置
にパターンを書ける。

```
-- これは
fromMaybe : a -> Maybe a -> a
fromMaybe def m = case m of
  Nothing -> def
  Just x  -> x

-- こう書いてもよい（複数 clause）
fromMaybe : a -> Maybe a -> a
fromMaybe def Nothing  = def
fromMaybe _   (Just x) = x
```

複数 clause で書くと Haskell っぽい雰囲気になる。意味は同じ。
コンストラクタ引数を受ける位置では `(Just x)` のように
括弧で囲む（文書 06 の `apat` ルール）。

Ruby で書くと素朴には次のような分岐になる。

```ruby
def from_maybe(default, maybe)
  case maybe
  in { tag: :Nothing }    then default
  in { tag: :Just, values: [x] } then x
  end
end
```

## `@` でまとめて束縛する

「コンストラクタの中も見たいが、値全体にも名前を付けたい」と
きは `@` パターンが使える（文書 06 §Atomic patterns の (PAs)）。

```
isPositive : Maybe Int -> Maybe Int
isPositive m = case m of
  Just n | True -> if n > 0 then Just n else Nothing  -- ※
  _ -> Nothing

-- @ パターンで書くと
isPositive : Maybe Int -> Maybe Int
isPositive m = case m of
  whole@(Just n) -> if n > 0 then whole else Nothing
  _ -> Nothing
```

`whole@(Just n)` は「`Just n` パターンに合致したら、その値
全体を `whole` という名前でも参照できる」ことを表す。

> ※ 一行目の例の `| True ->` 風のガード構文は、現在のドラフトでは
> 採用されていない（文書 06 OQ 1）。説明用に書いた疑似コード
> なので、`@` 版の方を本物として読んでほしい。

## 型変数を持ちまわる

`Maybe a` のような多相な型は、関数を書くときも自然に多相に
なる。

```
mapMaybe : (a -> b) -> Maybe a -> Maybe b
mapMaybe f Nothing  = Nothing
mapMaybe f (Just x) = Just (f x)
```

これで `mapMaybe (\n -> n + 1) (Just 41)` は `Just 42`、
`mapMaybe (\s -> s ++ "!") (Just "hi")` は `Just "hi!"` になる。

この `mapMaybe` の正体は、後の章で出てくる `Functor` 型クラスの
`fmap` メソッドである。今は「`Maybe` の中身に関数を適用する」
くらいに思っておけばよい。

## 型と値の名前空間

Sapphire では、

- **大文字** で始まる識別子は **型・コンストラクタ** にしか使えない、
- **小文字** で始まる識別子は **値・関数・型変数** にしか使えない、

という規律が一貫している（文書 02、03、06）。

```
-- これは静的エラー: Just は大文字始まり、コンストラクタなのに
-- それを値変数として束縛しようとしている。
let Just = 1 in Just
```

Ruby の「`Foo` は定数、`foo` はローカル変数」のしきたりが、
もう少し厳しく徹底されていると思えばよい。

## ちょっと書き下してみる

`Maybe` を使った具体例。

```
-- 整数を二倍するが、入力が None なら None を返す。
doubleMaybe : Maybe Int -> Maybe Int
doubleMaybe Nothing  = Nothing
doubleMaybe (Just n) = Just (n + n)

-- 二つの Maybe を足し合わせ、片方でも Nothing なら Nothing。
addMaybe : Maybe Int -> Maybe Int -> Maybe Int
addMaybe Nothing  _        = Nothing
addMaybe _        Nothing  = Nothing
addMaybe (Just a) (Just b) = Just (a + b)
```

Ruby ではこれを書くと、ぐにゃっと `if x.nil?` の連打になりがち
である。Sapphire（あるいは関数型一般）では

- 「ない」を `Nothing` として正面から型に乗せる、
- 分岐は `case` / 複数 clause で書き、コンパイラに網羅性を
  検査させる、

という体裁になる。後の章では、この `addMaybe` のような連鎖を
もっと省力に書ける `do` 記法と `Monad` の話につながる。

## 仕様への気付き

- 03（データ型）と 06（パターンマッチ）の組み合わせは Ruby
  読者にも比較的素直に伝わる。`case x in { tag: :Just, values: [n] }`
  のような Ruby 3 の構文と直感的に対応していて、橋渡しがしやすい。
- 一方で 06 §Pattern typing にある双方向判定 `Γ ⊢ p ⇐ τ ⊣ Γ'` の
  記法は、入門者がいきなり読むと壁になる。チュートリアルからは
  完全に隠せたので、仕様書の側でもこの形式記号は「実装者向け
  付録」のような扱いに切り出す余地がありそう。
- ここまでの体験は **Elm 同等** の重さで収まっている。次章以降の
  「重さ」は型クラスとモナドが入ってから。
