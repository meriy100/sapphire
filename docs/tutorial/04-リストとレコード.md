# 04. リストとレコード

前章で ADT とパターンマッチを手に入れたので、いよいよ
**実用的なコレクション** を扱える。Sapphire で日常的に使う
データ構造は大きく二つ:

- **リスト** — 同じ型の値を順に並べる（Ruby の `Array` 相当）。
- **レコード** — 名前付きフィールドの集まり（Ruby の `Hash` /
  `Struct` 相当）。

主に文書 04（レコード）、06（パターンマッチ）、09（prelude）を
背景にしている。

## リスト

### 三つの書き方

`List a` 型は文書 09 で次のように定義されている。

```
data List a = Nil | Cons a (List a)
```

つまり前章でやった ADT の典型例である。が、毎回
`Cons 1 (Cons 2 (Cons 3 Nil))` と書くのは辛いので、専用の
**糖衣構文** が用意されている（文書 09 §List）。

```
[]                  -- Nil と同じ
[1, 2, 3]           -- Cons 1 (Cons 2 (Cons 3 Nil)) と同じ
1 :: 2 :: 3 :: []   -- 同上、:: で連結
1 :: rest           -- rest の先頭に 1 を足したリスト
```

`::` は **cons 演算子** といい、右結合・段 5（文書 05 §Operator
table）。Ruby の `[1, *rest]` の感覚に近いが、こちらは「先頭に
要素を一つ付ける」操作で、両端任意の連結ではない。

### パターンマッチでの分解

リストは前章の延長線上で素直に分解できる。

```
describe : List Int -> String
describe []        = "空"
describe [x]       = "一つだけ: " ++ show x
describe (x :: xs) = "先頭は " ++ show x
```

- `[]` パターンは空リストにのみマッチ。
- `[x]` パターンは「ちょうど一つの要素を持つリスト」にマッチし、
  その要素を `x` で受ける。
- `(x :: xs)` は「先頭 `x`、残り `xs`」に分解する。

`case` の腕は **上から順に試される**（前章のルール）。
`describe [1]` は `[]` に一致せず、`[x]` にマッチして最初の `x`
を `1` に束縛するため `"一つだけ: 1"` を返す。`[1, 2]` は `[]`
にも `[x]` にも一致せず、`(x :: xs)` で `x=1`、`xs=[2]` に束縛
される。

最後の `(x :: xs)` パターンは、Ruby の

```ruby
case xs
in [x, *rest] then ...
end
```

に近い。違いは Sapphire では `xs` の方も `List a` 型だ
（決して別物のスライスではない）こと。

### 高階関数: `map` / `filter` / `foldr`

文書 09 §Utility functions が prelude に置いている主要な
関数を、Ruby の対応物と並べておく:

| Sapphire | 型 | Ruby 相当 |
|----------|----|-----------|
| `map f xs` | `(a -> b) -> List a -> List b` | `xs.map { |x| f.(x) }` |
| `filter p xs` | `(a -> Bool) -> List a -> List a` | `xs.select { |x| p.(x) }` |
| `foldr f z xs` | `(a -> b -> b) -> b -> List a -> b` | `xs.reverse.inject(z, &f)` 風 |
| `foldl f z xs` | `(b -> a -> b) -> b -> List a -> b` | `xs.inject(z, &f)` |
| `length xs` | `List a -> Int` | `xs.length` |
| `head xs` | `List a -> Maybe a` | `xs.first`（ただし `Maybe`） |
| `tail xs` | `List a -> Maybe (List a)` | `xs[1..]`（ただし `Maybe`） |
| `null xs` | `List a -> Bool` | `xs.empty?` |
| `concat xss` | `List (List a) -> List a` | `xss.flatten(1)` |
| `concatMap f xs` | `(a -> List b) -> List a -> List b` | `xs.flat_map(&f)` |

例:

```
let xs = [1, 2, 3, 4, 5] in
let doubled = map (\n -> n + n) xs in    -- [2, 4, 6, 8, 10]
let evens   = filter (\n -> n % 2 == 0) xs in -- [2, 4]
let total   = foldr (+) 0 xs in           -- 15
total
```

Sapphire の `map` / `filter` は **新しいリストを返す関数** で
あって、`xs` 自身は変わらない。そもそも Sapphire には「変更
する」概念がない。

### `head` と `tail` はなぜ `Maybe` なのか

Ruby の `xs.first` は空配列に対して `nil` を返す。Haskell の
`head` は例外を投げる。Sapphire は両方を避けて、`head xs` の
型を `List a -> Maybe a` にしている（文書 09 §Design notes）。

```
case head [1, 2, 3] of
  Nothing -> "空でした"
  Just n  -> "先頭は " ++ show n
```

「失敗するかもしれない」を関数の型に正面から書くのが、
Sapphire の流儀である。

### `foldr` の気持ち

`foldr f z [1, 2, 3]` は

```
f 1 (f 2 (f 3 z))
```

と展開される。たとえば `foldr (+) 0 [1, 2, 3]` は
`1 + (2 + (3 + 0)) = 6` となる。

Ruby の `inject` は左から畳む `foldl` 寄り。`foldr` を
Ruby で素直に書くと

```ruby
def foldr(f, z, xs)
  xs.reverse.inject(z) { |acc, x| f.call(x, acc) }
end
```

になる。「リストを右から組み立てる」と覚えるとよい。新しい
リストを構築する場面で出番が多い。

### 仮コード: 数値リストの合計と平均

```
sumOf : List Int -> Int
sumOf = foldl (+) 0

averageOf : List Int -> Maybe Int
averageOf [] = Nothing
averageOf xs = Just (sumOf xs / length xs)
```

`sumOf = foldl (+) 0` は **point-free** な書き方。`sumOf xs =
foldl (+) 0 xs` を、引数 `xs` を省いた形で書いている
（カリー化のおかげで成立する）。Ruby で言えば
`Method#to_proc` を駆使した感じだが、Sapphire ではこれが
普通の書き方の一つ。

なお `/` は文書 05 により `Int -> Int -> Int`（整数除算）。
`averageOf [1, 2]` は `Just 1` を返し、`Just 1.5` にはならない。
`Float` を含む数値多相は現仕様では保留中（05 OQ 1 / 07 OQ 6）。

## レコード

### 構造的な「名前付き Hash」

```
let origin = { x = 0, y = 0 } in origin
```

レコードリテラルは Ruby のハッシュ `{ x: 0, y: 0 }` に似ている
が、いくつか違いがある（文書 04）。

- フィールドの型は静的に決まっている。`{ x = 0, y = 0 }` の型は
  `{ x : Int, y : Int }`。
- フィールドの **順序は型に影響しない**。`{ x : Int, y : Int }`
  と `{ y : Int, x : Int }` は同じ型。
- 構造的等価。**型に名前はない**。`Point` として宣言したから
  と言って、別の `Vector` 型と区別されるわけではない。
- 末尾コンマは許されない。`{ x = 1, }` は構文エラー。

### フィールドの取り出し

```
let p = { x = 3, y = 4 } in p.x   -- 3
```

`式.フィールド名` で取り出す。`.` はメソッド呼び出しではなく
**構文上のフィールド選択** であり、ドットの右に来るのは
小文字始まり識別子に限られる（文書 04 §Abstract syntax）。

`p.x.y` は `(p.x).y` と読む。左結合。

### 関数的な更新

```
let p  = { x = 3, y = 4 } in
let p' = { p | x = 10 } in
p'              -- { x = 10, y = 4 }
```

`{ レコード | フィールド = 新しい値, ... }` で、**元のレコードを
変更せず、指定フィールドだけ差し替えた新しいレコード** を作る
（文書 04 §Record update）。

Ruby に当てはめるとだいたい

```ruby
p_prime = p.merge(x: 10)
```

である。ただし Sapphire の `{ p | x = 10 }` は

- **存在しないフィールドを足せない**（型が変わるから）、
- **既存フィールドの型を変えられない**、

という制約がある。常に「同じ型のレコードを返す」と決まっている。

### レコードのパターンマッチ

```
distanceFromOrigin : { x : Int, y : Int } -> Int
distanceFromOrigin { x = a, y = b } = a * a + b * b
```

レコードパターンは **必要なフィールドだけ** 書けばよい
（文書 06 §Record patterns）。`{ x = a }` だけ書いても、
`{ x = 0, y = 0 }` のような値にマッチする。これはレコード
リテラル側の「全フィールド書け」ルールとは非対称になっている
が、ergonomics 優先での意図的な選択である。

注意: ここで「必要なフィールドだけ」と言っているのは
**パターンが言及するフィールド** の話。レコードの **型** 自体は
あくまで全フィールドを持っている（文書 04 が **行多相** を
採らず、閉じた構造的レコードを採用しているため）。

### レコードと `data` の関係

レコードはあくまで **構造的な型** で、ADT のように名前で
区別したいときは

```
data Point = MkPoint { x : Int, y : Int }
```

のように `data` でラップする…と書きたくなるが、文書 04 OQ 2
（`data` のフィールド付きコンストラクタ）はまだ正式には決着
していない。当面は

```
data Point = MkPoint Int Int
```

のような **位置引数** で書くか、ラップせず **構造的レコードを
そのまま型として使う** のが現状の落としどころ:

```
-- 関数のシグネチャでレコード型を直接書く
distance : { x : Int, y : Int } -> { x : Int, y : Int } -> Int
distance p q =
  let dx = p.x - q.x in
  let dy = p.y - q.y in
  dx * dx + dy * dy
```

実践上は構造的レコードをそのまま使うのが文書 04 の想定する
素直な道で、わざわざ `data` で名前を付けるのは「同じ形を別物
として区別したい」とき。

```
type Point = { x : Int, y : Int }   -- 文書 09 OQ 2 で議論中の type alias
```

のような型エイリアスはまだ仕様にないが、入っていれば名前付き
構造の出番が増える。今のチュートリアルでは構造的レコードの
ままで通す。

## 組み合わせ例: タスクリスト

`type` エイリアスがまだ無いので、レコード型はシグネチャごとに
書き下す。同じ型を何度も書くのは冗長だが、現状の draft では
こうなる:

```
allTasks : List { name : String, done : Bool }
allTasks =
  [ { name = "spec を読む", done = True }
  , { name = "tutorial を書く", done = False }
  , { name = "実装言語を決める", done = False }
  ]

undone : List { name : String, done : Bool }
       -> List { name : String, done : Bool }
undone tasks = filter (\t -> not t.done) tasks

names : List { name : String, done : Bool } -> List String
names tasks = map (\t -> t.name) tasks
```

- 構造的レコード `{ name : String, done : Bool }` をそのまま
  値・型として使っている。
- `undone` は未完了タスクだけを残す。
- `names` は名前のリストを取り出す。

Ruby と並べると、

```ruby
tasks = [
  { name: "spec を読む",     done: true  },
  { name: "tutorial を書く",  done: false },
  { name: "実装言語を決める", done: false },
]

undone = tasks.reject { |t| t[:done] }
names  = tasks.map    { |t| t[:name] }
```

の感覚に近い。違いは Sapphire 側ではすべての段階で型が静的に
チェックされていて、`t.done` が `Bool` であることや `names` の
返り値が `List String` であることを compiler が知っている、と
いう点。

## タプルが無いことについて

Haskell の `(1, "hi")` のようなタプル構文は、現在の Sapphire の
仕様にはない（文書 09 §Design notes）。`fst` / `snd` は
**二フィールドのレコード** を取る関数として定義されている:

```
fst : { fst : a, snd : b } -> a
snd : { fst : a, snd : b } -> b
```

タプルを書きたいときは `{ fst = 1, snd = "hi" }` と素直に
書く、というのが現在の方針。文書 09 OQ 1 で「タプル構文を
入れるかどうか」自体が未決問題として残っている。

## 仕様への気付き

- リストは Sapphire を Ruby 利用者に説明する道具立てとして
  ハマりがよい。`xs.map(&:foo)` が `map (\t -> t.foo) xs` に
  なる、というだけでもかなりの量の Ruby 慣用が翻訳できる。
- レコードの「閉じた構造的」設計は、Ruby の Hash の柔らかさに
  慣れた人にはやや窮屈に感じる可能性がある。とくに「フィールド
  を後から足す」ができないのは Hash 派には違和感の元。
  ただこれは Elm 0.19 が一度行多相を捨てて閉じた方向に倒した
  経緯（文書 04 §Design notes）に従っているので、現段階では
  そのままで妥当だと思う。
- タプルが無いことは説明上どうしても引っかかる。文書 09 OQ 1
  に上がっている通りで、入門者向けには `(1, 2)` 構文があると
  説明が楽になる。優先度は低いが、いずれ判断が要る。
