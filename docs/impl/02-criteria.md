# 02. 評価基準

ホスト言語候補を比較する軸をここに定義する。各軸に **重み** と
**評価時の具体ポイント** を付す。スコアリングは
`04-matrix.md` で行う。

## 基準セット

### C1. パターンマッチと ADT の表現力（**重み: 高**）

Sapphire は ADT（03）とパターンマッチ（06）を中核に持つ言語。
コンパイラ内部の AST・型・パターン検査器自身が ADT とパターン
マッチを大量に使う。ホスト言語がこれらを **言語機能として** 持
つかどうかは、実装量と可読性に直結する。

評価時のポイント：

- sum type / product type を直接書けるか
- `match` 相当が exhaustive 検査付きで動くか
- 代替（object hierarchy、discriminated union の真似事）で代
  用できるかの読みにくさ
- 再帰データ（AST の子ノード）の扱いやすさ

### C2. コンパイラ実装のエコシステム（**重み: 高**）

言語コンパイラを書くのに役立つライブラリ・先行実装・知見の豊富さ。

評価時のポイント：

- パーサコンビネータ or パーサジェネレータの質
- 既存のコンパイラ実装（その言語で書かれている）から学べる例
- 型検査・SSA・コード生成を支援するライブラリ

### C3. target (Ruby) との相互運用（**重み: 中**）

コンパイラは Ruby コードを出力する（`docs/build/` 参照）。ホスト
言語が Ruby と同じプロセスで動くか、外部プロセスとして起動する
かで、開発体験が変わる。

評価時のポイント：

- Ruby runtime へ埋め込み可能か（Ruby 自体 or RubyGems 経由の
  FFI）
- Ruby コード片（`:=` スニペット）の検査をどう実装するか
- Bundler / gem 配布との親和性

### C4. コントリビューターのアクセシビリティ（**重み: 中**）

Sapphire は Ruby 出身ユーザが主要想定層（user 含む）。コンパイラ
本体への貢献を user と将来の協力者に開いておくには、**Ruby 出身者
が学ぶコストが低い** ことが効く。

評価時のポイント：

- Ruby プログラマの学習曲線
- 構文の読みやすさ（初見で何をしているか分かるか）
- 公式の学習リソース（日本語資料の有無を含む）

### C5. 型システムの強さ（**重み: 中**）

コンパイラ実装は型エラーが多発する領域（AST の変換、型推論、
elaboration）。ホスト言語の型システムがバグをどれだけ防げるかで、
実装品質が変わる。

評価時のポイント：

- 静的型か動的型か
- パターンマッチの exhaustiveness 検査（C1 と重なるが独立評価
  可）
- null / undefined の扱い（Maybe 相当が native か）
- 不変性・所有権などのハザード避け

### C6. パフォーマンス（**重み: 低**）

初回実装では重要度低。コンパイラ速度が UX 上問題になるのは、
プロジェクトが大きく育ってから。ただし「インタプリタ言語
ホスト」を選ぶと将来的な悩みになる。

評価時のポイント：

- 起動時間（1 ファイル compile の UX）
- バッチコンパイルのスループット
- メモリ占有

### C7. 配布・インストール（**重み: 中**）

ユーザが Sapphire コンパイラを手元に入れる手順。Ruby コミュニ
ティが相手なので、**`gem install sapphire` に近い体験** が取れ
るかどうかで導入障壁が変わる。

評価時のポイント：

- ビルド済みバイナリの提供可否（クロスコンパイル等）
- OS 別の手間
- 依存ランタイム（JVM・.NET・Node.js など）の有無

### C8. Self-hosting / Bootstrap の将来性（**重み: 低**）

長期的に Sapphire で書いた Sapphire コンパイラ（self-host）を作
れるか。最初の実装では関係ないが、後から乗り換えるコストが大き
いホスト言語は避けたい。

評価時のポイント：

- 初期言語で書いたコンパイラを、将来 Sapphire で書き直す
  ときの辛さ
- 中間表現（IR）の設計自由度

### C9. 長期保守性 / コミュニティの勢い（**重み: 中**）

今後 5〜10 年、ホスト言語自体が停滞・縮小して maintenance が困
難になるリスクを評価する。bus factor の低い言語に乗ると後戻りが
難しい。

評価時のポイント：

- 言語自体の開発活動（定期リリース、近年の主要機能追加）
- 商用バックアップの有無
- パッケージレジストリの健全性（サプライチェインのセキュリティ
  を含む）

### C10. ライセンス / 法務の親和性（**重み: 低**）

ホスト言語のツールチェイン・標準ライブラリ・代表的依存の
ライセンスが、Sapphire のバイナリ配布や将来の商用利用と干渉し
ないか。

評価時のポイント：

- ランタイム同梱のライセンス（GPL 伝搬の有無等）
- 標準ライブラリとパッケージレジストリ主流のライセンス
- 配布バイナリの再ライセンスが必要になるパターン

## 重みの扱い

重みは目安（高 = 3、中 = 2、低 = 1）。総合スコアはあくまで
**議論の出発点** で、最終判断は user の判断。

現状の重み割り当て：
- 高 (×3)：C1 / C2
- 中 (×2)：C3 / C4 / C5 / C7 / C9
- 低 (×1)：C6 / C8 / C10

`04-matrix.md` では各候補に対し C1〜C10 の **5 段階評価**（1〜5）
を入れ、総合は `Σ (重み × スコア)` で算出する。
