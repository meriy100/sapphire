# 06. パターンマッチ

状態: **draft**。prelude (M6) による `Bool` とリスト構文の確定に
伴い改訂されうる。

本文書は `docs/spec/06-pattern-matching.md` の日本語訳である。英語
版が規範的情報源であり、BNF 生成規則・型付け規則・規則名・番号付き
未解決の問いは英語版と一致させて保つ。

## 動機

本文書は文書 03 以来 Sapphire に欠けていた消去形を導入する。これ
まではデータ値とレコードを **構築** することはできたが分解はでき
なかった。`case ... of` はこの欠落を埋め、パターンマッチをデータ
コンストラクタとレコードの両方に通す。

範囲内：

- `case ... of` 式構文、および 02 の `of` とのレイアウト相互作用。
- パターン言語：ワイルドカード、変数、リテラル、コンストラクタ、
  レコード、`@`-パターン、cons パターン、括弧付き型注記。
- パターンの型付け、環境の拡張、`case` 式の型付け。
- 網羅性・冗長性検査。

決着させる：

- 03 未解決の問い 5（コンストラクタ名の名前空間とシャドウイング）—
  `upper_ident` は式位置・パターン位置ともに常にコンストラクタで
  あり、パターン先頭の `lower_ident` は常に束縛である。コンストラ
  クタ名を値束縛で覆うことは静的エラー。
- 05 の「パターン位置型注記」（05 `::` 曖昧性の後半）— 綴りは
  `(pat : type)` で確定。

部分的に決着：

- 01 未解決の問い 3（`if` はプリミティブか糖衣か）— 糖衣展開のため
  の機構は本文書で利用可能になるが、(If) を
  `case c of { True -> t ; False -> f }` に畳むかどうかは M6 の
  `Bool` 決着に依存する。§設計メモ を参照。

保留：

- ガード節（`pat | guard -> e`）。06 未解決の問い 1。
- リストリテラルパターン `[p₁, ..., pₙ]` の糖衣。綴りのみ予約、実
  体は M6 のリストを待つ。06 未解決の問い 3。
- or パターン（`pat₁ | pat₂ -> e`）。06 未解決の問い 2。
- view パターン・active パターン。計画しない。

## 抽象構文（BNF）

文書 01・03・04・05 を拡張：

```
expr    ::= ...                                   -- (01, 03, 04, 05)
          | 'case' expr 'of' case_alts            -- (新)

case_alts  ::= case_alt (';' case_alt)*           -- レイアウト依存；
                                                  -- §レイアウト 参照

case_alt   ::= pat '->' expr

pat     ::= apat
          | upper_ident apat+                     -- 引数を持つコンス
                                                  -- トラクタパターン
          | pat '::' pat                          -- cons パターン
                                                  -- (第 5 位、右結合)

apat    ::= '_'                                   -- ワイルドカード
          | lower_ident                           -- 変数束縛
          | lower_ident '@' apat                  -- @-パターン
          | literal                               -- リテラルパターン
          | upper_ident                           -- 0 引数コンストラクタ
          | '{' field_pats? '}'                   -- レコードパターン
          | '(' pat ')'                           -- グループ化
          | '(' pat ':' type ')'                  -- 型注記付き

field_pats ::= field_pat (',' field_pat)*
field_pat  ::= lower_ident '=' pat
```

`apat` は **原子パターン** クラスである。コンストラクタの引数には
`apat` を要求する。それより複雑なパターン（自身に引数を持つコンス
トラクタパターン、cons パターン）をコンストラクタ引数の位置に置く
ときは括弧で囲む必要がある。例：`Just (Cons 1 xs)` は中身が `Cons`
である `Just` に一致し、`Just Cons 1 xs` は構文エラー（その位置の
`Cons` はせいぜい 0 引数コンストラクタとしか解釈できない）。

リテラルパターン中の `literal` は文書 01 のリテラル文法（`INT`・
`STRING`・`BOOL`）を再利用する。`True` / `False` がリテラルパター
ンかコンストラクタパターンかは M6 における 02 未解決の問い 1 の決
着に依存する。いずれの読みでも表層形は一致する。

### パターンにおける `::` の優先度

cons パターンは `::` 式演算子（05 §演算子表）と同じ優先度・結合性
を持つ：第 5 位、右結合。`x :: y :: zs` は `x :: (y :: zs)` として
解析される。本形式は M3 で予約されるが、M6 で `List` が導入される
まで使用不能。

### 型注記付きパターン

`(pat : type)` はサブパターンに型注記を付ける。これは 05 で約束し
た `::` の代替綴りである：05 は `::` をリスト cons に固定したため、
パターン位置の型注記は既に予約されている `:` 区切りを括弧内で再利
用する形となる。注記はチェックのヒントであり、パターンがどの値に
一致するかは変えず、マッチした束縛に与えられる型のみを定める。

### 本層では名前付きフィールドコンストラクタパターンは採らない

コンストラクタパターン `K apat₁ apat₂ ...` は位置引数である。レ
コード構文は `{ ... }` レコードパターン形でのみ現れ、値の推論型が
レコード（04）である場合にのみ使える。`data` コンストラクタが名
前付きフィールド payload を宣言できるか
（`data Shape = Circle { cx : Int, cy : Int, r : Int }`）、したがっ
てその場合に `Circle { cx = p₁, cy = p₂, r = p₃ }` でマッチできる
かは 04 未解決の問い 2 のままである。それが決着するまで M3 は位置
引数のコンストラクタパターンのみ認める。

## パターンの型付け

パターンの型付けは **双方向** 判断

    Γ ⊢ p ⇐ τ ⊣ Γ'

である。読みは「Γ のもとでパターン `p` は型 `τ` に照合され、拡張
された環境 `Γ'` を生む」。拡張 `Γ'` は `Γ` に `p` が導入する
`lower_ident` への新鮮な束縛を加えたもの。パターンが既存の束縛を
覆うことはない（§設計メモ 参照）。

### 原子パターン

```
——————————————————————————    (PWild)
 Γ ⊢ _ ⇐ τ ⊣ Γ


        x ∉ dom(Γ)
——————————————————————————    (PVar)
 Γ ⊢ x ⇐ τ ⊣ Γ, x : τ


 Γ ⊢ x ⇐ τ ⊣ Γ₁     Γ₁ ⊢ p ⇐ τ ⊣ Γ₂
—————————————————————————————————————    (PAs)
        Γ ⊢ x@p ⇐ τ ⊣ Γ₂


         ℓ はリテラル型 τ₀ を持つ
—————————————————————————————————————    (PLit)
       Γ ⊢ ℓ ⇐ τ₀ ⊣ Γ
```

(PLit) はリテラルの型が照合する型と等しいことを要求する。自由に
単一化はしない。整数リテラルパターン `3` は `Int` 型の値とのみ一致
する。`String` 型の値に対して用いれば型エラー。

### コンストラクタパターン

```
  C : ∀ a₁ ... aₙ. τ₁ → ... → τₖ → T a₁ ... aₙ    ∈ Γ
  τ は T σ₁ ... σₙ と単一化する（スキーム中の αᵢ に対応する
  新鮮な σᵢ）
  各 i ∈ {1..k} について：Γᵢ₋₁ ⊢ pᵢ ⇐ τᵢ[σⱼ/aⱼ] ⊣ Γᵢ
  —————————————————————————————————————————————————————    (PCon)
          Γ₀ ⊢ C p₁ ... pₖ ⇐ τ ⊣ Γₖ
```

ここで `Γ₀ = Γ` が出発点、各 `Γᵢ` は単調に拡張される。コンストラ
クタのスキームは 03 の `data` 宣言が導入したものである。

0 引数コンストラクタパターン（`k = 0`）はスキームをインスタンス化
するがサブパターンを消費しない。`Γᵢ₋₁ ⊢ pᵢ ⇐ ...` の前提が 0 個で
ある (PCon) の特殊ケース。

### レコードパターン

```
  τ は { f₁ : τ₁, ..., fₙ : τₙ } と単一化する（各 fᵢ は相異）
  { fⱼ₁, ..., fⱼₘ } ⊆ { f₁, ..., fₙ }（各 fⱼᵢ は τ に存在）
  各 i ∈ {1..m} について：Γᵢ₋₁ ⊢ pⱼᵢ ⇐ τⱼᵢ ⊣ Γᵢ
  —————————————————————————————————————————————————————————   (PRec)
   Γ₀ ⊢ { fⱼ₁ = pⱼ₁, ..., fⱼₘ = pⱼₘ } ⇐ τ ⊣ Γₘ
```

レコードパターンはパターンが関心を持つフィールドのみを挙げる。
レコード型の全フィールドに言及する **必要はない** — これは 04 の
レコード **リテラル** 形式（全フィールド必須）との意図的な非対称
である。マッチしたレコードはパターンが無視する追加のフィールドを
持ってよい。レコード型は完全なフィールド集合であり、投影ではない。

合理性。すべてのパターンですべてのフィールドを書くのは人間工学的
コストが大きく、04 のリテラル形式との非対称のコストよりはるかに
重い。典型的なパターン利用は「1〜2 フィールドだけに関心がある」
である。

04 が **クローズドな構造的レコード**（行変数なし）を採ったため、
(PRec) が発火する時点で被検査型 `τ` は全フィールド集合
`{ f₁, ..., fₙ }` を列挙できる程度に具体化されている必要がある。
レコードパターン単独で被検査型の形を駆動することはできない；形は
型注記・呼び出し先・周囲の証拠から来なければならない。これを緩め
るには行多相（04 未解決の問い 1）が必要である。

### cons パターン

```
  τ は List α と単一化する（新鮮な α）
  Γ₀ ⊢ p ⇐ α ⊣ Γ₁
  Γ₁ ⊢ q ⇐ List α ⊣ Γ₂
  ———————————————————————   (PCons)
  Γ₀ ⊢ p :: q ⇐ τ ⊣ Γ₂
```

M6 が `List` を導入すれば使用可能。M3 では予約のみ。

### 注記

```
  Γ ⊢ p ⇐ τ ⊣ Γ'    τ = τ_ann
  ———————————————————————————————   (PAnn)
  Γ ⊢ (p : τ_ann) ⇐ τ ⊣ Γ'
```

## case 式の型付け

```
  Γ ⊢ e : τ_s
  各 i ∈ {1..n} について：
      Γ ⊢ patᵢ ⇐ τ_s ⊣ Γᵢ
      Γᵢ ⊢ eᵢ : τ_r
  パターン (pat₁, ..., patₙ) が集合的に τ_s を網羅する
  —————————————————————————————————————————————————————————————   (Case)
  Γ ⊢ (case e of pat₁ -> e₁ ; ... ; patₙ -> eₙ) : τ_r
```

各アームは同じ結果型 `τ_r` を生まなければならない。パターンは
**被検査型** `τ_s` に対してチェックされる。`τ_s` の値に一致し得
ないパターンは型エラーであり、無音の dead アームにはならない。

網羅性は型付けの前提であって別パスの解析ではない。§網羅性と冗長
性 参照。

## `case` のレイアウト

02 は `of` をブロック開始キーワードとして予約した。本文書はその
予約を有効化する。`case e of` の直後のトークンが `{` でないとき、
レイアウトブロックが開く。参照列 `c` は最初の `pat` トークンの列
である。以後、列 `c` にちょうど位置するトークンは新しい `case_alt`
を開始し、`c` より大きい列のトークンは現アームを継続し、`c` より
小さい列のトークンはブロックを閉じる。

例：

```
case maybeX of
  Nothing ->
    0
  Just x ->
    x + 1
```

`case` ブロックの参照列は `Nothing` の列。`0` と `x + 1` のインデ
ントは各アームの継続である。最後のアームは、参照列より小さい列に
次のトークンが来た時点（または入力終端）で閉じる。

明示的な波括弧形も常に利用可能：

```
case maybeX of { Nothing -> 0 ; Just x -> x + 1 }
```

波括弧内ではレイアウト規則が無効化され、アームはセミコロンで区切
る（02 §レイアウト）。

両方使われた場合 — `of` 直後に明示的 `{` — レイアウトブロックは開
かず、対応する `}` までがブレース形のブロックとなる。

## 網羅性と冗長性

`case` 式が **網羅的** であるとは、被検査型の任意の値に対して、
リスト中に少なくとも一つ一致するパターンが静的に存在することを言
う。非網羅的な `case` 式は本層では **静的エラー** であり、警告で
はない。

- 閉じた ADT `data T ... = C₁ ... | Cₘ ...` について、`case` が網
  羅的なのは、各 `Cᵢ` が少なくとも一つのパターン（ワイルドカード
  や変数パターンを通じてでもよい）でカバーされているとき。
- レコードに対しては網羅性は自明（レコード型は一形のみ）。指定し
  た各フィールドのサブパターンが網羅的であれば、任意のレコード値
  がレコードパターンに一致する。
- リテラル型（`Int`・`String`）の場合、網羅性のためには最終的な
  catch-all（ワイルドカードまたは変数パターン）が必要。整数リテ
  ラルを有限個列挙し fallback が無い `case` は網羅的でない。

`Bool` は M6 が 02 未解決の問い 1 を決着させるまで、リテラルと
ADT の境界に跨る。M6 が `Bool` を `data Bool = False | True` と
して確定すれば、`case b of { False -> ... ; True -> ... }` は
上の第一項（閉じた ADT）により網羅的である。M6 が `True` / `False`
を独立した `BOOL` リテラルクラスのままにすれば、`Bool` は現文の
第三項における非列挙リテラル型となり、catch-all が要求されること
になる。ADT の読みが好ましく、06 はその前提の下で起草している
が、現文言はその選択を強制しない。

**無人** ADT `data Void = `（コンストラクタ 0 個）は `Void` 型の
被検査値について空虚な意味で網羅的である。文法
`case_alts ::= case_alt (';' case_alt)*` が少なくとも 1 アームを
要求するため、`case e of { }` は書けない。実務上の帰結は `Void`
値を直接 `case` できないことである。`Void` 値は構築不可能なので、
`Void` を消去する必要がある型検査済みパスは既に到達不能である。
教育目的で空 `case_alts` 形を導入するかは 06 未解決の問い 7。

パターンが **冗長** であるとは、そのパターンが一致しうるあらゆる
値が、同じ `case` 中の先行パターンで既に一致されていることを言
う。冗長パターンは本層では診断レベル（警告）であり、エラーではな
い。除去は義務ではないが常に安全。

静的エラー（警告ではなく）とする理由。Sapphire は `.sp` ロジッ
クを埋め込む生成 Ruby モジュールを対象とする。実行時の非網羅的マ
ッチは Ruby 側の黙示的例外に落ち、モジュール境界での静的型検査
の目的を損なう。非網羅性をエラーへ昇格させる摩擦は引き合う。

## 設計メモ（非規範的）

- **名前空間規律（03 未解決の問い 5 を決着）。** 式位置でもパター
  ン位置でも `upper_ident` は常にコンストラクタ参照である。パタ
  ーン先頭の `lower_ident` は常に束縛である。`let Just = 1 in
  Just` は静的エラー（LHS の `Just` は `upper_ident` なので束縛
  にはできず、コンストラクタに予約されている）。どのスコープに
  おいてもコンストラクタの再束縛は認めない。これによりシャドウ
  イングの副問いと、パターン中の `{ x }` が「`x` を束縛」か「`x`
  に照合」かの問いが同時に閉じる — 常に前者である。

- **レコードパターンは構造的かつ部分参照可能。** レコードリテラ
  ルと違い、レコードパターンはレコードのフィールドの部分集合を挙
  げてよい。パターン `{ x = 0 }` は `x` が `0` である任意の
  `{ x : Int, ... }` に一致する。04 のクローズド構造的レコード決
  定と組み合わさり、これはマッチした束縛の **推論型** が依然とし
  てレコードの全フィールドを持つことを意味する — パターンは完全
  な型上のフィルタであって、より小さな型への射影ではない。より小
  さなレコード型への射影には行多相（04 未解決の問い 1）が必要。

- **`if` を `case` の糖衣にする（01 未解決の問い 3 を部分的に決
  着）。** `case` が使えるようになり、
  `if c then t else f` ≡ `case c of { True -> t ; False -> f }`
  の機構は整った。(If) を実際にこの糖衣に畳むかは M6 における 02
  未解決の問い 1 の決着に依存する：`Bool` が `data` ADT として定
  義されれば、糖衣の読みが正式となる。`Bool` が字句的に独立のま
  ま（`True` / `False` が専用 `BOOL` トークン）なら、糖衣は使え
  ず (If) はプリミティブのまま残る。M3 はこれを一方的に決めない。

- **`Int` / `String` に対するリテラルパターンには catch-all が必
  要。** 非網羅性を静的エラーとすることは、列挙的でない型の
  `case` ではワイルドカードまたは変数パターンを書くことを強制す
  る。これは意図的な人間工学である：構造的に（ADT で）マッチする
  か、既定値を与えるかのいずれかを求める。

- **ガード、or パターン、view パターンは保留。** 本層のパターン
  言語は意図的に小さく保つ。人間工学的拡張（ガード、or パターン）
  は未解決の問い、表現力拡張（view パターン、active パターン）は
  計画しない。

- **パターン型付けは宣言的。** 双方向判断 `Γ ⊢ p ⇐ τ ⊣ Γ'` は型
  検査アルゴリズムとしてそのまま読めるが、本文書は特定の実装戦略
  にコミットしない。(PCon) と (PRec) で単一化を行うアルゴリズム
  W 風のソルバで十分である。

## 未解決の問い

1. **ガード。** `pat | guard -> e` アームを認めるか（`guard` は
   パターン一致成功後に評価される真偽値式、偽ならば次のアームへ
   落ちる）。Elm はガードを持たず、Haskell は持つ。draft はガード
   なし。`if/else if` のネストによる回避は当面許容。

2. **or パターン。** `pat₁ | pat₂ -> e` を、どちらのサブパターン
   でも発火する単一アームとして認めるか。ガード（構文で `|` を再
   利用）および変数束縛（or パターンの両側で同じ変数名集合を両立
   可能な型で束縛しなければならない）と相互作用する。draft は否
   定。

3. **リストリテラルパターン。** `[]`、`[x, y, z]`、`[x, y | rest]`
   のパターン糖衣。綴りは予約済み（02 §演算子トークンが `[` と
   `]` を予約）。実体は M6 のリスト構文を待つ。draft では `::`
   cons パターンと（M6 着地後は）0 引数 `Nil` のみが使える。

4. **`Int` / `String` パターンの網羅性。** 現規則は列挙でないリ
   テラル型に catch-all を要求する。将来の代替：range パターン
   （`1..10`）を認めて一般的ケースでの catch-all 要求を減らす。
   現時点では計画しない。

5. **パターン付き let 束縛。** `let pat = e in body`（Elm・OCaml・
   Haskell）形式。現文法は `let lower_ident = e` のみ。`let` の
   LHS をパターンに拡張するのは自然な拡張だが M3 では必須ではな
   い。M10 か follow-up 文書に送る。概念的には単一の refutable
   パターンに対する `case e of { pat -> body }` と等価。
