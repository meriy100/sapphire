# 05. 演算子と数値

状態: **draft**。M3（パターンマッチ）、M6（prelude）、および後続のアド
ホック解決機構の着地に伴い改訂されうる。

本文書は `docs/spec/05-operators-and-numbers.md` の日本語訳である。
英語版が規範的情報源であり、BNF 生成規則・演算子表・型付け規則・規
則名・番号付き未解決の問いは英語版と一致させて保つ。

## 動機

文書 01・03・04 はいずれも演算子を確定させないまま仮定してきた。
01 の動機例はすでに `>` と `+` を用い、03 の ADT 値は算術なしでは
演習できず、04 のフィールド選択例も数値演算と組み合わせてこそ有用
である。本文書は以下を固定する：

- 本層で認める **組み込み演算子** の集合、**優先度**、**結合性**、
  **型**。
- **単項マイナス** の扱い。
- 本層の **数値タワー**。
- 02 の `::` トークンの用途。

以下を決着させる：

- 01 未解決の問い 4（数値タワー）— 部分的：本層は `Int` のみ。
  `Float` および統一 `number` 変数は 05 未解決の問い 1 に再度立て
  る。
- 01 未解決の問い 5（組み込み演算子）— 演算子は prelude 束縛の原
  始値とし、以下に固定した単相型を与える。
- 02 未解決の問い 2（単項マイナス）— 単項 `-` は `negate` の表層
  糖衣。
- 02 未解決の問い 3（演算子表）— Elm 風固定表を採用。ユーザー定義
  fixity は 05 未解決の問い 3 に再度立てる。
- 02 未解決の問い 6（`::` 曖昧性）— `::` はリスト cons（実体は
  M6）。パターン位置の型注記は別の綴りを採り、文書 06 で
  `(pat : type)` に確定した。

## 抽象構文（BNF）

文書 01 を拡張（03・04 とは直交的に両立）：

```
expr   ::= ...                                 -- (01, 03, 04)
         | expr op expr                        -- 二項適用、§演算子表 に
                                               -- 従って解析
         | '-' expr                            -- 単項マイナス、前置
         | '(' op ')'                          -- 関数としての演算子

op     ::= '+' | '-' | '*' | '/' | '%'
         | '<' | '>' | '<=' | '>=' | '==' | '/='
         | '&&' | '||'
         | '++'
         | '::'
```

上記の `op` 集合は文書 02 の `op` 非終端の **真部分集合** である：
02 は `op_char` の列をどうトークン化するか（最大貪欲）を定め、本文
書は式層における演算子として実際にどのトークンを認めるかを定める。
この集合に含まれない `op_char` 列が式位置に現れた場合は本層では
パース時エラー。他の列を後続文書のために予約する措置は変わらない。

二項適用の生成規則 `expr op expr` は文脈自由文法としては **曖昧**
である。§演算子表 の優先度・結合性で解釈を一意に定める。例えば
`e₁ + e₂ * e₃` は表に従ってちょうど一通り（`e₁ + (e₂ * e₃)`）に
解析される。

`-` トークンは二項生成規則 `expr op expr`（`op` の一員として）と
単項生成規則 `'-' expr` の両方に現れる。両者は衝突しない：下記
§単項マイナス が単項として解釈される構文位置を厳密に定めており、
それ以外の位置では必ず二項として解釈される。

関数適用はすべての二項演算子と単項マイナスより強く結合する。
`f 3 + g 5` は `(f 3) + (g 5)` として解析される。

## 演算子表

| 優先度     | 結合性    | 演算子                          | 型                              |
|------------|-----------|---------------------------------|---------------------------------|
| (app)      | 左        | 関数適用                        | —                               |
| 9          | 前置      | `-`（単項）                     | `Int -> Int`（`negate` として） |
| 7          | 左        | `*`  `/`  `%`                   | `Int -> Int -> Int`             |
| 6          | 左        | `+`  `-`                        | `Int -> Int -> Int`             |
| 5          | 右        | `++`                            | `String -> String -> String`    |
| 5          | 右        | `::`                            | `∀ a. a -> List a -> List a` *（後述参照） |
| 4          | 非結合    | `==`  `/=`  `<`  `>`  `<=`  `>=` | `Int -> Int -> Bool`           |
| 3          | 右        | `&&`                            | `Bool -> Bool -> Bool`          |
| 2          | 右        | `\|\|`                          | `Bool -> Bool -> Bool`          |

`::`（表の `*` 印）は本層では優先度と型のみ予約する。`List` 型と
`::` の実体束縛は M6 で導入する。M6 が着地するまで `::` はトーク
ン化され予約されるが、式としては使用できない。パイプ演算子（05
未解決の問い 5。こちらはまだ優先度エントリを予約しない）との非対
称は意図的である：パイプ演算子はまだ約束されていないのに対し、
リスト cons は M6 のほぼ確定した export である。

関数適用の実効優先度は 9 位（単項マイナス）より強く結合する。参考
として上の表に載せた。

第 4 位の比較演算子は **非結合**：`a < b < c` は本層では構文エラー
である。明示的な括弧を要求する。

同じ優先度・同じ結合性の演算子は同一グループを成す：第 5 位右結
合の `x :: xs ++ ys` は `x :: (xs ++ ys)` として解析される。これ
は型とは独立のパース時の事実である（`xs ++ ys` は `::` を満たす
ために `List a` 型が要るが、本層の `++` は `String` 型なので、`::`
が M6 で具体化された後、この例は型エラーになる）。異なる優先度で
のグループ化は通常の precedence-climbing の振舞いである。

本層のすべての演算子は **単相**。特に `==` と `/=` は `Int -> Int
-> Bool` に限定する。これらおよび順序比較を他の型にも広げる件は
05 未解決の問い 2。

## 単項マイナス

単項 `-` は M6 で導入される prelude 束縛 `negate : Int -> Int` の
**表層糖衣** である。パーサが `-` を単項と認識した位置では、`-e`
は `negate e` の省略形として扱われる。

構文位置。`-` が単項演算子となるのは **式開始位置** に置かれた場
合のみ：トップレベル定義または `let` 束縛の右辺の先頭、あるいは
`(`・`{`（更新形 `{ e | ... }` で `{` の直後に式が来る場合）・`=`・
`->`・任意の二項演算子・キーワード `if`・`then`・`else`・`in` の
直後。それ以外の位
置の `-` は二項減算（第 6 位）である。

これまでに公開された文書（01 / 03 / 04）が上記の位置を定義して
いる。後続文書はこの一覧を単調に拡張する：M6 はリストリテラル着
地に伴い `[` を加え、M3 は `case`・`of`・`case` 式内のパターン
分岐の `->` を加える。

優先度。単項 `-` は第 9 位：すべての二項演算子より強く、関数適用
より弱く結合する。従って

    -3 * 2   解析は   (negate 3) * 2         = -6
    -f 5     解析は   negate (f 5)
    -a - b   解析は   (negate a) - b

`-` を直後の整数リテラルに字句融合する規則は存在しない。`-3` は 2
トークン（`-` 演算子と整数リテラル `3`）である。負の整数リテラル
が字句形式ではないとする 02 の決定は変更しない。

## 数値タワー

本層の数値タワーは単一の基底型 **`Int`** のみを持つ。算術・比較・
単項マイナスの各演算子はすべて演算子表の通り `Int` 上で単相的に型
付けされる。

これにより 01 未解決の問い 4 は本層では決着する。「`Float` を第 2
の基底型として追加する」か「Elm 流の `number` 制約変数で `Int` /
`Float` を統一する」かは 05 未解決の問い 1 に委ねる。いずれの道も
本層の単調拡張として両立する：`Float` を加えるなら `Int` 用のエン
トリに `Float` 用のエントリを追加するだけで済み、`number` の道は
本文書が提供しない制約解決機構が必要になる。

## 型付け規則

§演算子表 の各演算子は、prelude が供給する初期環境 Γ₀ に、表に示し
た型を持つ **原始束縛** として寄与する。中核言語レベルではこれで
十分：二項形 `e₁ op e₂` は通常の適用 `(op) e₁ e₂` として型付けさ
れ、単項 `-e` は `negate e` として型付けされる。新しい判定形式は
不要であり、01 の (Var)・(App)・(Abs) で足りる。

関数化形 `(op)` は対応する prelude 束縛のスキームを持つ原子式であ
り、01 の (Var) で型付けされる。

以下の 2 規則は **派生であってプリミティブではない** — 表層で型付
けを明示するために書き下すに留まる。規範的な内容は prelude 束縛
に対する 01 の (Var) + (App) である。

```
   (op) : τ₁ → τ₂ → τ₃ ∈ Γ    Γ ⊢ e₁ : τ₁    Γ ⊢ e₂ : τ₂
   ————————————————————————————————————————————————————————   (BinOp)
                   Γ ⊢ e₁ op e₂ : τ₃


   negate : Int → Int ∈ Γ    Γ ⊢ e : Int
   ——————————————————————————————————————                     (UMinus)
             Γ ⊢ -e : Int
```

## `::` 曖昧性の解決

演算子表の通り、`::` は **リスト cons**。第 5 位右結合、型
`∀ a. a -> List a -> List a`。`List` 型と `::` 束縛の実体は M6 で
導入する。本文書はトークンとその優先度・意図した型を予約するのみ
である。

パターン位置の型注記は別の綴りを用いる：`(pat : type)` — 括弧で
包んで既存の `:` 注記を再利用する形。これは文書 06（パターンマッ
チ）で確定した。この選び方は、典型的な Sapphire プログラムで `::`
のほうがパターン位置型注記より頻出するという判断に基づく。

## 設計メモ（非規範的）

- **演算子は prelude 束縛の原始値であり、字句的プリミティブでは
  ない。** 字句解析 (02) が演算子をトークン化する。本文書は式層で
  どのトークンを演算子とするかを制限し、その型を固定する。実際の
  束縛 — 記号名 `+`・`*`・`==`・`negate` — は prelude (M6) に存
  在する。通常の prelude 値として扱うことで中核式言語の一様性が
  保たれる：`(+) 3 5` と `3 + 5` は型が等しく、（解析を除けば）評
  価も等しい。

- **本層での等価は Int のみ。** `==` / `/=` を 1 ステップで文字
  列・真偽値・ユーザー定義データ型にまで広げるには、型クラス機構
  （01 未解決の問い 5 のもう一方の枝）または実行時の普遍等価プリ
  ミティブが必要になる。draft は狭い道を採り、05 未解決の問い 2
  で再考する。

- **演算子セクションは採らない。** Sapphire は本層で `(+ 5)` や
  `(5 +)` の部分適用を認めない。`\x -> x + 5` と書く。セクション
  は単項マイナスとの相互作用が非自明（`(- 5)` は「右オペランドが
  `5` の二項 `-` のセクション」か「単項マイナス適用 `negate 5`」
  か曖昧）で、05 未解決の問い 4 を参照。

- **ユーザー宣言 fixity は採らない。** Elm 風固定表を採る。Haskell
  の `infixl N` 宣言は未解決の問い 3。

- **`-` 周辺の空白に字句的意味はない。** 単項 / 二項の区別はパー
  サレベルで、`-` が式開始位置にあるかどうかで決まる。`a - b`・
  `a-b`・`a -b` は同じトークン列である。パース結果が異なるとした
  らそれは `-` の直前に来るものが何かによる（3 者とも `a` の後で
  あり、いずれの場合も `-` は二項）。

- **比較の非結合性。** `a < b < c` は受理せず拒絶する。`(a < b) < c`
  と解釈すれば型エラー（比較結果は `Bool` であって `Int` ではな
  い）になる。連鎖比較 `a < b ∧ b < c`（Python のような）を Sapphire
  は採らない。第 4 位を非結合とすることで、後の型エラーよりはっき
  りしたパースエラーにする。

## 未解決の問い

1. **`Float` と数値多相。** `Float` を別の基底型として認めるか、
   Elm 流 `number` 制約変数で統一するか。`number` の道は 01 未解
   決の問い 5 の型クラス枝が導入する制約機構を必要とし、`Float`
   の道は演算子エントリの追加で足りる。draft は `Int` のみ。

2. **構造的型に対する多相等価 / 順序比較。** `==`・`/=`・`<`・`>`・
   `<=`・`>=` を「等価 / 順序を許す」任意の型に広げるか（Elm の実
   行時制約、関数型を除外する方式）。肯定の答えは型クラス機構か
   実行時普遍等価プリミティブに依存する。肯定の場合でも関数矢印
   を含む型は `==` / `/=` から除外されなければならない（Elm の既
   定規則）：関数の外延的等価は決定不能であり、関数値に対して黙
   って `False` を返す普遍等価プリミティブは落とし穴になる。draft
   は `Int` のみで単相。

3. **ユーザー宣言演算子 fixity。** `infixl N` / `infixr N` 宣言
   （Haskell）を認めるか。draft は Elm 風固定。肯定の答えは M4
   （モジュール）と相互作用する：インポートされた演算子はインポー
   ト先で一貫した fixity を持つ必要がある。

4. **演算子セクション。** `(+ 5)` / `(5 +)` の部分適用。単項マイ
   ナス `(- 5)` との相互作用が非自明であり、別の綴りを採らない限
   り否定方向に倒れる。draft は否定。

5. **パイプ演算子。** Elm の `|>`（左→右）と `<|`（右→左）。便利
   だが本文書の主題ではないため保留。自然には第 0 / 1 位の右 / 左
   結合エントリになる。

6. **冪乗 `^`。** `^ : Int -> Int -> Int` を `*` より高い位に入れ
   るか。draft は含めない。利用者は乗算の反復か prelude 関数を用
   いる。
