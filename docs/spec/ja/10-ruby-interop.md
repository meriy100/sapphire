# 10. Ruby インタロップ — 埋め込みとデータモデル

状態: **draft**。M8（Ruby 評価モナド）と M9（例題プログラム）が
境界を使い込む過程で改訂されうる。

本文書は `docs/spec/10-ruby-interop.md` の日本語訳である。英語版
が規範的情報源であり、BNF 生成規則・データモデル表・番号付き未解
決の問いは英語版と一致させて保つ。

## 動機

Sapphire は Ruby へのコンパイルを想定する。その目的は 2 つの異な
る関心を帯びる：

1. **Sapphire から Ruby を呼ぶ。** Sapphire プログラムは Ruby コ
   ードのスニペットを埋め込み、その結果は Sapphire のパイプライン
   へ戻る。
2. **Ruby から Sapphire を呼ぶ。** 生成された Ruby モジュールが
   Sapphire のエクスポート束縛を露出し、周囲の Ruby アプリケーシ
   ョンからの利用を可能にする。

本文書は両方向のデータモデル側 — Sapphire 値と Ruby 値の対応、お
よびエラーの流れ — を固定し、Ruby ソースを埋め込む表層構文を導入
する。埋め込まれた Ruby コードの評価意味論は固定 **しない**；そ
れは文書 11（Ruby 評価モナド）の範疇。本文書はモナドを不透明型
`Ruby` として扱う。11 が命名と完全な規定を行う。

範囲内：

- 埋め込み形式：本体が Sapphire 式ではなく Ruby ソーススニペット
  である定義形（02 で予約された `:=` 句読点を回収する）。
- 複数行 Ruby ソース用の三連引用符文字列リテラル（02 文字列リテ
  ラル文法への M7 拡張）。
- 各 Sapphire 型（`Int`・`String`・`Bool`・レコード・ADT・関数・
  `Maybe`・`Result`・`List`・`Ordering`、および不透明 `Ruby a`
  ラッパ）のマーシャリング規則。
- Ruby 側の例外モデル — Ruby で上がった例外が Sapphire 側にどう
  現れるか。
- 生成 Ruby モジュールの形と命名。

範囲外（文書 11）：

- Ruby アクションを連鎖させたときの評価順序。
- Ruby 実行のスレッド／並行モデル。
- `Ruby` の `return` / `pure` / `>>=`。
- `run` 関数と `Monad Ruby` インスタンス。

## 埋め込み形式

文書 01・07・09 を拡張：

```
decl ::= ...                                      -- (01, 07)
       | IDENT lower_ident* ':=' ruby_string      -- Ruby 埋め込み

ruby_string ::= triple_string                     -- §三連引用符文字列
                                                  -- リテラル 参照
              | string_lit                        -- 02 の単行 Ruby
```

Ruby 埋め込み束縛は次の形を取る：

    name p₁ p₂ ... pₙ := ruby_source

これは 07 の `clause` 形 `name p₁ ... pₙ = expr` の Ruby インタ
ロップ版 — ただしパラメータは単純な `lower_ident` に限定される
（ワイルドカードや分解パターンは不可）。分解が必要なら、包む純粋
関数の側で行う。`:=` 句読点（02 がまさにこの用途のために予約）は、
右辺が Sapphire 式ではなく Ruby ソース文字列であるときに、純粋な
`=` を置き換える。

束縛の型はスコープに明示的シグネチャで **必ず** 示さなければなら
ない（束縛の前に `:` 形で置く）。08 の境界規則はエクスポートされ
た束縛について既にこれを要求するが、10 はマーシャリング境界が参
照する型として、エクスポートの有無によらず **すべての** `:=` 束
縛にシグネチャを要求する。シグネチャの結果型は（さらに `-> τ'` の
矢印に包まれていてもよいが）最終的に `Ruby τ` の形でなければなら
ない。`:=` 束縛で宣言された結果型が純粋型であるのは静的エラー。

例：

```
rubyUpper : String -> Ruby String
rubyUpper s := "s.upcase"

rubyReadLines : String -> Ruby (List String)
rubyReadLines path := """
  File.readlines(path).map(&:chomp)
"""
```

`rubyReadLines` では、Ruby の `File.readlines(...).map(&:chomp)`
が Ruby 側で `Array<String>` を生成し、これを境界が §データモデル
（§リスト）に従って `List String` へ逆マーシャルする。

Ruby ソースは各 `lower_ident` パラメータを **同名の Ruby 局所変
数** として見る。変数は §データモデル のマーシャリング規則で事前
に値を与えられる。

### 三連引用符文字列リテラル

文書 02 §文字列リテラル は複数行文字列形を保留していた。本文書が
これを有効化する。字句規則は maximal-munch：

    `triple_string` ::= `"""` に続いて、エスケープされていない
                        `"""` を **含まない** 最長のコードポイン
                        ト列とエスケープ、そして終端 `"""`。

三連引用符リテラル内部では：

- 裸の改行（`\n`）は認められ、結果文字列値に保存される。
- 02 §文字列リテラル のエスケープ（`\n`・`\t`・`\r`・`\\`・`\"`・
  `\u{...}`）は引き続き機能し、最終 `String` 値へ通常通り後処理
  される。
- 本体中の 1 つの `"` や連続する 2 つの `"` は文字通り認められる。
  リテラルを終わらせるのはエスケープされていない `"""` のみ。
- 本体中に `"""` を書きたい場合、少なくとも 1 つの `"` を `\"` で
  エスケープする必要がある：例 `""\""` や `"\""\"`。

**Ruby スニペットの内容はエスケープ後の `String` 値。** スニペッ
トを供給する文字列リテラル形（単行 `string_lit` か `triple_string`）
のどちらでも、Ruby インタプリタは 02 / 10 のすべてのエスケープが
解釈された後の `String` 値を見る。Ruby ソースで文字通りの
backslash-n を書きたい場合は `\n` ではなく `\\n` と書く。単行
`string_lit` はソース中の裸の `\n` を字句エラーとする 02 の規則
を引き続き守る — 複数行 Ruby スニペットは `triple_string` 形を
要求する。

三連引用符リテラルは通常の文字列リテラルと同じく型 `String` を持
つ。純粋な構文拡張である。主な用途は Ruby 埋め込みだが、
`string_lit` が現れるどこにでも置いてよい。

## データモデル

本節は Sapphire 値が境界を越えるときの Ruby 側表現を各値について
固定する。対応は **全域的**：サポートされた型の各 Sapphire 値は
ちょうど 1 つの Ruby 側表現を持ち、Ruby スニペットが返す各 Ruby
値はこれに従うものと期待される。

### 基底型

| Sapphire | Ruby 表現                             |
|----------|---------------------------------------|
| `Int`    | `Integer`                             |
| `String` | `String`（エンコーディング UTF-8）    |
| `Bool`   | `true` / `false`                      |

### レコード

レコード `{ f₁ = v₁, ..., fₙ = vₙ }` は **シンボルキー** の Ruby
`Hash` にマーシャルされる：

```
{ f1: ruby(v1), f2: ruby(v2), ..., fn: ruby(vn) }
```

（`ruby(·)` は再帰的なマーシャリング関数）。フィールド順序は
Ruby 側で保存されない — Ruby 1.9+ では `Hash` は挿入順だが、
Sapphire のレコードは型レベルで順序非依存（04）なので、どの挿入
順でも許容する。

Ruby 側からレコード型を期待する場所へ戻される値は、キー集合がレ
コード型のフィールド集合と完全に一致する `Hash` でなければなら
ない。過剰または不足のキーは境界実行時エラー（§例外モデル で露
出）。

### ADT（コンストラクタ形）

ADT `T` の値 `K v₁ v₂ ... vₖ` はタグ付き `Hash` にマーシャルされ
る：

```
{ tag: :K, values: [ruby(v1), ..., ruby(vk)] }
```

0 引数コンストラクタ（`k = 0`）では `values` は `[]`。タグは
コンストラクタ名を Ruby シンボル化したもの。

戻り経路では、境界は正確にキー `:tag` と `:values` を持つ `Hash`
を期待する。`:tag` は期待される型のコンストラクタを命名し、
`:values` は正しい長さの `Array`。

最も一般的な 2 つの ADT — `Maybe` と `Result` — は次のようにマー
シャルされる：

```
Nothing        → { tag: :Nothing, values: [] }
Just x         → { tag: :Just,    values: [ruby(x)] }
Err e          → { tag: :Err,     values: [ruby(e)] }
Ok  a          → { tag: :Ok,      values: [ruby(a)] }
```

（表層の便利機能として、`Maybe a` に対し Ruby 側の payload を直接
返し `nil` を `Nothing` として扱う案もある。魅力的だが `Just nil`
と `Nothing` を同一視する。draft は否定。10 未解決の問い 1 参照。）

ユーザのレコード型 `{ tag : String, values : List Int }` は（§レ
コード により）Ruby 側のハッシュに、ADT エンベロープと形が区別不
能な形でマーシャルされる。実際に曖昧性はない：境界はマーシャリン
グ規則を **期待される Sapphire 型** で選び、Ruby ハッシュを検査し
ない。ただし両方向を跨ぐ手書き Ruby コードでは、エンベロープキー
（`tag`・`values`）と一致するフィールド名を避けるのがよい（将来
の緩和でヒューリスティックな曖昧性解消が入る可能性があるため）。

### リスト

`List a` 値は Ruby `Array` にマーシャルされる：

```
Nil                → []
Cons x xs          → [ruby(x), *ruby(xs)]
```

戻りでは、各要素が `a` として逆マーシャル可能であれば任意の Ruby
`Array` が `List a` として許容される。

### `Ordering`（特殊扱い）

**Ordering は §ADT 規則への意図的な例外である。** その 3 つの 0
引数コンストラクタ `LT`・`EQ`・`GT` は 3 つの Ruby シンボル
`:lt`・`:eq`・`:gt` にマーシャルされ、タグ付きハッシュでは **な
い**。

他のすべての ADT — 0 引数のみの ADT も含む — は §ADT のタグ付き
ハッシュ規則に従う。シンボル短縮形は `Ordering` のために予約する。
Ruby の `<=>` 比較慣習（`-1 / 0 / 1` を返す）と綺麗に相互運用し、
Sapphire 側で曖昧性を保ちつつ Ruby でも自然に読める形を選ぶ。

### 関数

Sapphire 関数 `f : a -> b` は Ruby `Proc`（より正確には `Lambda`）
にマーシャルされる。Ruby が Ruby 側引数でマーシャルされた Proc を
呼ぶと、境界がそれを `a` として逆マーシャルし、Sapphire 関数を走
らせ、結果を `b` としてマーシャルして返す。

戻り方向（Ruby が Sapphire に関数を返す）では、Ruby 側は `Proc`
/ `Lambda` を供給しなければならない。カリー化の意味論は保持する：
カリー化された Sapphire 関数 `a -> b -> c` は、1 引数で呼ぶと別
のラムダを返す Ruby ラムダとして露出する（2 引数ラムダではない）。

### `Ruby a`

不透明型 `Ruby a` の値はデータとして境界を越えない — それらが
**境界である**。Sapphire 側の `Ruby a` は、走らせると Ruby 側の
副作用を起こし、`a` として逆マーシャルされる Ruby 値を返す停止さ
れた計算を表す。M8 がモナド演算と、`Ruby a` を完走まで駆動する
`run`-形の関数を定義する。

本文書は **マーシャリング契約** のみを述べる：`:=` 束縛本体に埋め
込まれた Ruby ソースは、マーシャル型が `a` である Ruby 値を生成
する（束縛の宣言型が `Ruby a` を仮定）。評価中のエラーは §例外
モデル で露出する。

## 例外モデル

Ruby コードは raise してよい。`Ruby a` アクションが走った（M8
による）とき、下層の Ruby コードが例外を上げたら、アクションの結
果は成功した `a` で **なく**、Sapphire 側に露出される例外となる。

契約：すべての Ruby 側 **ユーザレベル例外**（`StandardError` と
その子孫）は境界で捕捉され、Sapphire 側の `RubyError` 値に変換
される。システムレベル例外（`SystemExit`・`Interrupt`・
`NoMemoryError`・`SystemStackError` およびその他の非
`StandardError` な `Exception` 子孫）は **捕捉されず** 境界を通
り抜け、通常通り Ruby プロセスを終了させる。これは Ruby 慣習（
`rescue => e`、すなわち `rescue StandardError` を捕捉範囲の既定
とする）に従う。

`RubyError` 型は本文書で次のように定義する：

```
data RubyError = RubyError String String (List String)
                         -- class_name   message    backtrace
```

（04 未解決の問い 2 の決着により位置引数のみ。フィールドの意味
はクラス名・メッセージ・バックトレースの順。）

`RubyError` は **prelude 隣接モジュール `Ruby`** に住み、
`Prelude` と並んで暗黙にインポートされる。形が Ruby 固有であり、
09 の中核 prelude の関心と無関係なため、`Prelude` 自体には入れな
い。文書 11 はモナド型 `Ruby` と `run` 関数を同モジュールに置き、
08 の孤児インスタンス問題を回避する。

`RubyError` がユーザコードに届く経路は文書 11 で固定される：
`Ruby a` 型は捕捉されたユーザレベル Ruby 例外をアクションの短絡
終了として扱い、文書 11 は `run : Ruby a -> Result RubyError a`
を pure 側の唯一の入口として露出する。本文書は `RubyError` 型と
「ユーザレベル Ruby 例外（`StandardError` 以下）は境界で捕捉され
る。システムレベル例外は境界を通り抜ける」という scoped な規則
のみを固定する。ユーザ向けインタフェースは文書 11 が埋める。

## 生成 Ruby モジュールの形

Sapphire モジュール `M₁.M₂. ... .Mₙ` はトップレベル `Sapphire` の
下の Ruby 名前空間へコンパイルされる。命名規則は：

- 先祖セグメント `M₁, ..., Mₙ₋₁` はそれぞれ Ruby `module` になる。
- 末端セグメント `Mₙ` は Ruby `class` になる。
- 完全修飾名は `Sapphire::M₁::M₂:: ... ::Mₙ`。

単一セグメントの Sapphire モジュール（例：`Main`）の場合、トップ
レベル `Sapphire` ラッパモジュールのみが emit され、その直下に末
端クラスが置かれる：`module Sapphire; class Main; ...; end; end`。

```
# Sapphire モジュール Data.List から生成
module Sapphire
  module Data
    class List
      ...
    end
  end
end
```

エクスポートされた各 Sapphire トップレベル束縛 `name` は生成クラ
スのクラスメソッドになる：

```
# エクスポート: map : (a -> b) -> List a -> List b
module Sapphire
  module Data
    class List
      def self.map(f, xs)
        # 生成実装
      end
    end
  end
end
```

生成メソッドは順序通りに Sapphire 引数を取り、§データモデル に
従って逆マーシャルし、Sapphire 側ロジックを走らせ、結果を戻すた
めにマーシャルする。

`Ruby` 型の束縛について：

- 生成メソッドは本体の Ruby 出力を Sapphire に戻すためのマーシャ
  ルを行わない。Ruby ソースをそのまま実行し、Ruby 値を直接返す
  （例外捕捉を除く）。
- 例外捕捉は本体を `begin ... rescue => e ... end` で包み、
  戻り経路で `e` を `RubyError` 形の `Hash` に再パッケージする。

ADT コンストラクタはファクトリクラスメソッドになる：

```
# data Maybe a = Nothing | Just a
module Sapphire
  class Prelude   # または Maybe 定義場所
    def self.Nothing
      { tag: :Nothing, values: [] }
    end

    def self.Just(x)
      { tag: :Just, values: [x] }
    end
  end
end
```

命名の詳細：

- コンストラクタ名はクラス上の Ruby メソッド名として文字通り保持
  される。Ruby は大文字始まりのメソッド名を受け入れる（`Foo.Bar`
  として呼ぶ）ので mangle は不要。**注意：** 引数を伴う大文字 Ruby
  メソッド呼び出しは呼び出し側で括弧を必須とする（`Foo.Just(1)`）。
  `Foo.Just 1` は定数アクセスとして解析されるためである。0 引数
  呼び出し（`Foo.Nothing`）は括弧なしで機能する。
- Sapphire の `(+)` や `(>>=)` のような演算子は `upper_ident` を
  含まない Ruby 表現を要する。`op_plus` / `op_bind` のような mangle
  された名前でメソッドになる。正確な mangle 方式は 10 未解決の問
  い 2。

## 他文書との相互作用

- **02。** 三連引用符文字列リテラルは、02 の追加拡張条項のもと、
  文字列リテラル文法を拡張する。既存の単行 `string_lit` は変わら
  ない。
- **07。** `:=` 束縛は `Ruby a` 値を生成する。`do` 記法を動かすに
  は `Monad` インスタンスが必要である。M8 が
  `instance Monad Ruby` を宣言し、本文書はそのインスタンスを前
  提とする。
- **08。** Sapphire モジュールの生成 Ruby クラスはそのモジュール
  名で鍵付けされる（08 §1 ファイル 1 モジュール 参照）。Sapphire
  側のインポートは Ruby 側のクラスレイアウトに影響しない — 各モ
  ジュールの Ruby クラスは独立。
- **09。** prelude の `print` stub は、やがて導入される `Ruby`
  型束縛の典型例である。現在の `Result String {}` 型は、M8 着地
  後に `Show a => a -> Ruby {}` のような形に置き換わる。

## 設計メモ（非規範的）

- **`:=` を採り、ブロック形を採らない理由。** 02 がまさにこの用途
  のために `:=` を予約済みであり、Haskell-FFI 風の境界では宣言レ
  ベルの演算子が入れ子ブロックより自然に感じられる。ブロック形は
  Ruby ソースが式の途中に現れうることを意味し、パーサの複雑化を
  招き、`do` 記法との相互作用も難しくする。

- **ADT はクラスではなくタグ付きハッシュ。** ADT をタグ付きハッシ
  ュで表現する（コンストラクタごとのサブクラスを持つ Ruby クラス
  ではない）ことで、マーシャリング契約を単純に保ち、Ruby 側が生
  成クラス定義に依存せずにすむ。Ruby 作者はハッシュをパターンマ
  ッチするか、`.dig(:tag)` を直接呼べる。

- **`nil` を `Nothing` として扱う近道は採らない。** 魅力的だが危
  険：`Just nil` と `Nothing` が区別不能になる。明示的なタグ付け
  は Ruby 側のわずかな冗長性と曖昧性なしを引き換える。

- **関数値はラムダとして越境する。** カリー化された関数が「ラムダ
  を返すラムダ」として現れるのは、Sapphire が型レベルで既に持って
  いる形だからである。複数引数ラムダを好む Ruby 呼び出し側は
  `.curry(n)` を呼んで適応できる。

- **三連引用符文字列は純粋な拡張。** 汎用的な「複数行文字列」未解
  決の問いに送るのではなく本文書で規定するに値するのは、Ruby 埋
  め込みが即座にこれを必要とするためである。他の用途（JSON リテ
  ラル、ドキュメント文字列）はおまけで付いてくる。

- **境界であって FFI ではない。** 本層の Sapphire は任意の Ruby
  gem の読み込み／モンキーパッチに自身を開放しない。各 `:=` 束縛
  は含まれたスニペットである。より大きな Ruby 統合は、生成
  Sapphire モジュールを Ruby コードから呼ぶ通常の経路を通る。逆
  方向はしない。

## 未解決の問い

1. **`nil`-for-`Nothing` の近道。** Ruby が返す `nil` を `Nothing`
   に、任意の非 nil 値を `Just x` にマップする近道を認めるか。
   `Just nil` と `Nothing` の区別を失う。draft は否定。

2. **演算子メソッド mangle 方式。** `(+)`・`(>>=)`・`(::)` のよう
   な演算子に対する正確な Ruby メソッド名マッピングは何か。候補：
   `op_plus` / `op_bind` / `op_cons`、あるいは
   `sapphire_op_7_6_plus`（位を含む）、あるいは Unicode 風エスケー
   プ。draft は未規定。実装時に確定させる。

3. **シンボルキー vs 文字列キーのハッシュ。** レコードはシンボル
   キー（`{ f1: ... }`）を使う。文字列キー代替（`{ "f1" => ... }`）
   は一部の Ruby ライブラリ（JSON round-trip）との相互運用が良い。
   draft はシンボルキー。

4. **例外バックトレース構造。** `RubyError.backtrace` は現在
   `List String`。より構造化した形（`List { file : String,
   line : Int, method : String }`）は、より豊かなエラー露出を
   可能にする。draft は Ruby の `Exception#backtrace` と直接一致
   する `List String`。

5. **ファイルからの Ruby ソース取り込み。** インライン `:=` スニ
   ペットを超えて、外部 Ruby ファイルを生成モジュールに引き込む
   `ruby_import "path.rb"` 宣言を認めるか。draft は否定。M9 の例
   題が要求すれば再訪。

6. **Ruby 3.x を超えるバージョンサポート。** データモデルは Ruby
   3.x（具体的には `docs/project-status.md` により Ruby 3.3）を
   固定する。それ以前の Ruby は範囲外 — これは決定であって未決
   ではない。ここに残す未決は、**将来の** Ruby（到来する 4.x な
   ど）がマーシャリング契約の修正を要求するかどうか。draft は
   「監視、現時点で対応不要」。

7. **高次コンストラクタ ADT。** タグ付きハッシュ表現は任意のコン
   ストラクタ arity に拡張する。本質的な上限はない。ただし人間工
   学のため Ruby 側で名前付きタプル風ラッパ（`OpenStruct`・
   `Struct`）が欲しくなるかもしれない。現時点では計画しない。
