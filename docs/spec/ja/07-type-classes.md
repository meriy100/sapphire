## 種 (kind) 体系

**種** は型を分類する。本層の種は次で与えられる：

```
κ ::= '*'
    | κ '->' κ
```

`*` は値を分類する型の種（03 が暗黙に使っていた唯一の種）。
`κ₁ -> κ₂` は `κ₁` から `κ₂` への型レベル関数の種。種は型の矢印と
同じく右結合。

例：

- `Int`・`Bool`・`String` — 種 `*`。
- `Maybe`・`List`・`IO` 類型 — 種 `* -> *`。
- `Either`・関数矢印 `(->)` — 種 `* -> * -> *`。
- レコードは種 `*`（レコード値を分類する）。

型 `τ` が種の仮定環境 `Δ` のもとで整形であることの判断

    Δ ⊢ τ :: κ

の規則は自然なもの：

- `Δ ⊢ T :: κ` （`Δ(T) = κ` のとき、型コンストラクタ探索）。
- `Δ ⊢ a :: κ` （`Δ(a) = κ` のとき、型変数探索）。
- `Δ ⊢ τ₁ τ₂ :: κ₂` （`Δ ⊢ τ₁ :: κ₁ -> κ₂` かつ `Δ ⊢ τ₂ :: κ₁`
  のとき）。
- `Δ ⊢ τ₁ -> τ₂ :: *` （`Δ ⊢ τ₁ :: *` かつ `Δ ⊢ τ₂ :: *` のとき）。
- レコード型 `{ f₁ : τ₁, ..., fₙ : τₙ }` は各 `τᵢ :: *` のとき種
  `*` を持つ。

値型（Sapphire 式の型）あるいは引数型（`->` の内側）として現れてよ
いのは種 `*` の型のみである。高階種の型コンストラクタは、スキーム
中の量化位置と型適用の頭位置にのみ現れる。

**種は推論される**、注記されない。種推論は標準的な手順（各型変数
に新鮮な種変数を割り当て、使用箇所から単一化で解く）。本層ではソ
ースにおける種構文を一切認めない。`(a :: * -> *)` のような注記を
認めるかは未解決の問い 5。

`data` 宣言 `data T a₁ ... aₙ = ...`（03）は `T` に種
`κ₁ -> ... -> κₙ -> *` を与える（各 `κᵢ` はコンストラクタ引数型
における `aᵢ` の使用から推論される）。

## 抽象構文（BNF）

文書 01・03・06 を拡張：

```
decl   ::= ...                                     -- (01, 03)
         | 'class' context? TCON TVAR 'where' class_body
         | 'instance' context? TCON type 'where' instance_body

context   ::= constraint_list '=>'
constraint_list ::= constraint                      -- 単独、または
               | '(' constraint (',' constraint)* ')'
constraint ::= TCON type                            -- 型への適用

class_body ::= layout_block_of(class_item)
class_item ::= IDENT ':' scheme                     -- メソッドシグネチャ
             | clause                               -- 既定メソッド本体

instance_body ::= layout_block_of(instance_item)
instance_item ::= clause                            -- メソッド実装

clause     ::= IDENT apat* '=' expr                 -- 関数節定義。
                                                    -- IDENT = \apat... -> expr
                                                    -- の省略形

scheme ::= 'forall' TVAR* '.' context? type         -- (01 を拡張)
         | context? type                            -- 表層の省略形

expr   ::= ...                                      -- (01, 03, 04, 05, 06)
         | 'do' do_block

do_block   ::= layout_block_of(do_stmt)
do_stmt    ::= pat '<-' expr                        -- モナドバインド
             | 'let' IDENT '=' expr                 -- let 束縛（`in` 無し）
             | expr                                 -- モナド式
```

`layout_block_of(…)` は 02 / 06 で既に確立した `of` 風ブロックの
レイアウト形。明示波括弧＋セミコロンが脱出ハッチ。

BNF への注記：

- `class_item` のシグネチャは `IDENT ':' scheme` 形式。型注記区切
  り `:` は 01 由来。演算子風メソッドは 05 の括弧付き演算子形
  `(==)` を用いる。本層では `(==)` は `IDENT` 扱い（表層綴りは括
  弧内の演算子）。
- `clause` 形式 `IDENT apat* '=' expr` はクラス既定・インスタンス
  本体（および今後のトップレベル定義）で使う表層糖衣：`f x y =
  body` は `f = \x -> \y -> body` と同値。01 の `decl ::= IDENT
  '=' expr` は 0 引数ケース（`apat* = ε`）であり、本文書はこれを
  全定義サイトで均一に一般化する。`apat` は 06 の原子パターンク
  ラス。
- 演算子型クラスメソッドは追加の表層形を認める：メソッド `(op)`
  の中置 LHS 節 `x op y = body`。これは基本 `clause` 文法上の便
  利形であり `(op) = \x -> \y -> body` と同値。下記のクラス宣言
  とインスタンス宣言の例は `(==)`・`(/=)`・`::` パターンでこの形
  を用いる。
- `instance_item` 本体は `clause` のみ。型シグネチャは持たない
  （クラスから来るため）。
- `scheme` の拡張は、型の前に省略可能な `context` を置き、
  `Eq a => a -> a -> Bool` のような制約を表現可能にする。
- `do_block` 本体は `case ... of` と同じレイアウト敏感ブロックを
  用いる：参照列は最初の `do_stmt` の列、同列のトークンが新しい
  項目、明示波括弧形式ではセミコロンで区切る。`do` はブロック開始
  キーワードであり、文書 02 のレイアウト一覧はこれに合わせて拡張
  される。

## 制約付き型スキーム

本層の **型スキーム** は次の形：

    ∀ a₁ ... aₙ . C₁ τ₁,₁, ..., Cₘ τₘ,kₘ => τ

各 `Cⱼ τⱼ,₁ ... τⱼ,kⱼ` は量化変数に対するクラス制約。`let`（01）
およびトップレベル（03）における一般化は拡張される：単相型 `τ` を
`Γ` の下で一般化するとき、結果のスキームは elaboration 中に収集し
た `ftv(τ) \ ftv(Γ)` 上のすべての制約を含む。

制約の作用範囲は右側の型全体に及ぶ：`Eq a => a -> a -> Bool` は
`(Eq a) => (a -> a -> Bool)` と解析され、制約はその後続の矢印型全
体を束縛する。

01 の (Var) 規則は単に拡張されるのではなく **置き換わる**（以下の
(Var')）。01 の全判断は既知の制約集合 `Q`（周囲のスキームからの仮
定、あるいは同じ elaboration 中の先行 (Var') 発火）を伴う。使用箇
所で制約付きスキームをインスタンス化すると、単相型と、`Q` が解か
ねばならない制約集合が生じる：

```
  σ = Γ(x)    instantiate(σ) = (τ, C)    Q ⊨ C
  —————————————————————————————————————————————   (Var')
                    Γ; Q ⊢ x : τ
```

`Q ⊨ C` は、`C` の各制約が `Q` の要素（すなわち周囲のスキームの
仮定）であるか、§インスタンス解決 の規則によりスコープ内に一致す
るインスタンスがあるかのいずれかであることを表す。本文書の他の規
則では `Q` を明示しないが（01 の記法に寄せるため）、概念的には全
判断がこれを持ち回ると理解すること。

## クラス宣言

```
class Eq a where
  (==) : a -> a -> Bool
  (/=) : a -> a -> Bool
  x /= y = not (x == y)
```

クラス `class C a where ...` は次を宣言する：

- 名前 `C`（クラス環境に追加）。
- メソッドシグネチャ `mᵢ : σᵢ`。クラス外部から見える各メソッドの
  スキームは `∀ a. C a => σᵢ` — クラスパラメータは使用箇所で量化・
  制約される。
- 省略可能な **既定メソッド** `mᵢ = eᵢ`。インスタンスが省略したと
  きのメソッド振る舞いを与える。既定は同クラスの兄弟メソッドを呼
  べる。

スーパークラス制約はクラス頭の前に、型と同じ `context '=>'` 形で
書く：

```
class Eq a => Ord a where
  compare : a -> a -> Ordering
  ...
```

`instance Ord T where ...` は `Eq T` を前提にできる：コンパイラは
スコープ内で `instance Eq T` を見つけるか、さもなくば `Ord T` 宣言
を拒絶する。

本層では **単一パラメータ** クラスのみ。多パラメータ型クラス
（`class C a b where ...`）は未解決の問い 1。

## インスタンス宣言

```
instance Eq Int where
  x == y = <Int の原始的な等価>
  -- /= はここでは定義せず、クラス既定を使う

instance Eq a => Eq (List a) where
  Nil       == Nil       = True
  Cons x xs == Cons y ys = x == y && xs == ys
  _         == _         = False
```

`instance C T` は、型 `T`（自由変数を持ちうる）がクラス `C` を支
持することを宣言する。インスタンス本体はクラスのメソッドに対する
定義を与える。省略されたメソッドはクラスの既定を受け取るか、既定
が無ければ静的エラー。

インスタンス宣言は文脈（上の `Eq a =>`）を伴ってよい。文脈はイン
スタンス頭に現れる自由型変数を制約する。

**インスタンス頭** は制限される。クラス `C` に適用する型は以下の
いずれか：

- 相異なる型変数に適用された型コンストラクタ（`Eq (List a)`・
  `Monad Maybe`・`Show (Tree a)`）。
- 基底型（`Eq Int`・`Show String`）。

裸の型変数（`instance Eq a where ...`）や任意の形（コンストラクタ
内部に具体型を含む `instance Eq (List Int) where ...`）は本層では
認めない。これは「Haskell 98」のインスタンス形であり、緩和は未解決
の問い 2。

### 孤児インスタンス

インスタンス `instance C T` が **孤児** であるとは、`C` も `T` の
最外型コンストラクタも、インスタンスと同じモジュールで定義されて
いないこと。本層では孤児インスタンスは **禁止** する。すべてのイ
ンスタンスは、自身のクラスあるいは型を定義するモジュール内に置く
こと。

これは Haskell の厳格方針。「クラスと型が両方スコープ内にあるとき
は常にインスタンスも自動的にスコープ内」という不変を保つ。下流の
コード（M8 の Ruby 評価型に対する `Monad` インスタンスを含む）が
これに依存する。

### 重複インスタンス

インスタンス `instance C T₁` と `instance C T₂` が **重複する** と
は、ある型 `T` が両方（制約を無視して）一致すること。本層では重複
インスタンスを **禁止** する。クラス／型頭のペアに対して、スコープ
内のインスタンスは高々 1 つ。

緩和（Haskell の `OverlappingInstances` 系）は未解決の問い 3。

## インスタンス解決

使用箇所で制約 `C τ` が与えられたら、解決は次の手順：

1. スコープ内の各インスタンス `instance ctx => C T` を列挙する。
2. 頭 `C T` が `C τ` と単一化する唯一のものを見つける。単一化する
   ものが無ければ制約は未解決（周囲のスキームがこれを文脈に量化す
   る場合を除き型エラー）。
3. インスタンス化用代入のもとで、インスタンスの文脈 `ctx` を再帰
   的に解く。
4. 解かれた各制約は実装レベルで **辞書** を生む（ユーザからは不可
   視）。辞書はメソッド呼び出しを通じて渡される。本文書は実装戦略
   を固定しないが、解決の複雑度を規定するために辞書に言及する。

解決は孤児なし・重複なし方針のもとで構成的に **コヒーレント**：任
意の使用箇所と制約について、解決経路は高々 1 本。

## 標準クラス

MTC ではこれらを **形**（メンバーのシグネチャと法則）として固定す
る。実際の prelude 側インスタンスは M6 の範疇。法則は非規範的に記
述するが、各インスタンスはこれを満たすべきである。

### `Eq`

```
class Eq a where
  (==) : a -> a -> Bool
  (/=) : a -> a -> Bool
  x /= y = not (x == y)
```

最小完全定義：`(==)`。インスタンスは `(==)` を定義しなければなら
ない。`(/=)` は既定で与えられる。

法則：`==` は同値関係（反射・対称・推移）。`x == y` iff
`not (x /= y)`。

### `Ord`

```
class Eq a => Ord a where
  compare : a -> a -> Ordering
  (<)  : a -> a -> Bool
  (>)  : a -> a -> Bool
  (<=) : a -> a -> Bool
  (>=) : a -> a -> Bool
  -- `compare` に基づく既定；ここでは実装省略
```

`Ordering = LT | EQ | GT` は prelude の ADT（M6）。`<`・`>`・
`<=`・`>=` は `compare` に基づく既定を持つ。**最小完全定義は
`compare`** であり、`compare` を与えるインスタンスは 4 つの比較
メソッドを継承する。

法則：`compare` は `Eq` の `==` と整合する全順序。

### `Show`

```
class Show a where
  show : a -> String
```

法則：なし（表示・デバッグ用の慣習的定義）。

### `Functor`

```
class Functor f where
  fmap : (a -> b) -> f a -> f b
```

ここで `f :: * -> *`、`f a`・`f b` における使用から推論される。

法則：

- `fmap id = id`
- `fmap (g . h) = fmap g . fmap h`

### `Applicative`

```
class Functor f => Applicative f where
  pure  : a -> f a
  (<*>) : f (a -> b) -> f a -> f b
```

法則（identity、composition、homomorphism、interchange）は Haskell
と同じ。詳細は Haskell の `Control.Applicative` 文書を参照。

### `Monad`

```
class Applicative m => Monad m where
  (>>=)  : m a -> (a -> m b) -> m b
  return : a -> m a
  return = pure
```

法則：

- `return a >>= f ≡ f a`                           （左単位元）
- `m >>= return ≡ m`                                （右単位元）
- `(m >>= f) >>= g ≡ m >>= \x -> f x >>= g`         （結合律）

`return` は `Applicative` の `pure` に基づく既定を持つ。インスタ
ンスは明示性のため上書きしてもよい。

演算子 `(>>=)` は 05 の演算子表に追加される（第 1 位、左結合）。
MTC 下での型スキームは
`∀ m a b. Monad m => m a -> (a -> m b) -> m b`。（演算子表の修正
は 05 → 07 の forward reference として適用する。`>>=` 他の行の最
終形は M6 が prelude の名前集合を固めた時点で埋まる。）

## `do` 記法

`do` 形式は `Monad` に対する **表層糖衣**。3 つの stmt 形は以下の
ように糖衣展開される。ブロックが

    do
      s₁
      s₂
      ...
      sₙ

（最後の文 `sₙ` は `expr` 形式でなければならない — バインドや
let は最終文に置けない）のとき：

- `pat <- e` の後に `rest` が続くときは
  `e >>= \pat -> desugar(rest)` に展開。
- `let x = e` の後に `rest` が続くときは
  `let x = e in desugar(rest)` に展開。
- 単独の `expr` 文 `e` の後に `rest` が続くときは
  `e >>= \_ -> desugar(rest)` に展開。
- 最終文 `e` は `e` そのもの。

結果の式はあるモナド型 `m τ` を持つ。`m` はブロック内の文から決定
される（全てのバインドと単独 `expr` 文は同じ `m` を共有しなければ
ならない）。

例：

```
do x <- readFile "a"
   y <- readFile "b"
   pure (x ++ y)
```

は

```
readFile "a" >>= \x ->
  readFile "b" >>= \y ->
    pure (x ++ y)
```

に展開される（ある `Monad m` について `readFile : String -> m
String` を想定）。

**`<-` のパターンは拒絶不能（irrefutable）。** `do` バインド
`Just x <- mx` は本層では型エラー（パターンは `Maybe a` に対して拒
絶可能）。拒絶可能バインドの統合には、`MonadFail` 風の脱出
（Haskell 方式）か、`case` への糖衣展開が必要。どちらも未解決の問
い 4。

## 既存 draft への影響

- **03（データ型）。** 「本層の種は自明である：全ての型は値を分類
  する」の記述は上書きされる。型コンストラクタの種は n パラメータ
  の `data` に対して `κ₁ -> ... -> κₙ -> *` になる。型適用の整形性
  は種を検査する（MTC §種体系）。03 の「高階種は後送り」を扱う OQ
  は本文書で **決着**。

- **05（演算子と数値）。** 等価・順序演算子（`==`・`/=`・`<`・
  `>`・`<=`・`>=`）は `Eq` / `Ord` のメソッドになる。05 の表にお
  ける型は **prelude インスタンス** `Eq Int` / `Ord Int` の型と
  読むこと。一般のクラススキームは `a -> a -> Bool`。05 未解決の
  問い 2（多相等価）は **決着**：可、`Eq` / `Ord` 経由。

  算術演算子（`+`・`-`・`*`・`/`・`%`）は本層でも単相
  `Int -> Int -> Int` のまま。`Num` クラスに格上げすることは大きな
  コミット（Haskell の `Num` は肥大で悪名高い）。未解決の問い 6。

  演算子 `(>>=)` は 05 の表に新たな位（第 1 位、左結合）として加
  わる。`>>`（モナド列化）も付随しうる。M6 prelude 参照。

- **06（パターンマッチ）。** 不変。パターンマッチは型クラスと直
  交する。制約付き被検査型でも `case` は同じく動作する。パターン
  束縛変数の制約に対するインスタンス解決は各アーム内の使用箇所で
  起きる。

- **04（レコード）。** 影響なし。行多相（04 未解決の問い 1）は直
  交軸であり、MTC はその決着を強制しない。

- **01（中核式言語）。** (Let) 規則のスキーム一般化が制約収集を含
  むよう変更される（§制約付き型スキーム 参照）。(Var) は (Var')
  に置き換わり、使用箇所で制約解決可能性を強制する。`decl` 生成
  規則の `IDENT '=' expr` 形は `clause` 形 `IDENT apat* '=' expr`
  に一般化される（§抽象構文 参照）。これによりトップレベルでも
  `f x y = body` が認められる（クラス本体・インスタンス本体と同
  様）。以上 3 点の改訂はいずれも単調拡張であり、制約を使わず 0
  引数定義のみのコードでは同じ振る舞いとなる。

## 設計メモ（非規範的）

- **本 draft は単一パラメータクラスのみ。** 多パラメータ型クラス
  （MPTC）は設計空間が大きい：関数従属、型族による回避、インスタ
  ンス解決の複雑化。draft を単一パラメータに保つことで、`Eq`・
  `Ord`・`Functor`・`Applicative`・`Monad` は全て表現でき、Haskell
  98 の機能集合と一致する。未解決の問い 1 で MTC 着地後に再訪。

- **Haskell 98 形のインスタンス頭。** `C T` 形のインスタンス頭で
  `T` が基底型または相異なる型変数に完全適用された型コンストラク
  タであることに限る制限は、いわゆる `FlexibleInstances` off の体
  制。解決を決定可能かつコヒーレントに保つ。緩和（`FlexibleInstances`
  ほか）は未解決の問い 2。

- **重複なし・孤児なし。** この二つの制約は、解決を決定的かつイン
  ポート順序非依存にする。どちらを緩めてもコヒーレンスの微妙な問
  題を招く。未解決の問い 3 で再訪。

- **辞書渡しは概念モデル。** 仕様は実装レベルで辞書渡しを強制しな
  い — 全プログラム特殊化・インライン化による方式も健全である。
  辞書への言及は解決の **決定可能性** を固めるためであって、実装
  を規定するものではない。

- **種体系は second-class。** 種は整形性判断にのみ現れ、第一級型
  ではない。種多相（Haskell の `PolyKinds` 風）や、ソース中の種変
  数は採らない。未解決の問い 5。

- **`do` はモナド専用。** `do` ブロックは `(>>=)` で糖衣展開され
  るため、ブロック全体に `Monad m` 制約が付く。Applicative のみの
  `do`（Haskell の `ApplicativeDo`）は認めない。表現的な `Functor`
  / `Applicative` のみの形が欲しくなったら、`do` を上書きするので
  はなく別の糖衣を追加する。現時点で OQ としない。

- **派生インスタンス構文は無し。** `deriving` 節（03 未解決の問い
  4 由来）は後送りのまま。本層ではインスタンスは手書き。

- **Ruby 評価との関係（M8）。** Ruby 評価型は具体的な `Monad` イ
  ンスタンスになる。その `return` / `>>=` は他のモナドと同じ解決
  機構を通る。M8 の命名の決定は **どの具体型か** の話であり、モ
  ナド機構そのものは MTC が提供する。

## 未解決の問い

1. **多パラメータ型クラス。** `class C a b where ...` とその付随
   機構（関数従属の可能性を含む）を認めるか。draft は単一パラメー
   タのみ。

2. **柔軟なインスタンス頭。** Haskell 98 形を超えるインスタンス形
   — 重複しない変数への適用ではないコンストラクタ、入れ子のコン
   ストラクタ、型別名頭 — を認めるか。draft は禁止。

3. **重複インスタンスと孤児。** 重複なし・孤児なし方針を緩めるか。
   緩めるならどの規則のもとで。draft は両方とも禁止。

4. **`do` の拒絶可能バインド。** `do` 内の `Just x <- mx` をどう
   扱うか — `MonadFail` 風の脱出、失敗時エラーの `case` への糖衣
   展開、あるいは不許可か。draft は不許可。

5. **ソースにおける明示的種注記。** 型式中の `(a :: * -> *)` や、
   宣言中の `type T (a :: * -> *) = ...` を認めるか。draft は否定、
   種は推論のみ。

6. **算術用の `Num` クラス。** `+`・`-`・`*`・`/`・`%` を `Num`
   クラスのメソッドに昇格させ、`Int` / `Float` / ユーザ定義数値型
   で多相にするか。draft は否定。算術は単相 `Int`、05 の決定に従
   う。05 未解決の問い 1（`Float` / `number`）と相互作用する。

7. **関連型と型族。** Haskell の `TypeFamilies` / `AssociatedTypes`
   相当を持つか。draft は無し。

8. **インスタンス自動派生のための `deriving`。** Haskell 風
   `data T = ... deriving (Eq, Show)` を認めるか。draft は無し。
   インスタンスは手書き。03 未解決の問い 4 の「表層に `deriving`
   節を認めるか」は本文書から再度参照される。最終判断は M6 また
   は後続文書で下る見込み。

9. **高階種制約のインスタンス連鎖解決。** `Monad m => Functor m`
   のような制約は整形（`Applicative m => Monad m` のスーパークラ
   スにより全モナドは Applicative、`Functor f => Applicative f`
   のスーパークラスにより全 Applicative は Functor）である。draft
   の意図：解決器は **スーパークラスを通じて自動的に連鎖する**。
   すなわち `Monad m` の仮定下で `Functor m` 制約は、ユーザーに
   明示的な `Functor` インスタンスを書かせることなく解決される。
   2 段階連鎖の例題は本文書ではなく後続文書に送る。
