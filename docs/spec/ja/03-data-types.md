# 03. データ型

状態: **draft**。パターンマッチ (M3)、モジュール (M4)、prelude (M6) の
着地で生じる制約は本層へ逆流しうる。

本文書は `docs/spec/03-data-types.md` の日本語訳である。英語版が規範的
情報源であり、BNF 生成規則・型付け規則・規則名・番号付き未解決の問い
は英語版と一致させて保つ。

## 動機

文書 01 は中核式言語を閉じた基底型集合（`Int`、`Bool`、`String`）の
上で固定した。本文書は **代数的データ型**（ADT）を導入し、Sapphire
がそれらの基底型定数を超えて新しい型を得る唯一の機構として位置付け
る。併せて文書 01 の未解決の問い 1（`let` における再帰）をここで決
着させる。

下流のマイルストーンはいずれも ADT に依存する：

- パターンマッチ (M3) がデータ値を分解する。
- prelude (M6) は `Bool`、`Maybe`、`Result`、リスト型を本層の機構で
  定義する。
- Ruby インタロップ (M7) は Ruby 側のエラー・結果の形を通常の
  Sapphire データ型としてモデル化する。

以下のプログラムは本層が受理しなければならない基準例である。表層の
字句形式（`data`、`|`、`upper_ident`）は文書 02 そのまま。

```
-- 列挙
data Bool = False | True

-- パラメトリック和型と、その値の一つ
data Maybe a = Nothing | Just a
let x = Just 42 in x

-- 再帰型と、その値
data List a = Nil | Cons a (List a)
let xs = Cons 1 (Cons 2 (Cons 3 Nil)) in xs

-- 型レベルの相互再帰
data Even = ZeroE | SuccE Odd
data Odd  = SuccO Even
let e = SuccE (SuccO ZeroE) in e

-- コンストラクタでも let-多相はそのまま働く
let pair = \a -> \b -> Cons a (Cons b Nil) in
let _xs = pair 1 2 in
pair "hello" "world"
```

上の例はいずれも、本文書の規則だけでは **分解できない**。分解は M3
の担当。本層はデータ型の導入と値の構築のみを扱う。

本層では意図的に保留：

- **パターンマッチと分解**。M3 の領分。
- **レコード形コンストラクタ**（`=` の右辺に名前付きフィールド）。
  匿名レコード・名前付きフィールドは M2 の領分。
- **高階種（higher-kinded types）**。本層では全ての型は種 `*` を持
  ち、型コンストラクタは常に全引数を伴う。
- **一般化代数的データ型 (GADT)、存在量化、phantom 型的技法**。
- **`deriving` 節** その他、型クラス・トレイトの自動生成。

## 抽象構文（BNF）

文書 01 の BNF を単調に拡張：

```
decl      ::= IDENT ':' type                     -- 型シグネチャ       (01)
            | IDENT '=' expr                     -- 定義               (01)
            | 'data' TCON TVAR* '=' data_alts    -- データ宣言         (新)

data_alts ::= data_alt ('|' data_alt)*
data_alt  ::= TCON atype*                        -- 0 個以上の引数を持
                                                 -- つコンストラクタ
                                                 -- 各引数は atomic 型。
                                                 -- 矢印型や型適用は括弧
                                                 -- で包むこと

type      ::= btype                              -- （01 から改訂）
            | btype '->' type                    -- 右結合

btype     ::= atype
            | btype atype                        -- 型適用、左結合

atype     ::= TVAR
            | TCON
            | '(' type ')'
```

文書 01 の `type` 非終端は精緻化される：基底定数 `Int`、`Bool`、
`String` は arity 0 の通常の `TCON` であり、型適用 `T τ₁ ... τₙ` は
`btype` 層で左結合の反復適用として解析される（項適用と同じ形）。

`upper_ident`（02 §識別子）は型コンストラクタと値コンストラクタの
双方を担う。ある出現がどちらの役を果たすかは位置で決まる：型位置な
ら型コンストラクタ、項位置なら値コンストラクタ。

`data` 宣言は本層では **トップレベル専用** である。`decl` は `program`
（01）から到達され、`expr` の内側からは到達されない。`let` ブロック
内に局所 `data` は存在しない。`data` が予約語集合に加わる点は文書 02
の追加拡張条項（02 §予約語）に従う。02 §予約語の表は本文書と同じ
変更で更新される。

`data` 宣言の集合が静的に整形であるためには：

- 2 本のトップレベル `data` 宣言が同じ `TCON` を導入してはならない。
- トップレベル別宣言のコンストラクタ名を `data_alt` が再利用しては
  ならない。
- `data_alt` 中の各 `τᵢⱼ` は、左辺の `TVAR` と、スコープに入ってい
  る型コンストラクタ（当該宣言自身の `TCON`、および同じ相互再帰
  グループ内の他の `TCON` を含む — §再帰参照）のみを参照できる。
  左辺に束縛されていない自由型変数が右辺に現れれば静的エラー（暗
  黙的量化は行わない）。

## 型

本層の種（kind）は自明である：全ての型は値を分類し、全ての型コンス
トラクタは固定の arity n ≥ 0 を持つ。TCON 適用 `T τ₁ ... τₙ` は
引数がちょうど `n` 個のときにのみ整形である。型コンストラクタの部分
適用は本層では許されない — 高階多相における役割は後送り。

宣言

    data T a₁ ... aₙ = C₁ τ₁₁ ... τ₁ₖ₁ | ... | Cₘ τₘ₁ ... τₘₖₘ

は同時に以下を導入する：

- arity `n` の型コンストラクタ `T`。
- 値コンストラクタ `C₁, ..., Cₘ`、それぞれに型スキームが与えられ
  る（§型付け規則参照）。

宣言の右辺では以下がスコープに入る：`T` 自身、`a₁, ..., aₙ`、およ
び周囲の相互再帰グループが導入する他の型コンストラクタ。

### 再帰と相互再帰（型）

`data` 宣言の右辺は `T` 自身を言及してよい（直接再帰型、例：
`data List a = Nil | Cons a (List a)`）。

複数のトップレベル `data` 宣言は単一の相互再帰グループを成す：
各宣言の右辺はテキスト順序によらず、他の宣言の型コンストラクタを
言及してよい。下記 §トップレベル宣言 はこのグループを値束縛にも
拡張する。

本文書では局所 `let` は（01 と同様）単一束縛のままなので、型レベル
の局所相互再帰は本層では到達不能である。多束縛 `let` と合わせて後
続文書に送る。

## 型付け規則

### コンストラクタの型スキーム

与えられた

    data T a₁ ... aₙ = C₁ τ₁₁ ... τ₁ₖ₁ | ... | Cₘ τₘ₁ ... τₘₖₘ

に対して、各値コンストラクタ `Cᵢ` は次のスキームで環境に導入される：

    Cᵢ : ∀ a₁ ... aₙ. τᵢ₁ → ... → τᵢₖᵢ → T a₁ ... aₙ

0 引数コンストラクタ（`kᵢ = 0`）は
`Cᵢ : ∀ a₁ ... aₙ. T a₁ ... aₙ` を受け取る。

各 `Cᵢ` は `a₁ ... aₙ` の **全て** に対して量化される。`Cᵢ` の引数型
に現れない型パラメータであっても量化対象である — 例えば
`data Maybe a = Nothing | Just a` における `Nothing` は
`Nothing : ∀ a. Maybe a` を受け取り、`Nothing : Maybe a₀` のような
暗黙の単相 `a₀` を持つわけではない。具体的な `a` は使用箇所で
`instantiate` により文脈から回収される。

値コンストラクタは elaboration 後は通常の項レベル識別子である。式
としての使用は文書 01 の (Var) 規則をこのスキームに適用して型付け
する。新しい判定形式は不要であり、"引数不足コンストラクタ" の特別
扱いも不要である。特に `Just`（部分適用、あるいは第一級関数として
の利用）は `∀ a. a → Maybe a` 型を持ち、`Just 1` は `Maybe Int` 型
を持つ。

### 再帰的 let（文書 01 の (Let) を改訂）

文書 01 は未解決の問い 1 として、`let` 束縛が自己を言及できるかを
残していた。本文書はこれを決着させる：**`let` 束縛は暗黙に再帰的で
ある。** 文書 01 の (Let) 規則は以下で置き換わる：

```
Γ, x : τ₁ ⊢ e₁ : τ₁     Γ, x : generalize(Γ, τ₁) ⊢ e₂ : τ₂
—————————————————————————————————————————————————————————————   (Let)
              Γ ⊢ (let x = e₁ in e₂) : τ₂
```

束縛 `x : τ₁` は `e₁` の elaboration 中スコープに入る（単相なので
`e₁` は `x` を単一の型で見る）。`e₂` の elaboration 中は一般化され
たスキームがスコープに入る。これは implicitly recursive `let` に対
する標準的な宣言的 Hindley–Milner 規則であり、`e₁` の elaboration
前に `x` 用の新鮮な単一化変数を一つ割り当てれば、アルゴリズム W あ
るいは J で実装可能である。

`e₁` 内での `x` の再帰的使用は同じ単相 `τ₁` で行われる：**多相再帰は
本層では許さない**。多相再帰（型注記なしでは独立に決定不能）のサポー
トは本文書では検討しない。

### トップレベル宣言

プログラムはトップレベル宣言列 `decl₁ ... declₙ` である。**全ての
トップレベル宣言 — `data` 宣言と値束縛 — は単一の相互再帰グループを
成す。** 具体的には：

- 全ての `data` 宣言の型コンストラクタと値コンストラクタは、他の全
  ての `data` 宣言の右辺・全てのトップレベル型シグネチャ・全ての値
  束縛の右辺のスコープに入る。
- 全てのトップレベル値束縛はテキスト順序によらず、他の全てのトップ
  レベル値束縛の右辺のスコープに入る。

これに対応する elaboration：

1. 全ての `data` 宣言から型コンストラクタとコンストラクタ・スキーム
   を集め、環境 `Γ_data` を得る。
2. 各トップレベル値束縛 `xⱼ = eⱼ` について、`xⱼ` を新鮮な単相変数
   `αⱼ` に束縛し、
   `Γ₀ = Γ_data, x₁ : α₁, ..., xₙ : αₙ` を作る。
3. 各束縛について `Γ₀ ⊢ eⱼ : αⱼ` を導く（生じる単一化制約を解く。
   01 と同じ要領）。
4. 解かれた各 `αⱼ` を **`Γ_data` の下でのみ** 一般化する — 他の
   `xⱼ` たちは同じ相互再帰グループの構成員であり、互いの一般化文
   脈から意図的に除外される。これによりその束縛のスキームが得られ
   る（後続のモジュールや下流の問い合わせで利用）。

これは相互再帰グループに対する通常の Damas–Milner elaboration であ
り、標準的なアルゴリズム W の変種で実装可能な想定である。

### 型シグネチャとの相互作用

トップレベル型シグネチャ `x : τ` が存在する場合、対応する束縛
`x = e` に注記される。上記 elaboration の手順 2 は、新鮮な単相変数
`αₓ` ではなく、シグネチャから得たスキーム（シグネチャ中の自由型変
数を Hindley–Milner の意味で普遍量化されたものとして扱う）を使う。

トップレベルシグネチャを必須とするか任意とするかは本層では決めな
い。01 未解決の問い 2 を参照、M4（モジュール）で決着する。

## 設計メモ（非規範的）

- **パターンマッチは後送り。** 本文書の規則だけでは、Sapphire プロ
  グラムはデータ型の値を構築できるが分解できない。M3（パターンマッ
  チ）が物語を完結させる。それまで `case ... of` は利用不可であり、
  本文書の例はいずれも "書き込み専用" である — M3 なしでは端から
  端まで動かせない。これは意図的な層構造の選択であり、型の導入（本
  文書）と消去形（M3）の導入を独立に扱えるようにするためである。

- **`Bool` はただの ADT。** 例 `data Bool = False | True` は例示で
  しかない。`Bool` をこの規則で定義して prelude (M6) に乗せるか、そ
  れとも 02 の `BOOL` トークンと 01 の (LitBool) 規則をプリミティブ
  のまま保つかは、02 未解決の問い 1 の主題であり、M6 で決着する。
  どちらの立場も本文書の `data` 機構を前提とする。違うのは `Bool`
  に限ってその機構を通すか否かだけである。

- **高階種は後送り。** `n` 個の引数を取る `data` 宣言は、その型コ
  ンストラクタが常に `n` 個の引数を伴って現れるよう強制する。`j < n`
  に対する部分適用 `T τ₁ ... τⱼ` は整形な型ではない。高階多相、型
  コンストラクタ上の型クラス / トレイト、`Functor` 形の抽象は全て
  保留。Sapphire が Elm に倣うか（`*` を超える種は決して認めない）、
  緩めるかは、アドホック多重定義の着地に結びつく後の判断（01 未解
  決の問い 5、02 未解決の問い 3）である。

- **暗黙的量化を行わない。** `data T a = Foo b` は拒絶される：左辺
  に束縛されていない型変数 `b` が右辺に現れれば静的エラーである。
  これは Elm と一致し、Haskell では `ScopedTypeVariables` なしの
  既定と一致する。

- **`let rec` ではなく暗黙再帰。** 全ての `let` を暗黙に再帰的とし、
  別途 `let rec` を要求しないのは Elm / Haskell の既定であり、再帰
  的な `data` 型が存在した瞬間から Sapphire の動機例も黙って前提と
  する形である。パフォーマンス動機での再帰/非再帰束縛の区別は本層
  では表層に露出しない。問い 1 を参照。

## 未解決の問い

1. **暗黙再帰 vs. 明示的 `let rec`。** 本文書は `let`（およびトッ
   プレベル）に暗黙再帰を採用する。これを見直す余地 — 再帰束縛用に
   `let rec` を分離し、非再帰束縛には 01 の原 (Let) を残す — は、
   読みやすさや、性能に敏感な非再帰束縛が一般的な関心事になった場
   合に残る。既定の再訪点は M10。

2. **局所多束縛 `let` と局所相互再帰。** 局所相互再帰には多束縛
   `let` 形式（`let x = e₁ ; y = e₂ in ...`）が必要。本文書は局所
   `let` を単一束縛のままとし、その形式と局所相互再帰を後続文書へ
   送る。そのような形式が単一の相互再帰グループを成すのか、それと
   も逐次シャドウイングの連鎖にすぎないのかは未決。

3. **評価戦略。** ここでの型付け規則は評価順序から独立である。
   Sapphire の表層意味論を strict-by-default（Elm / ML 流）に固
   めるのか、lazy-by-default（Haskell 流）に固めるのか、あるいは
   実装言語フェーズに委ねるのかは保留。

4. **`deriving` / 自動インスタンス生成。** 表層に
   `data T ... = ... deriving (Eq, Show)` 節を認めるか、それとも派
   生は全て後続の型クラス / トレイト / 慣習駆動機構に委ねるか。本
   文書は派生機構を一切導入しない。問いは後続文書がそれを導入すべ
   きかどうか。

5. **コンストラクタ名の名前空間とシャドウイング。** 関連する二つの
   副問い。いずれも本文書は沈黙している。(a) 値コンストラクタは通常
   の項レベル識別子と同じ名前空間を共有するのか、それとも別の構文
   クラス（例えば `Just` は無条件にコンストラクタであり `let Just = 1`
   は構文エラー）なのか。(b) 名前空間を共有する場合、入れ子スコープ
   でコンストラクタ名を再束縛してよいか（`let Just = 1 in Just`）。
   どちらの決着も M3（パターンマッチ）と相互作用する — M3 ではコ
   ンストラクタパターンと変数パターンを構文的に区別しなければなら
   ず、両問いはおそらく同時に決着する。

6. **局所 `data` 宣言。** 本文書は `data` をトップレベル専用とす
   る。後続文書が局所 `data`（例えば `let` ブロック内）を導入する
   かは未決。M2/M3 は明らかには要求しないが、M4（モジュール）と
   M6（prelude）で要求が出る可能性はある。既定の再訪点は M10。
