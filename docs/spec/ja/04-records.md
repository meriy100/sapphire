# 04. レコード

状態: **draft**。パターンマッチ (M3) やモジュール (M4) の着地に伴い
改訂されうる。

本文書は `docs/spec/04-records.md` の日本語訳である。英語版が規範的
情報源であり、BNF 生成規則・型付け規則・規則名・番号付き未解決の問い
は英語版と一致させて保つ。

## 動機

レコードは Sapphire の **名前付きフィールドを持つ product 型** で
ある。文書 03 は既に位置引数を持つ ADT（`data Pair a b = Pair a b`）
を導入しているが、位置引数は 2〜3 要素を超えると扱いにくく、宣言順
序の情報がすべての利用箇所に漏れ出す。レコードは各要素に名前を付
けることで、構築・アクセスを宣言順序から切り離す。

本文書は以下を固定する：

- レコード **型** の書き方と、2 つのレコード型の等価性。
- レコード **値** の構築方法。
- レコードから個別のフィールドを **選択** する方法。
- いくつかのフィールドを置き換えた新しいレコードを得るための
  **更新** 方法。

以下のプログラムは本層が受理しなければならない基準例である：

```
-- レコード型とリテラル
let origin = { x = 0, y = 0 } in origin

-- フィールド選択
let p = { x = 3, y = 4 } in p.x

-- 関数的更新（新しいレコードを作る）
let p = { x = 3, y = 4 } in
let p' = { p | x = 10 } in
p'

-- データ型の内部にあるレコード
data Shape = Circle { cx : Int, cy : Int, r : Int }
           | Rect   { x1 : Int, y1 : Int, x2 : Int, y2 : Int }
```

最後の例は **例示目的のみ**：レコード形のコンストラクタ payload を
本層で認めるかは未解決の問いである。§設計メモ と問い 2 を参照。本文
書の残りは、式と型の形として現れる裸のレコードだけを規定する。

本層では意図的に保留：

- **レコードの分解／パターンマッチ**。M3。
- **行多相**（`{ f : Int | r }` 形式の拡張可能レコード）。
  draft は **クローズドな構造的レコード**（Elm 0.19 準拠）を採用
  する。行多相は問い 1。
- **レコードパニング構文**（`{ x, y }` を `{ x = x, y = y }` の
  省略形とする）。問い 3。
- **モジュール修飾フィールド名**。M4 で修飾名と合わせて決着。
  `.` と修飾名の相互作用は §設計メモ で触れる。

## 抽象構文（BNF）

文書 01 および 03 を拡張：

```
expr       ::= ...                                (01, 03)
             | '{' field_bindings? '}'            -- レコードリテラル
             | '{' expr '|' field_bindings '}'    -- レコード更新
             | expr '.' lower_ident               -- フィールド選択

field_bindings ::= field_binding (',' field_binding)*
field_binding  ::= lower_ident '=' expr

type       ::= ...                                (01, 03)
             | '{' field_types? '}'               -- レコード型

field_types ::= field_type (',' field_type)*
field_type  ::= lower_ident ':' type
```

フィールド名は `lower_ident`（02 §識別子）。通常の項識別子や型変数
と同じクラス。

レコード型およびリテラルが静的に整形であるためには：

- 一つのレコード型（またはリテラル）内のフィールド名は互いに相異
  でなければならない。`{ x : Int, x : String }` は静的エラー。
- 空レコード `{}` は型としても値としても許される。§設計メモ 参照。

下記 §曖昧性 で文法の曖昧性解消について触れる。

### 曖昧性

`{ e | ... }`（レコード更新）と `{ f = e, ... }`（レコードリテラル）
は同じ開き波括弧で始まるため、パーサは 1 トークン以上の先読みが必
要になる。どちらの選択枝かを決めるのはパーサの関心事であり、言語
レベルの曖昧性ではない。

`expr '.' lower_ident` の `.` は、02 がモジュール修飾名（M4 で決
着）用に予約した `.` と同じ記号である。本文書での解決：パース位置
において、式の直後に置かれ `lower_ident` が続く `.` はフィールド
選択。`upper_ident`（モジュール名）と任意の識別子のあいだの `.`
は修飾名であり、それは M4 まで導入されない。`forall TVAR* '.' type`
におけるスキーム記法の `.` は（01）スキーム位置のみで現れ、式レベ
ルのフィールド選択と衝突しない。

フィールド選択は **左結合**：`p.x.y` は `(p.x).y` として解析され、
`p.(x.y)` ではない。これは `.f` を直前の式に適用する演算子として
扱う BNF から自然に従う。

`field_bindings` および `field_types` では末尾カンマは **認めない**。
BNF `field_binding (',' field_binding)*` により各 `,` の後には続く
束縛が必ずなければならない。`{ x = 1, }` は構文エラー。

## 型

レコード型

    { f₁ : τ₁, ..., fₙ : τₙ }   ただし各 fᵢ は互いに相異

は種 `*` を持つ。2 つのレコード型が等しいのは、フィールド名の集合
が一致し、共有する各フィールドの型が等しい場合かつその場合に限る。
**フィールド順序は意味論的に有意ではない** — `{ x : Int, y : Int }`
と `{ y : Int, x : Int }` は同じ型。

レコード型はフィールド型中に型変数を含みうる。そのようなレコード
型自体は何も量化しない。一般化は他のすべての型と同じく `let`（01）
またはトップレベル束縛（03）でのみ行われる。

レコードは **構造的**：レコード型にはフィールド集合以外の nominal
タグは存在しない。異なる 2 つの `data` 宣言が "同じ" レコード形を
異なる型として包むことは可能だが、レコード層自身はそれらを区別し
ない。

## 型付け規則

### レコードリテラル

```
Γ ⊢ e₁ : τ₁    ...    Γ ⊢ eₙ : τₙ     （各 fᵢ は互いに相異）
————————————————————————————————————————————————————————————————   (Rec)
 Γ ⊢ { f₁ = e₁, ..., fₙ = eₙ } : { f₁ : τ₁, ..., fₙ : τₙ }
```

空レコードは (Rec) の `n = 0` の場合に相当する：

```
————————————————————     (RecEmpty)
 Γ ⊢ { } : { }
```

### フィールド選択

```
 Γ ⊢ e : { f₁ : τ₁, ..., fₙ : τₙ }    j ∈ {1, ..., n}
—————————————————————————————————————————————————————   (Sel)
              Γ ⊢ e.fⱼ : τⱼ
```

レコード以外の型に対する選択は型エラー。レコードの静的型に存在し
ないフィールド名の選択も型エラー。暗黙の構造包摂は行わない：
`{ x : Int, y : Int }` 型のレコードが本層で `{ x : Int }` 型も持
つわけではない（別の選択肢については問い 1 参照）。

### レコード更新

```
Γ ⊢ e : { f₁ : τ₁, ..., fₙ : τₙ }
Γ ⊢ eᵢ₁ : τᵢ₁    ...    Γ ⊢ eᵢₖ : τᵢₖ       {i₁, ..., iₖ} ⊆ {1, ..., n}
更新対象フィールド名はすべて相異
——————————————————————————————————————————————————————————————————————   (Upd)
Γ ⊢ { e | fᵢ₁ = eᵢ₁, ..., fᵢₖ = eᵢₖ } : { f₁ : τ₁, ..., fₙ : τₙ }
```

更新は入力レコードの型を保つ：新しいフィールドを導入することも、
既存のフィールドを取り除くこともない。置き換え式の型は対応するフ
ィールドの宣言された型と等しくなければならない。`e` の静的型に存
在しないフィールドを更新しようとすれば型エラー。

0 個のフィールドを更新する `{ e | }` は文法に含まれない：
`field_bindings` は最低 1 個のフィールド束縛を要求する。

## 設計メモ（非規範的）

- **クローズドな構造的レコード。** 本層は完全に指定されたレコード
  型のみを認める。レコード引数を取る関数は期待するフィールドを正
  確に述べなければならず、「少なくとも `x : Int` を含む任意のレ
  コード」のように抽象化することはできない。別の選択肢 — 行多相
  — は後送り（問い 1）。Elm はクローズドな道を選択（初期バージョ
  ンで拡張可能レコードを試した後に撤退）しており、Sapphire は第一
  歩としてそれに倣う。

- **空レコード。** `{}` は型 `{}` を持つ。レコード層における単位
  的な値であり、主にデータを運ばない Ruby インタロップ (M7) payload
  の型付きプレースホルダーとして機能する。

- **レコード vs. 名前付きフィールド付き `data`。** 文書 03 はレコー
  ド形のコンストラクタ payload を本文書に明示的に送っていた。
  `data` 宣言のコンストラクタが名前付きフィールドを持てるか
  （`data Shape = Circle { cx : Int, cy : Int, r : Int }`）、そして
  そのような宣言が位置引数 `data` に匿名レコードを包んだ形に糖衣展
  開されるかは問い 2。動機例冒頭はその糖衣形式を可読性のために仮定
  しているが、規範ではまだ許可していない。

- **フィールド名は項識別子名前空間を共有する。** フィールド名は
  `lower_ident`。スコープ内の項レベル束縛と一致してもよい。
  `let x = 1 in let p = { x = 2 } in p.x` における `p.x` は `2`
  であり `1` ではない — `.f` は環境中で `f` を参照しない専用の選
  択形である。

- **本層での行単一化はない。** 行多相がないため、レコード型は
  フィールド集合上の構造等価性で決定される — 行変数もなく、行単
  一化も要らない。後から行多相を導入するには行単一化の手続きが必
  要になる。保留することで、M2 は 01 と 03 が既に前提とする機構
  で決定可能に保たれる。

- **`this` とメソッドディスパッチはスコープ外。** Sapphire はオブ
  ジェクト指向言語ではない。レコードはデータであり、暗黙の receiver
  も継承もメソッドも持たない。呼び出し側は射影されたフィールドに
  対する通常の関数適用を行う（`p.describe.x` ではなく
  `(describe p).x`）。

## 未解決の問い

1. **行多相。** Sapphire は拡張可能レコード型
   （`{ f₁ : τ₁, ..., fₙ : τₙ | r }`、ここで `r` は行変数）を許す
   べきか。`はい` の答えは「少なくともこれらのフィールドを持つ任
   意のレコード」に対して多相な関数を可能にするが、代償として型検
   査器に行単一化が入り、01 未解決の問い 5 の「型クラス風解決」の
   選択肢とも相互作用する。draft の回答は **いいえ**（クローズド
   レコード、Elm 0.19 風）。

2. **`data` におけるレコード形コンストラクタ payload。** 順に 2 つ
   の問い。第一に、`data` コンストラクタが名前付きフィールドの宣
   言を持てるか（例：`data Shape = Circle { radius : Int }`）。第
   二に、もし持てるなら、その表層形は **プリミティブ**（各フィー
   ルドがそれ自身トップレベルのセレクタになる、Haskell のレコード
   構文付き ADT 風）か、それとも **糖衣** か — 糖衣の場合、単一
   の匿名レコード引数（型は本文書のレコード型）を取る位置引数コン
   ストラクタへ展開される。宣言位置では両者は同じ見た目だが、生じ
   るセレクタとパターンが異なる。本文書は立場を取らない。この設計
   はパターンマッチ (M3) と相互作用する。
   *2026-04-18 決定*: 最初の実装では名前付きフィールドコンストラ
   クタ payload を認めない。コンストラクタ引数は位置引数。名前付
   きアクセスが欲しいユーザは、レコード型を宣言し、それを単一引
   数としてコンストラクタに渡す（例：
   `type Circle = { radius : Int }; data Shape = Circle Circle`）。
   需要が出れば (b)「糖衣形」を純粋追加として再訪する余地は残る。

3. **パニング短縮形。** レコードリテラルで `{ x, y }` を
   `{ x = x, y = y }` の省略形として認めるか（M3 が入った後は、パ
   ターンとしての `{ x, y }` も）。`はい` の答えは PureScript / Elm
   に揃う。`いいえ` の答えはパーサも教材もより簡単。
   *2026-04-18 決定*: いいえ。ユーザは明示形 `{ x = x, y = y }`
   を書く。

4. **対称的更新（フィールド追加・削除）。** `{ r | f = v }` は現
   状、既存フィールドの値を置き換えてレコードの型を変えない。直交
   的な拡張として、`r` にまだ **含まれない** フィールドについての
   `{ r | f = v }` を許し、1 つフィールドが増えたレコードを生む、
   という方向もありうる。これは行多相（問い 1）が存在する場合にの
   み意味を持つ。完全性のためここに記す。

5. **レコード間のフィールド名衝突。** `{ x = 1 }.x` と
   `{ x = "hi" }.x` はいずれも異なる型で well-typed。後続のモジュー
   ル／型クラス機構が、インポート間で曖昧なフィールド名の非修飾使
   用を禁止するか（Haskell の `DuplicateRecordFields` 論争）は本
   文書の範囲外。M4 で再訪する。
