# 01. 中核式言語

プ）で見つかった制約は core に反映されうる。

本文書は `docs/spec/01-core-expressions.md` の日本語訳である。英語版が規範
的情報源であり、BNF 生成規則・型付け規則・予約語集合・番号付き未解決の問い
は英語版と一致させて保つ。

## 動機

中核式言語は、今後のすべてのレイヤー（ADT、レコード、モジュール、Ruby イン
タロップ、`RubyEval` モナド）がその上で語られる基盤である。意図的に最小構
成に抑えてある：λ、適用、`let`、`if`、リテラル、および let-一般化を伴う
Hindley–Milner 型。

以下のプログラムはこの層が受理しなければならない基準例である。表層構文は
暫定であり、厳密な字句規則（演算子、予約語、レイアウト規則）は別文書の主
題とする。

```
-- 恒等関数を適用
let id = \x -> x in
id 42

-- 大きい方を選ぶ
let max = \a -> \b -> if a > b then a else b in
max 3 7

-- let-多相：同じ `id` を異なる二つの型で使う
let id = \x -> x in
let _x = id 1 in
id "hello"

-- 明示的シグネチャを伴うトップレベル
double : Int -> Int
double = \n -> n + n
```

これ以上のもの（データコンストラクタ、レコード、モジュール、作用）は後続
文書へ送り、文法と型規則を単調に拡張するのみとする。

## 抽象構文（BNF）

字句トークン（`IDENT`、`INT`、`STRING`、`BOOL`、`TVAR`、`TCON`）は本層で
は所与として扱う。具体的な正規表現は字句規則文書の担当である。

```
program   ::= decl*

decl      ::= IDENT ':' type                     -- 型シグネチャ
            | IDENT '=' expr                     -- 定義

expr      ::= literal
            | IDENT                              -- 変数
            | '\' IDENT '->' expr                -- λ（単一引数）
            | expr expr                          -- 適用、左結合
            | 'let' IDENT '=' expr 'in' expr     -- 局所束縛
            | 'if' expr 'then' expr 'else' expr  -- 条件式
            | '(' expr ')'

literal   ::= INT | STRING | BOOL

type      ::= TVAR                               -- 型変数
            | TCON                               -- Int | Bool | String
            | type '->' type                     -- 右結合
            | '(' type ')'

scheme    ::= 'forall' TVAR* '.' type            -- 表層では暗黙、内部用
```

本層では意図的に保留：

- 多引数 λ `\a b -> ...`：方針決定後に、単引数 λ の入れ子への糖衣構文とし
  て扱う。
- 組み込み演算子（`+`、`-`、`>`、…）：本層のプリミティブではない。ADT と
  解決機構が揃ってから導入するか、あるいは prelude で型が与えられた通常
  の識別子として導入する。
- `let rec` と相互再帰：下の *未解決の問い* を参照。

## 型

本層には次の三つの基底型定数のみが存在する：

- `Int`
- `Bool`
- `String`

関数矢印 `->` は右結合。

**型スキーム** は（空でありうる）普遍量化された型変数の接頭を持つ。一般化
は `let` 束縛で行われる。λ 束縛変数は決して一般化されない。

## 型付け規則

Γ を識別子から型スキームへの環境とする。判断

    Γ ⊢ e : τ

は「Γ のもとで式 `e` は型 `τ` を持つ」を表す。

```
                   σ = Γ(x)
——————————————————————————————————————            (Var)
        Γ ⊢ x : instantiate(σ)


       Γ, x : τ₁ ⊢ e : τ₂
————————————————————————————————                  (Abs)
 Γ ⊢ (\x -> e) : τ₁ -> τ₂


Γ ⊢ e₁ : τ₁ -> τ₂     Γ ⊢ e₂ : τ₁
———————————————————————————————————               (App)
       Γ ⊢ e₁ e₂ : τ₂


Γ ⊢ e₁ : τ₁     Γ, x : generalize(Γ, τ₁) ⊢ e₂ : τ₂
———————————————————————————————————————————————————   (Let)
        Γ ⊢ (let x = e₁ in e₂) : τ₂


Γ ⊢ c : Bool     Γ ⊢ t : τ     Γ ⊢ f : τ
——————————————————————————————————————————        (If)
       Γ ⊢ (if c then t else f) : τ


——————————————  (LitInt)    ——————————————  (LitBool)    ————————————————  (LitStr)
 Γ ⊢ n : Int                 Γ ⊢ b : Bool                 Γ ⊢ s : String
```

補助定義：

- `instantiate(∀α₁…αₙ. τ)` は各 `αᵢ` を新鮮な単一化変数で置換した単相型を
  返す。
- `generalize(Γ, τ) = ∀(ftv(τ) \ ftv(Γ)). τ`、ここで `ftv(τ)` は `τ`
  の自由型変数の集合、`ftv(Γ) = ⋃ { ftv(σ) | (_ : σ) ∈ Γ }` は同じ概
  念を環境へ持ち上げたもの。

これらは標準的な宣言的 Hindley–Milner 規則であり、アルゴリズム W（あるい
は J）で驚きなく実装可能な想定である。

## 設計メモ（非規範的）

- `if ... then ... else ...` は **式** であって文ではない。両枝は同じ型を
  持たなければならない。ADT として `Bool = True | False` が利用可能にな
  った時点で、`if` は `case` の糖衣構文として再規定されうる —— 下の問 3
  を参照。
- `BOOL` トークンと (LitBool) 規則は、`True` と `False` がリテラルとして
  中核層に届くことを前提とする。後の仕様作業が `True` / `False` を通常
  の prelude 束縛識別子と決めた場合、(LitBool) は (Var) から導出可能と
  なり、本層はそれを落とせる。字句構文文書の問 1 を参照。
- トップレベル型シグネチャは構文的には許容されるが、必須とはまだ宣言され
  ていない。問 2 を参照。
- 表層言語はここで明示的な `let ... in ...` を選んでいる。トップレベル
  の形が現行ドラフトの `decl*` に近いのか、それとも一つの大きな
  `let ... in main` のようになるのかは、モジュールとエントリポイント設
  計と結びついた後続決定である。

## 未解決の問い

1. **`let` における再帰と一般化。** すべての `let` は一般化 *かつ* 自己参
   照を許すべきか、それとも再帰は別の `let rec` 形式を要求するべきか？
   （現ドラフトは自己参照について沈黙しており、書かれた (Let) 規則では
   `e₁` が `x` を言及することは許されない。）
   *文書 03 で決着*（データ型）：`let` は暗黙に再帰的とする。文書 03 は
   本文書の (Let) 規則を置き換える。この未解決の問いは経緯を残す目的で
   本文書に残置し、M10 で再訪しうる。
2. **トップレベルシグネチャ：必須か任意か。** 必須方針はモジュール境界
   でのエラーメッセージと主要型問題を単純化する。任意方針はより親しみや
   すいが、将来の型クラス風機構と相互作用する。
   *文書 08 で決着*：境界規則。エクスポートされるトップレベル値束縛は
   明示的シグネチャを要求、プライベートなものは不要。
3. **`if` はプリミティブか糖衣か。** (If) をプリミティブ規則として保持す
   るか、あるいは `Bool = True | False` が存在した時点で
   `if c then t else f` を `case c of { True -> t ; False -> f }` の糖衣
   として再規定し、プリミティブ規則を落とすか。
   *文書 06 で部分的に対処*：`case` が利用可能になり、糖衣の読みは機構
   的には可能になった。
   *文書 09 で決着*：糖衣。prelude が `Bool` を ADT
   `data Bool = False | True` として定義することにより、
   `if ... then ... else ...` は対応する `case` の表層糖衣となる。
   (If) は読者のための導出可能な補題として 01 に残す。
4. **数値階層。** 本層では単一の `Int` のみを置き、`Float` を別型として
   加えるか、統一 `Number` を導入するかは後の決定とする。本文書は前者が
   少なくとも可能であることを前提とする。
   *文書 05 で部分的に決着*：M5 層では数値タワーは `Int` のみ。`Float`
   か統一 `number` かの選択は 05 未解決の問い 1 に再度立てる。
5. **組み込み演算子。** 算術と比較の演算子は prelude で固定単相型を持つ
   プリミティブ定数か、それとも後のアドホック / 型クラス風機構で解決さ
   れるか。動機例は `+` と `>` を使うが、本文書はまだ型を割り当てていな
   い。
   *文書 05 で決着*：演算子は prelude 束縛の原始値として固定単相型を持
   つ。表は文書 05 が定める。`Int` を超えた等価・順序比較の拡張は 05
   未解決の問い 2 に再度立てる。
