# 08. モジュール

状態: **draft**。prelude (M6) および Ruby 相互運用文書 (M7 / M8)
がモジュール境界を使い込む過程で改訂されうる。

本文書は `docs/spec/08-modules.md` の日本語訳である。英語版が規範的
情報源であり、BNF 生成規則・規則名・番号付き未解決の問いは英語版と
一致させて保つ。

## 動機

本文書までの Sapphire ソースファイルは、すべて単一の無名名前空間
の中にあった。本文書は **モジュール** を導入する：名前を持つファイ
ル単位のコンテナで、可視性を区切り、修飾名記法（02 の `.`）を共有
し、ファイル間の合成を well-defined にする。

範囲内：

- モジュール宣言構文（エクスポートリストを含む）。
- スコープへ名前を取り込む `import` 構文、修飾／非修飾。
- 修飾名解決（`Mod.name`）。
- モジュール階層（`Data.List` 形式）。
- 可視性規則と再エクスポート。
- モジュール境界におけるシグネチャ規律（01 未解決の問い 2 を決
  着）。
- 07 の孤児インスタンス規則を実体化した形で grounding する。

意図的に保留：

- 循環モジュール依存。本層では禁止。M9 例題プログラムが要求する場
  合のみ再考する。
- パッケージ境界（複数パッケージ、依存解決、ビルド成果物）。
- 型を公開してコンストラクタを隠す形（Haskell の `Type(..)` と
  `Type` の差）。08 未解決の問い 2。
- モジュール間可視性との相互作用を伴う `deriving` 節。依然として
  03 未解決の問い 4 / M6 の範疇。

## 1 ファイル 1 モジュール

Sapphire のソースファイルは **それ自体がモジュール** である。モジュ
ール名はプロジェクトルートからのパスに対応する：

- `src/Main.sp` はモジュール `Main` を定義する。
- `src/Data/List.sp` はモジュール `Data.List` を定義する。
- `src/Data/List/Extra.sp` はモジュール `Data.List.Extra` を定義
  する。

プロジェクトルートとモジュールパスの対応はビルドツールの慣習であ
り、本層では規範ではない。**規範であるもの**：

- 1 ファイルにつき `module` 宣言は正確に 1 つ。`module` キーワー
  ドを含むときは、それがファイル内で最初の非コメント・非空白トー
  クンでなければならない。
- 宣言したモジュール名はコンパイルルートからのファイルのドット
  付きパスと一致しなければならない（不一致はビルドツールが報告）。
- 同じコンパイル内で同名のモジュールを複数のファイルが宣言しては
  ならない。

明示的な `module` 宣言を持たない単一ファイルのコンパイルは、
`module Main where ...` の糖衣として認める。これは小さな例のため
の利便機能である。

## 抽象構文（BNF）

文書 01 / 03 / 04 / 05 / 06 / 07 を拡張：

```
program    ::= module_header? decl*                 -- (01 を改訂)

module_header ::= 'module' mod_name export_list? 'where'

mod_name   ::= TCON ('.' TCON)*                     -- ドット付き UpperIdent

export_list ::= '(' export_item (',' export_item)* ')'
              | '(' ')'                             -- 何もエクスポートしない

export_item ::= IDENT                               -- 値をエクスポート
              | TCON                                -- 型だけエクスポート
                                                    -- （コンストラクタは隠す）
              | TCON '(' '..' ')'                   -- 型と全コンストラクタ
              | TCON '(' TCON (',' TCON)* ')'       -- 型と選択した
                                                    -- コンストラクタ
              | 'class' TCON                        -- クラス名のみ
                                                    -- （メソッドは隠す）
              | 'class' TCON '(' '..' ')'           -- クラスと全メソッド
              | 'module' mod_name                   -- モジュール再エクスポート

decl       ::= ...                                  -- (01, 03, 07)
             | import_decl

import_decl ::= 'import' mod_name import_tail
              | 'import' 'qualified' mod_name import_tail

import_tail ::= ε
              | 'as' TCON
              | 'as' TCON '(' import_item (',' import_item)* ')'
              | '(' import_item (',' import_item)* ')'
              | 'hiding' '(' import_item (',' import_item)* ')'

import_item ::= IDENT
              | TCON
              | TCON '(' '..' ')'
              | TCON '(' TCON (',' TCON)* ')'
              | 'class' TCON
              | 'class' TCON '(' '..' ')'
```

BNF への注記：

- `mod_name` は 02 の `upper_ident` と `.` 予約を再利用する。モ
  ジュール名は空でないドット付き `upper_ident` の列。
- 明示的な空エクスポートリスト `module M () where ...` は認める —
  何も公開しないモジュールを命名する。インスタンス解決のためだけ
  にインポートされる純粋な内部モジュールで有用：名前は見えないが、
  モジュールのインスタンスは依然としてモジュールと共に移動するた
  め（§インスタンスとモジュール 参照）、`import M` はそれらのイン
  スタンスをスコープに入れる。
- エクスポートリストの **省略** と空リストは異なる：括弧なしの
  `module M where ...` は、全トップレベル非インポート束縛・全
  `data` コンストラクタ・全 `class` メソッドをエクスポートする。
  Haskell の既定と一致。ライブラリコードでは明示リストを推奨。
- `import qualified M` はインポート側モジュールから `M.name`（あ
  るいは `as L` を使えば `L.name`）経由のみの参照を要求する。非
  修飾の裸の `name` は解決しない。
- `hiding (...)` は、列挙された名前を除いた通常 `import` に等価。
  明示 `(...)` インポートリストとは併用できない。
- `as L` はモジュールの修飾プレフィックスをローカルで `L` に改名
  する。`qualified` を伴わない限り、元の名前も修飾アクセス用に残
  る。`qualified` を伴う場合、別名のみが解決する（Haskell の同じ
  キーワード組み合わせの意味論と一致）。
- 結合形 `import M as L (foo, bar)` は 2 つの役を同時に果たす：修
  飾プレフィックスは `L` になり、非修飾スコープは `foo` と `bar`
  に制限される（型／クラスの項指定も同様に選別）。`L.otherName`
  による修飾アクセスはリスト外のエクスポートにも依然として可能。
- 新規キーワード：`as`・`hiding`・`qualified`。いずれも文書 02 の
  予約語集合に追加拡張条項のもと加える。

## 可視性

モジュール `M` のトップレベル宣言は次のいずれか：

- **エクスポート済** — `import M` するモジュールから可視。
- **プライベート** — `M` 内部からのみ可視。

明示的なエクスポートリストを伴う場合、リストされた名前がエクスポー
トされ、それ以外はプライベート。エクスポートリスト省略の場合、全
トップレベル名がエクスポートされる（§BNF への注記 参照）。

エクスポートされた `data` 宣言は型コンストラクタをエクスポートする
が、`export_item` が明示的に列挙しない限り、値コンストラクタはエク
スポートしない（`Maybe(..)` や `Maybe(Just)`）。これは Haskell の抽
象データ型の慣習と一致する：エクスポートリスト中の裸の `Maybe` は
型名をシグネチャで使えるようにするが、その表現を露出しない。

エクスポートされた `class` 宣言はクラス名をエクスポートするが、
`class C(..)` と列挙しない限り、メソッド名はエクスポートしない。
エクスポートリストの `class C` はクラス名のみの露出であり、インポ
ート側は `C a => ...` のような制約位置で `C` を参照できるが、そ
のメソッドを識別子として使うことはできない。特に、メソッド名が
スコープに入らないため、外部モジュールは `instance C T where m =
...` を書けない。`(..)` なしでエクスポートされたクラスは、インス
タンスが全て宣言元モジュールで定義される閉じた抽象として振る舞う。
`(..)` を伴って公開されたクラスは通常の開いた抽象であり、外部モジ
ュールはそのメソッドを参照し、自前の型に対するインスタンスを書け
る（§インスタンスとモジュール の孤児規則に従う）。

クラスメソッドのエクスポート粒度は本 draft では **全か無か**
（`class C` または `class C(..)`、個別選択なし）。より細かい粒度
は 08 未解決の問い 7。

### インポートされた名前とスコープ

非修飾の `import M` 文は `M` のエクスポート名をインポート側モジュ
ールの **非修飾** スコープに全て入れる。同時に `M`（あるいは `as L`
が与えたエイリアス）で修飾したスコープにも入る。

`import qualified M` はエクスポート名を **修飾** スコープにのみ
入れる。非修飾参照は解決しない。

`import M (foo, Bar(Baz))` は非修飾スコープを列挙した名前に限定す
る（この例では `foo` と型 `Bar` のコンストラクタ `Baz`）。修飾形
`M.otherName` はインポートされていないエクスポートにもなお解決する
— インポート済みモジュールについては修飾アクセスが常に許される。

2 つの非修飾インポートが同じ名前を露出する場合、使用箇所で名前は
**曖昧** になる。非修飾での使用は静的エラーとなる。ユーザは参照を
修飾するか（`M1.foo` vs `M2.foo`）、片方のインポートを削除するこ
と。

## 修飾名解決

修飾参照は 01 の `expr` と 03 の `type` を修飾形で拡張する：

```
qualified_name ::= mod_name '.' IDENT                -- 修飾値
                 | mod_name '.' TCON                 -- 修飾型 / コンストラクタ
```

（`mod_name` は §抽象構文 で定義した、ドット付きの `upper_ident`
列。）

修飾名 `M.x` は次の手順で解決する：

1. `M` をモジュール接頭（`Data.List` のようなドット付きパスでもよ
   い）、`x` を識別子として扱う。
2. `M` という名のモジュール（または `as M` で別名設定されたモジュ
   ール）のエクスポート束縛集合から名前を探索する。
3. スコープ内に `M` という名のモジュールが無いか、`x` がそれに
   エクスポートされていなければ、参照は静的エラー。

修飾アクセスは 04 のレコードフィールド選択と衝突しない。レコード
選択 `r.f` は左が **式**、右が **`lower_ident`**（04 §抽象構文）。
修飾アクセスは左が **`mod_name`**（`TCON` またはそのドット付き列）、
右が `lower_ident` または `TCON`。パーサは 02 の `lower_ident` /
`upper_ident` の字句的区別で判別する：`X.y` の最左トークンが
`upper_ident`（`TCON`）なら修飾参照であってレコード選択ではない。
逆にレコード型の識別子は常に `lower_ident`。`Maybe.Just` は修飾
コンストラクタであり、値レベル `lower_ident` `p` に対する `p.x`
はレコード選択。

## トップレベルシグネチャ

文書 01 は未解決の問い 2 として、トップレベル型シグネチャを必須と
するか任意とするかを残した。本文書は **境界規則** で決着させる：

- 各 **エクスポートされた** トップレベル値束縛は、モジュール内の
  どこかに明示的な型シグネチャ（`x : scheme`）を持たなければなら
  ない。シグネチャは束縛自身の前でも後でも構わないが、存在しなけ
  ればならない。
- エクスポートされた **クラス** メソッドのシグネチャはクラス宣言
  自身から来る。別途のトップレベル形は不要。
- エクスポートされた **data** および **class** 宣言は別途のシグ
  ネチャを要しない。種は宣言から推論される。
- **プライベート** なトップレベル値束縛はシグネチャを省略しても
  よい。型は通常通り推論される。

合理性。モジュール境界のシグネチャは、下流モジュールがコンパイル
時に頼る束縛の型を固定する。シグネチャが無いと、内部実装の変更が
公開型を静かに変えうる。内部束縛にはそのリスクが無いので、シグネ
チャを義務化しても純粋に紙仕事が増えるだけである。

この規則は 07 の制約付きスキームと綺麗にかみ合う：エクスポートさ
れた束縛はその完全な制約付きスキーム（`foo : ∀ a. Eq a => a -> a
-> Bool`）を保持し、それがインポート側から見える。

## 循環インポート

モジュールグラフ — 節点をモジュール、辺を `import` 関係とする有向
グラフ — は **非循環** でなければならない。モジュール `M₁` が
（推移的に）`M₂` をインポートするなら、`M₂` が（推移的に）`M₁`
をインポートすることは許されない。循環インポートは本層の静的エラー。

合理性。モジュール DAG は型推論順序を well-defined に保ち、インス
タンス解決閉包を 1 パスで計算可能にする。緩和には遅延モジュール評
価（付随する順序の罠）か Haskell 風 `.hs-boot` 署名ファイルが必要
になる。現 draft の例題からはどちらの導入も正当化されない。08 未
解決の問い 4 を脱出口の問いとして参照。

## インスタンスとモジュール（07 の精緻化）

文書 07 はインスタンス `instance C T` が `C` か `T` を定義するモジ
ュールに属さねばならない（孤児なし）と述べた。本文書はその規則を
grounding する：

- インスタンス宣言は、モジュール `M` のインポート閉包が当該インス
  タンスをホストするモジュールを含むとき、かつそのときのみ `M` の
  スコープに入る。
- 孤児なし規則は **静的エラー**：コンパイラ（あるいはチェッカ）は、
  `C` も `T` の最外型コンストラクタも定義していないモジュール内の
  `instance C T` を拒絶しなければならない。
- インスタンスは名前を持たず、エクスポートリストに **列挙されな
  い**。モジュールと共に移動する：モジュールをインポートすると、
  そのモジュールが定義する（あるいはエクスポートリスト中の
  `module M` で再エクスポートする）全インスタンスもインポートさ
  れる。

この機構こそが、07 の「コヒーレント解決」をモジュール横断で成立さ
せる鍵である：各インスタンスには唯一のホームモジュールがあるため、
解決はインポート順序に依存せず、推移的インポート閉包のみに依存す
る。

**不変条件。** 上記の孤児規則と推移的可視性規則により、任意の
`(C, T)` クラス／型頭ペアと任意のインポート側モジュール `M` につ
いて、`M` のスコープに入るインスタンス `instance C T` は高々 1 つ
である。これは 07 の孤児なし規則を具体的なモジュールシステムに
grounding した見返りであり、07 の重複なし保証が追加機構なしに単
一ソースから推移的インポート閉包全体へ一般化する。

## 再エクスポート

エクスポート項目 `module Other.Module` は、`Other.Module` 自身が
エクスポートする全ての名前を同名で再エクスポートする。再エクスポ
ート対象のモジュールは、再エクスポート側でも既にインポート済みで
なければならない — `module M` エクスポート形はスコープに入ってい
る名前を参照する。`import` はどの形式でもよい（`import M`、
`import qualified M`、`import M as L` 等）。前提条件はスコープに
入っていることであり、非修飾であることではない。

```
module Data.Containers
  ( module Data.List
  , module Data.Map
  ) where

import Data.List
import Data.Map
```

再エクスポートは **推移的**：`Data.List` が `module
Data.List.Internal` を再エクスポートしているなら、
`module Data.Containers ( module Data.List )` も（推移的に）
`Data.List.Internal` のエクスポートを露出する。これが「再エクスポ
ートは単調に合成される」の意味。

**再エクスポート間の名前衝突。** 2 つの再エクスポートが同じ名前を
露出するとき、アンブレラのエクスポートリストは §可視性 の二重イン
ポートケースと同じ規則に従う：

- 2 つの再エクスポート名が **同じ** 束縛（つまり再エクスポート連
  鎖を辿った原宣言が同一）に解決するなら、アンブレラは 1 つだけを
  エクスポートし、曖昧性は生じない。
- **異なる** 束縛に解決するなら、アンブレラのエクスポートリストは
  静的エラー。再エクスポート側は、片方を（`as` によるモジュール
  インポートで）改名するか、エクスポートリストから除くか、
  `module M` ではなく選択した名前を個別にエクスポートすること。

再エクスポートにより、複数の実装モジュールを束ねた curated API を
アンブレラモジュールで提示できる。より選択的な再エクスポート形
（`module Data.List (foo, bar)` 風）を採るかは未決。08 未解決の問
い 3 参照。

## 設計メモ（非規範的）

- **1 ファイル = 1 モジュール。** パスベースのモジュール名とぴた
  り噛み合う。ファイルシステムを反映することでツーリングが単純に
  なり、`Data.List` がページ上で明示的になる。

- **既定は非修飾インポート。** Haskell と Elm の一般ケースに一致。
  `import qualified` は 1 語長いが、名前衝突が迫るときは明示性に
  見合う。

- **ライブラリコードでは明示的エクスポートリスト。** 仕様は両形式
  を認めるが、ライブラリ作者は明示形を優先すべき。M9 の例題は明示
  リストを使う。

- **エクスポート済シグネチャ。** 「エクスポート済束縛はシグネチャ
  必須」規則は、書きやすい任意体制と読みやすい全必須体制の折衷で
  ある。コストを、見返りがある場所 — モジュール境界 — にちょうど
  配置する。

- **孤児なしをモジュールシステムが強制。** 07 の孤児なし規則は
  「C を定義するモジュール」の具体的な概念なしには強制できなかっ
  た。本文書がそれを与える。Haskell の既定（孤児を警告するだけ）
  より強い姿勢で、現代的 Haskell のスタイル、PureScript、Idris の
  慣習と一致する。

- **循環インポートは禁止。** §循環インポート がこれを規範として
  定める。設計メモ補足：M9 の例題プログラムが将来回避口（Haskell
  風 `.hs-boot` 署名）を要求するかもしれないが、それまでは DAG 制
  約を保つ。

- **`import` は `decl` の一種。** BNF は `import_decl` をトップレ
  ベル `decl` に置くが、実務上インポートはファイル頭の `module`
  ヘッダの後に固まる。本文書はその配置を構文的には強制しない — ツ
  ーリング・スタイル慣習の領分。実コードはインポートを先頭に置く。

## 未解決の問い

1. **`Maybe` と `Maybe(..)` の既定。** 型がコンストラクタ節なし
   でエクスポートリストに現れた場合、Sapphire の既定は「型のみ、
   コンストラクタなし」。Haskell も同様。代替として
   `Maybe` ≡ `Maybe(..)`（既定で全てエクスポート）も考えられる。
   draft は否定。

2. **漏洩したプライベート型の診断タイミング。** プライベート型を
   エクスポート済シグネチャで使うと、インポート側が名前で参照で
   きない識別子を漏らすことになる。コンパイラは、定義時点で拒絶
   する（定義モジュールで早期・明瞭なエラー）か、黙って受理して
   インポート側の使用箇所で失敗させる（遅延・使用箇所で具体的な
   エラー）か。draft は沈黙する — どちらも擁護しうる。

3. **選択的再エクスポート。** `module M (foo, bar)` 風の選択的再
   エクスポートを認めるか。draft は全モジュール再エクスポートの
   みを認める。選択的形式（`module M (foo)` か `M.foo`）は未解決
   の問い。

4. **モジュール跨ぎ相互再帰。** 本層では厳格に禁止。Haskell は
   `hs-boot` ファイルで許容する。Sapphire でも同様の脱出が必要に
   なるかもしれないが、既定は否定。

5. **ヘッダ無し既定 `module Main`。** draft は小さな例のための糖
   衣として認める。ライブラリファイルが明示的 `module` ヘッダを必
   要とするか、任意 `.sp` ファイルが省略してよいかは未解決。
   「ライブラリファイルはヘッダを要求、単一ファイルスクリプトは
   省略可」寄り。

6. **モジュールレベル fixity 宣言。** 05 / 07 は固定演算子表を用
   いる。05 未解決の問い 3（ユーザー宣言 fixity）が肯定に転じたら、
   fixity 宣言はモジュールスコープとなり、本文書がそのエクスポー
   ト／インポート意味論（`module Other.Module` 再エクスポートが
   fixity 宣言も再エクスポートするか等を含む）を規定する必要があ
   る。現時点では計画しないが完全性のために明記する。

7. **クラスごとのメソッドエクスポート。** draft は `class C` か
   `class C(..)` のみを認める。Haskell は選択したメソッドをエクス
   ポートする `class C(m1, m2)` を認める。本文書の `export_item`
   文法を同様の形に拡張するかは未決。
