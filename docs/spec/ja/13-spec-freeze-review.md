# 13. 仕様凍結レビュー

状態: **review** — 本文書は spec-first フェーズ自身の締めくくり。
非規範（新しい言語機構を導入しない）だが、規範的な結果を生む：
Sapphire の仕様が次フェーズに進めるほどに安定しているか、未解決の

## 目的

Sapphire の spec-first フェーズ（`docs/project-status.md` による
定義）は 12 の draft 文書 01〜12 を生んだ。本文書は：

1. 各 draft の状態を要約する。
2. draft 横断の全 OQ を 1 つの監査表に統合し、各項目に対する処理
   案を付す。
3. 文書間整合チェックを可視化する。
4. `CLAUDE.md` の phase-conditioned rules を再検討し、実装言語
   フェーズ開始時に何を変えるかを提案する。
5. **凍結判断** を述べる：仕様が蓄積を止め実装を始めてよいとみな
   される条件。

本文書の成果は user のサインオフ後に周辺リポジトリへ反映される —
roadmap 更新、draft → final のステータス更新、`CLAUDE.md` の規則
改訂、そして実装時へ送られる OQ のここでの削除（必要なら）。

### M10 で着地する決定

上表の **D** 行は各々先行 draft を修正するため、ユーザのサインオ
フを要する。要約：

1. **レコード形コンストラクタ payload（04 OQ 2）。** 最初の実装
   では認めない。`data T = T { ... }` は予約構文だが未決のまま、
   実装経験後の後続マイルストーンまで据え置く。
2. **型別名（09 OQ 2）。** `type T = τ` を透明な別名（名目的
   identity なし）として認める。09 に小節を追加、12 の例 3 は
   未解決の問いではなく M10 の決定への参照に更新。
3. **`readInt` / `readFloat` を prelude へ（12 OQ 6）。** 両者
   を 09 の utility 集合に追加。`readFloat` は数値タワー決定に連
   動。09 と 12 を更新。

**C** 行は排版的／阻害要因にならない修正で、M10 コミットと共に
さらなる決定なしで着地できる：02 OQ 4 / 02 OQ 5、04 OQ 3、05 OQ

## `CLAUDE.md` phase-conditioned rules

現行 `CLAUDE.md` の "Phase-conditioned rules (spec-first phase)"
節：

- ホスト言語中立性（仕様内で「Rust / OCaml / Haskell なら…」を避
  ける）。
- コンパイラ scaffolding / toolchain install なし。
- 実装を含意する要求は tension として上げる。
- 仕様決定は `docs/spec/` 内に残す。

これらのうち：

- **ホスト言語中立性** は **実装言語選定プロセスの間も** 有効に
  保つ。ホストが選ばれた時点で緩和する。言語が選ばれたら、その言
  語のコードとビルド scaffolding は両方とも許容され、期待される。
- **コンパイラ scaffolding なし** は実装言語決定が着地するまで
  有効に保つ。決定後は反転し、scaffolding が作業の一部になる。
- 他の 2 規則はフェーズ非依存で、そのまま残す。

M10 は `CLAUDE.md` そのものを修正しない。次フェーズの改訂に向け
たユーザ可視の推奨を生むに留める。推奨する次フェーズ
`CLAUDE.md` 節は：

> ## Phase-conditioned rules (implementation-language-selection phase)
>
> - spec-first フェーズは終了した。01〜13 は draft であり、OQ は
>   13 で処理された通り。閉じられた問いを新鮮な決定なしに再開し
>   ない。
> - 実装言語選定が主活動である。候補・trade-off 表・プロトタイ
>   プ結果・選定決定そのものは `docs/impl/`（新しいツリー）に
>   住む。
> - 言語が選定されるまで、仕様ツリー（`docs/spec/`）ではホスト
>   言語中立性を続ける。`docs/impl/` 内の trade-off 議論は適用外。
> - 実際のコンパイラ scaffolding は選定着地まで待つ。
> - 仕様決定は引き続き `docs/spec/` に記録する（spec-first フェー
>   ズから引き継ぐ規則）。チャットは真実の源ではない。実装決定は
>   `docs/impl/` に記録する。

これは opt-in の提案。ユーザが決める。

   複合表層形はいずれも 02 の予約を隙間なく消費する。

## 設計メモ（非規範的）

- **Freeze と freezing。** 13 における「凍結」は「新しい層の追加
  を止める」であって「以後編集一切なし」ではない。draft は draft
  のまま。実装フェーズのフィードバックが更なる改訂を駆動する。
  本当の "final" ステータスは、少なくとも 1 つの完全なコンパイラ
  実装がすべての隅を伸ばした後に初めて意味を持つ。

- **実装言語選定は次に。** M10 はホスト言語を選ばない。それは次
  フェーズ最初の活動。候補・trade-off 基準・プロトタイプ結果は
  ここではなく `docs/impl/` に属する。

- **なぜここでもっと OQ を解かないか。** 大半の「K」「L」問いは
  実装経験か深い言語設計の思考のいずれかを要する。監査文書で答
  えるのは設計を急かすか、実装が証拠なしに防衛せねばならない立
  場に先行コミットするかのいずれかになる。残すのが規律ある道。

- **文書間チェックこそバグの隠れ場所。** §文書間整合チェック の
  一貫性表は本レビューの load-bearing 部分。将来の仕様編集はすべ
  てそれらを再実行すべき（予約語集合、演算子表、予約句読点、暗黙
  インポート、prelude シグネチャ）。
