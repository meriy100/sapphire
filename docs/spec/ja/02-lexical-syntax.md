# 02. 字句構文

状態: **draft**。レイアウト感受性、文字列エスケープ、演算子表が後続文書で
実際に使用される過程で改訂されうる。

本文書は `docs/spec/02-lexical-syntax.md` の日本語訳である。英語版が規範的
情報源であり、BNF 生成規則・予約語集合・予約記号・番号付き未解決の問いは

## 動機と範囲

文書 01（中核式言語）はトークン `IDENT`・`INT`・`STRING`・`BOOL`・`TVAR`・
`TCON` を所与として扱っていた。本文書はそれらを確定する。また、後続文書
（ADT、レコード、モジュール、Ruby インタロップ）が単調に拡張していく表層
形状も固定する。

範囲内：

- 文字集合と論理行の構造。
- 空白とコメント。
- 識別子（項レベル・型レベル）。
- 予約語と予約記号。
- リテラル（整数、文字列；ブール値は識別子の節で扱う）。
- 演算子文字のトークン。
- レイアウト：Sapphire はレイアウト感受性を持ち、明示的なブレースによる
  脱出口を持つ。

後続文書へ送る項目：

- 完全なレイアウトアルゴリズム（Haskell 2010 の `L` 関数相当）。`where`
  と `case` が存在する前に完全には定義できない。
- 文字リテラル、浮動小数点リテラル、複数行文字列リテラル。
- 演算子の優先度 / 結合性の表（問 3 参照）。
- 16 進・8 進・2 進整数リテラル。

## ソーステキスト

Sapphire のソースは UTF-8 で符号化された Unicode コードポイントの列であ
る。論理行は `\n`（LF）で終端する。`\r\n`（CRLF）は入力時に `\n` に正規化
される。単独の `\r` は字句エラーである。ファイル先頭の UTF-8 BOM が存在す
る場合、これは無視される。

行番号は 1 から始まる。列番号は 1 から始まり、BOM 除去後の Unicode コー
ドポイントを数える。

## 空白とコメント

空白文字は半角スペース（U+0020）、水平タブ（U+0009）、および改行
（U+000A）である。タブ幅は意味的に **有意ではない**。*レイアウト* 節の問
4 を参照。

コメントは次の二形式であり、いずれも空白と同じく振る舞う：

```
-- 行コメント。論理行の末尾まで続く
{- ブロックコメント。複数行にまたがってよく、
   別のブロックの内側で {- 入れ子にできる -} -}
```

ブロックコメントは入れ子にできる。終端のないブロックコメントは字句エラ
ー。文字列リテラル内部の `--` はコメントではない。

コメントと演算子トークン化は通常の最長一致（maximal munch）則で相互作用
する：字句解析器はまず可能な限り長い `op_char` の連続を取り、その結果が
`--` に一致した場合にのみ行コメントとして解釈する。したがって `-->` は
単一の演算子トークンであり、`-` に続く行コメントになることはない。一方、
ソース行の最長 `op_char` 連続がちょうど `--` である場合は、その `--` が
行コメントを開始する。

## 識別子

先頭文字によって二つの識別子クラスを区別する：

```
lower_ident ::= [a-z_] [A-Za-z0-9_']*        -- 項名、型変数
upper_ident ::= [A-Z]   [A-Za-z0-9_']*       -- 型コンストラクタ、
                                             -- データコンストラクタ、モジュール名
```

プライム文字 `'` は先頭文字以外のどこにでも現れてよい。

文書 01 で使われるトークン名は上記のクラスに次のように対応する：

- `IDENT` は項位置に現れる `lower_ident`。
- `TVAR`  は型位置に現れる `lower_ident`。
- `TCON`  は型位置に現れる `upper_ident`。

単独のアンダースコア `_` は予約されたワイルドカード（後の文書でパターン
に現れる）であり、束縛識別子ではない。

## 予約語

次の小文字の語は予約されており、`lower_ident` として出現できない：

```
let    in     if     then   else   case   of
module import export where  forall data
class  instance do
as     hiding qualified
```

`data` は文書 03 が、`class`・`instance`・`do` は文書 07（型クラス
と higher-kinded types）が、`as`・`hiding`・`qualified` は文書 08
（モジュール）が、いずれも下記の追加拡張条項に基づき加えた。

`True` および `False` は予約語では **ない**。表層形式は通常の
`upper_ident` と同じである。それでも字句解析器がこれらを独立したトーク
ンクラスとして認識するのか、それとも通常の `upper_ident` のまま扱い
prelude 束縛により `Bool` の二値に解決するのかは、下の未解決の問い 1 を
参照。§ブールリテラル では値側から同じ論点を述べる。

予約語集合は、後続文書がデータ宣言・パターンマッチング・Ruby インタロップ
モナドを導入するにつれて増える。仕様段階の成長は厳密に追加的（additive）
であることが期待される。

## リテラル

### 整数リテラル

```
int_lit ::= [0-9] [0-9_]*
```

アンダースコアは桁区切りとして許され、値計算では無視される：`1_000_000`
は `1000000` と等しい。先頭のゼロは許されるが、8 進解釈を導入しない。16
進・8 進・2 進形式は後続文書へ送る。

負の整数リテラルは字句形式では **ない**：`-3` は演算子 `-` の適用である
（問 2 参照）。

### 文字列リテラル

```
string_lit   ::= '"' string_char* '"'
string_char  ::= <'"' と '\' を除く任意の Unicode コードポイント>
               | '\' escape
escape       ::= 'n' | 't' | 'r' | '\\' | '"' | 'u' '{' hex_digit+ '}'
hex_digit    ::= [0-9a-fA-F]
```

`\u{…}` は 1 〜 6 桁の 16 進数で Unicode スカラー値を指定する。サロゲー
トコードポイントは字句エラー。未知のバックスラッシュエスケープは字句エ
ラー。

本層では文字列リテラルは論理行をまたがない —— クォート内部の生の `\n`
は字句エラー。複数行文字列リテラルは後続文書へ送る。

### ブールリテラル

文書 01 で使われる `BOOL` は表層形式 `True` と `False` を指す。これらを
独立したトークンクラスとして字句解析器が認識するのか、それとも通常の
`upper_ident` であって prelude が `Bool` の二値に束縛するのかは、本文書
では決定しない —— 下の未解決の問い 1 を参照。表層構文（プログラマが書く
文字列）はどちらの解釈でも一致するが、01 の (LitBool) 規則がプリミティ
ブであるか、(Var) と prelude 束縛から導出可能であるかの位置付けが異な
る。

## 演算子トークン

演算子トークンは次の記号文字から構成される：

```
op_char ::= '+' | '-' | '*' | '/' | '%' |
            '<' | '>' | '=' | '!' | '&' | '|' |
            '^' | '?' | '~' | ':' | '.'
op      ::= op_char+     -- 次節で列挙する予約形式を除く
```

演算子トークンは常に `op_char` の **極大** 連続であり、かつ次節で列挙す
る予約形式のいずれとも一致してはならない。

丸括弧・角括弧・波括弧 `(`、`)`、`[`、`]`、`{`、`}` はそれぞれ単一文字
トークンであり、`op_char` ではない。このうち `[` と `]` は現時点のどの表
層生成規則からも参照されていない —— 後続文書で導入するリスト構文に先回
りして予約している。

カンマ `,` とセミコロン `;` はそれぞれ単一文字トークン。

上で定義した `op` 非終端は、本層のどの BNF 生成規則からも参照されない。
ここで確定しているのは `op_char` の連続がどのように単一トークンにまとま
るか、どの並びが予約されているかというトークン化規則である。それらのト
ークンを消費する文法 —— 優先度と結合性を含む。問 3 を参照 —— は後続文書
の責任とする。

### 予約記号

次の演算子形式は、ユーザー演算子ではなく予約構文である。これらは上の
`op` 非終端から除外される。

```
=     ->    <-    =>    :     ::    :=    |     .
\     @
```

- `=` は定義形式（`decl` および `let` 内部で使用）。
- `->` は関数矢印（型・λ で使用）。
- `<-` は `do` 記法のモナドバインドのために予約（文書 07）。
- `=>` はスキームにおける制約 / 型クラス矢印（文書 07）。
- `:` は型注釈の区切り（文書 01、`decl ::= IDENT ':' type` 参照）。
- `::` は予約（候補：リスト cons、パターンレベル型注釈）。選択は後続文書
  で行う。
- `:=` は将来の代入風形式（例：Ruby インタロップの束縛）のために予約。
  本層では導入しない。
- `|` はデータコンストラクタの選択肢、case パターンの選択肢、および
  レコード更新構文（`{ r | f = v }`、文書 04）のために予約。
- `.` は修飾名（`Mod.name`）、レコードフィールドアクセス、および文書
  01 のスキーム記法 `forall TVAR* '.' type` のために予約。前二者の重複
  はモジュール文書とレコード文書で解決する。
- `\` は λ バインダ。
- `@` は as-pattern のために予約。

`op` 非終端は `op_char` の極大連続を取り、かつ上記の予約形式を除外する
ため、単独の `:` は常に型注釈トークンであり、ユーザー演算子にはなりえ
ない。`:>` や `:|` のようなより長い並びは通常の演算子である（`:` 単独
でもなければ他の予約形式とも一致しないため）。

## レイアウト

Sapphire は Haskell / Elm の伝統に従い、既定で **レイアウト感受性** を持
つ。明示的なブレースが脱出口として利用可能。

以下の規則は、後続文書で複数束縛形式を獲得する予定のブロック開始キーワ
ード（`let` を含む）を念頭に書かれている。文書 01 の単一束縛
`let IDENT = expr in expr` はブロック開始の振る舞いを実質的には使わな
い：キーワード `in` が単一項目ブロックの明示的終端として働くため、列
比較が発火する機会が存在しない。

非形式的な規則（`where` と `case` 導入後の後続文書で完全に精密化する）：

- *ブロック開始キーワード* は `let`、`where`、`of`、`do`。`do` は
  文書 07（型クラス / `do` 記法）が §予約語 の追加拡張条項に基づ
  き加えた。
- ブロック開始キーワードに続く最初の非空白トークンが `{` でない場合、新
  しいブロックが開く。そのトークンの列がブロックの **基準列** `c` とな
  る。
- 以降のトークンで列が `c` と等しいものは、ブロックの新しい項目を開始
  する。
- 以降のトークンで列が `c` より大きいものは、現在の項目を継続する。
- 以降のトークンで列が `c` より小さいものは、ブロックを閉じる。

明示的なブレース / セミコロン形式は常に利用可能で、ブレース内ではレイア
ウトが無効化される：

```
let { x = 1 ; y = 2 } in x + y
```

タブ文字はレイアウトアンカーとして使われ **ない**。論理行の最初の非空白
トークンより前に出現する（すなわちレイアウト字下げを設定または検査する
位置に出現する）水平タブは字句エラー。行内部、最初の非空白トークンより
後のタブは通常の空白である。これは既定で厳格な扱いである。問 4 を参照。

## 文書 01 への字句対応

便宜上、文書 01 が参照するトークンは次のように解決される：

- `IDENT`  — 項位置の `lower_ident`。
- `TVAR`   — 型位置の `lower_ident`。
- `TCON`   — 型位置の `upper_ident`。
- `INT`    — `int_lit`。
- `STRING` — `string_lit`。
- `BOOL`   — 表層形式 `True` と `False`。これが独立した字句クラスなの
  か、それとも prelude 束縛の `upper_ident` 二つなのかは未解決の問い
  1 を参照。

## 未解決の問い

1. **`True` / `False` を独立した字句クラスとするか prelude コンストラ
   クタとするか。** 予約語にする選択肢は §予約語 で既に否定されてい
   る。残る二択。(a) 字句解析器が認識する独立クラスとする —— 01 の
   `BOOL` トークンと整合し、(LitBool) をプリミティブ規則として保てる。
   (b) prelude に束縛された通常の `upper_ident` とする —— `Bool` を将
   来のユーザー定義 ADT と統一的に扱える代償として、01 の (LitBool)
   は (Var) から導出可能な派生規則となる。(b) はさらに、prelude コン
   ストラクタのシャドーイングを言語レベルで禁止するのか lint で扱うの
   かという問いを生む。
   *文書 09 で決着*：(b)。`Bool` は prelude の ADT
   `data Bool = False | True`。`True` と `False` は通常の
   `upper_ident` コンストラクタ。01 の (LitBool) はこれらコンストラク
   タスキームに対する (Var) から導出できる。コンストラクタ名を値に
   再束縛することは文書 06（§設計メモ、名前空間規律）により禁じら
   れる。ユーザの `data` 宣言によって prelude のコンストラクタをシャ
   ドウイングする場合は、文書 08 のインポート衝突規則の対象となる
   （同名の定義が 2 つスコープに入ると、使用箇所で曖昧になる）。
2. **単項マイナス。** 式先頭の `-` は固定優先度を持つ単項演算子か、そ
   れとも否定は `negate x` / `(0 - x)` と書かなければならないか。前者
   は親しみやすいが `-` を単項・二項双方の演算子とし両者の相互作用を規
   定する必要が生じる。後者は仕様化が単純。
   *文書 05 で決着*：単項 `-` は式開始位置で `negate` の表層糖衣として
   認める。
3. **演算子表。** Elm 風に言語組み込みの固定優先度 / 結合性を採るか、
   Haskell 風にユーザー宣言可能な固定性（`infixl N`）を採るか。後続文
   書でのアドホック多重定義と相互作用する。
   *文書 05 で決着*：Elm 風固定表を採る。ユーザー宣言 fixity は 05 未
   解決の問い 3 に再度立てる。
4. **レイアウト位置のタブ。** 字下げ位置のタブを字句エラーとするのは
   厳格だが曖昧さがない。代替案はタブ幅を固定すること。既定厳格が現在
   の草案。
5. **識別子の文字集合。** ASCII に制限するか、`lower_ident` /
   `upper_ident` に Unicode 文字を許すか。Ruby インタロップは Unicode
   方向に誘引しうる（Ruby のメソッド名は ASCII だがユーザーコードはそ
   うとは限らない）が、Unicode 識別子はツーリングとレビューを複雑化す
   る。
6. **`::` の曖昧性解消。** リスト cons とパターンレベル型注釈のどちら
   が `::` を得るか。両方を望むなら他方に別の綴りを与える必要がある。
   *文書 05 で部分的に決着*：`::` はリスト cons。パターン位置の型注記
   を導入するかどうか、また導入する場合の綴りは未決であり、M3 で確
   定する。
